## 二、15 分钟架构设计讲解

### (1) 系统总体架构 (3 min)

- **微服务 + Agent 模式选择理由**: 领域解耦、故障隔离、独立扩容
- **7 个容器**: api-gateway / requirements-agent / matching-agent / evaluation-agent / knowledge-base / refactoring-agent / neo4j
- **编排层**: LangGraph StateGraph (START→extract→match→evaluate→trace→END), 手动 fallback

### (2) LLM + 知识图谱双驱动 (3 min)

**三层推理架构**:

```
Layer 1: 规则引擎 (score_style)  — 确定性匹配, 永远运行
Layer 2: 知识图谱 (graph_matcher) — Neo4j 关系推理, 可选
Layer 3: LLM (evaluation-agent)   — 语义投票+摘要, 可选
```

**融合策略**: `final_score = rule_score + min(graph_score, rule_score // 2)`

**设计原则**: "规则保证下限, 图谱增强关系, LLM 提升上限"

**降级路径**:
- 全功能: 规则 + 图谱 + LLM
- 无 Neo4j: 规则 + LLM (图谱得分为 0)
- 无 LLM: 规则 + 图谱 (LLM 摘要用模板)
- 最小模式: 纯规则引擎 (JSON 知识库, 零外部依赖)

### (3) Neo4j 知识图谱设计 (2 min)

**节点类型**: ArchitectureStyle(10) / QualityAttribute(10) / Scenario / Risk / Feedback / ADR

**关系类型**: HAS_QUALITY / SUITABLE_FOR / HAS_RISK / COMPLEMENTS / RECOMMENDS / BASED_ON

**JSON fallback**: `KNOWLEDGE_BACKEND=auto/json/neo4j` 环境变量控制, Neo4j 不可用时自动降级读取 JSON 文件

### (4) Few-shot Prompt Engineering (2 min)

- **需求提取**: 6 个中文示例 (模糊高并发/否定语义/安全合规/数据密集/强一致/重构倾向)
- **评估摘要**: 3 个完整报告示例 (Event-Driven/Microservices/Layered+CQRS)
- **降级**: ImportError 时自动回退零样本 Prompt

### (5) ADR 决策溯源 (1.5 min)

- 每次推荐自动生成 ADR, 包含完整决策链
- 双重存储: JSON 文件 (主) + Neo4j 节点 (可选)
- 失败不阻塞: ADR 写入失败不影响推荐结果

### (6) 组合推荐 + 重构建议 (1.5 min)

- 5 种组合模式 (微服务+事件驱动 / 分层+CQRS / 管道+事件驱动 / 六边形+微服务 / CQRS+ES)
- 5 种重构模式 (绞杀者 / 防腐层 / 模块化单体 / CQRS / 事件驱动迁移)
- 评分公式: 分数和 + 覆盖互补 + 图谱加分 - 复杂度惩罚

### (7) LLM 缓存 (1 min)

- 请求级缓存: 相同需求+模型+知识库版本 → 直接返回

## 三、高频问答准备

### Q1: 为什么不用单一通用 Agent 而是多 Agent 协作?

**A**: 单一 Agent 难以同时处理特征提取(规则密集)、风格匹配(知识库查询)、评估(LLM 调用)三种异构任务。多 Agent 协作将不同职责隔离在不同进程中: requirements-agent 纯计算, matching-agent 规则+图谱, evaluation-agent LLM 调用。每个 Agent 独立优化、独立降级, LLM 波动不影响特征提取和规则匹配。

### Q2: Neo4j 挂了怎么办?

**A**: `KNOWLEDGE_BACKEND` 支持三种模式: json/neo4j/auto。默认 auto 模式优先 Neo4j, 不可用时自动回退 JSON 文件。JSON 模式零外部依赖, 系统完全可运行。图数据库是增强能力而非必要条件。

### Q3: 规则引擎和 LLM 如何分工?

**A**: 规则引擎负责确定性匹配(关键词→特征→标签→风格), 保证下限。LLM 负责语义级辅助判断(补全规则未命中的特征、自然语言摘要), 提升上限。两者独立运行, LLM 的结果经过严格校验(投票结果必须在候选列表, 摘要失败回退模板)。

### Q4: 缓存如何保证不返回过期结果?

**A**: 缓存键包含 `knowledge_version`(styles 文件 MD5 前 8 位)。知识库内容变更时 knowledge_version 改变, 旧缓存自动失效。也可通过 `POST /cache/clear` 手动清空或设置 `CACHE_TTL_SECONDS` 控制过期时间。

### Q5: 组合推荐和单一推荐的优先级?

**A**: 单一推荐是基础, `recommended_style` 字段始终存在。组合推荐是新增字段 `recommended_combination`, 不影响原有 Top 3 检查。组合评分在单一评分之后进行, 使用独立评分公式。

### Q6: 重构建议的可靠性如何?

**A**: 重构建议采用规则引擎主导 + LLM 润色的双层架构。规则引擎基于 14 个关键词和 5 种检测规则生成确定性模板, LLM 仅用于润色表述。LLM 不可用时直接使用规则模板, 建议仍然具体和工程化。
