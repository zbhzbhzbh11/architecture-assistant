## 为什么需要多层测试

### 一个"体检套餐"的类比

你不会只用一种检查来判断身体状况——血常规看基础指标、CT 看内部结构、心电图看心脏功能。不同检查覆盖不同层次，各有侧重。

软件测试也一样：

| 测试层 | 类比 | 覆盖什么 | 执行速度 |
|--------|------|---------|---------|
| **单元测试** | 血常规 | 单个函数的输入输出 | 秒级 |
| **回归测试** | 定期体检 | 20 条用例的端到端链路 | 分钟级 |
| **冒烟测试** | 快速筛查 | 核心链路能否跑通 | 秒-分钟 |
| **自动验收** | 入职体检 | 43 项课程要求逐条检查 | 分钟级 |

## 四层测试体系详解

### 第一层：76 条单元测试

运行方式：`pytest tests/unit/ -v`

| 测试模块 | 测试数量 | 覆盖内容 |
|---------|---------|---------|
| `test_requirements.py` | ~12 | 关键词匹配、否定过滤、LLM 语义补全降级 |
| `test_matching.py` | ~15 | score_style 规则评分、blend_scores 图谱融合、组合推荐 |
| `test_evaluation.py` | ~12 | LLM 投票/摘要、降级摘要、风险分析 |
| `test_knowledge.py` | ~15 | JSON 读写、Neo4j 可用性、ADR CRUD、学习权重 |
| `test_gateway.py` | ~12 | 缓存键生成、双引擎切换、API 路由 |
| `test_refactoring.py` | ~10 | 坏味检测、重构模式选择 |

**单元测试的关键特征**：不需要外部服务运行。可以脱离 Docker 独立运行，适合开发时快速反馈。

### 第二层：20 条回归用例

运行方式：`python tests/run_regression.py --gateway-url http://localhost:8000/api/v1/recommend`

需要系统运行（`docker compose up`）。

**测试数据集**：[tests/datasets/requirements_cases.json](../tests/datasets/requirements_cases.json)

```
20 条用例覆盖场景：
- 即时通讯 (2条): 不同措辞的同场景变体
- 电商平台 (1条): 高并发 + 强一致
- 企业系统 (3条): 审批/数据中台/内容管理
- 金融系统 (2条): 银行账务/支付网关
- 大数据系统 (3条): 日志分析/ETL/科研数据
- 特殊场景 (4条): 医疗/政务/物联网/证券
- 其他 (5条): 在线教育/短视频/SaaS/外卖/校园
```

每条用例有 5 个验证维度：

```python
# tests/run_regression.py:22-41
def evaluate_response(data):
    return {
        "has_top3": len(candidates) >= 3,              # 候选风格 ≥ 3
        "mainstream_covered": any(style in MAINSTREAM), # 主流风格在候选集
        "has_final": bool(final_report.recommended),    # 有最终推荐
        "explainable": has_rule_reasons and has_llm_field,  # 有规则理由
        "has_matrix": len(matrix) >= 3,                 # 对比矩阵 ≥ 3 行
        "pass": all(...),                                # 5 项全部通过
    }
```

**5 维验证的设计意图**：

| 维度 | 验证什么 |
|------|---------|
| `has_top3` | 系统能返回至少 3 个候选（满足课程要求） |
| `mainstream_covered` | 核心推荐不掉链子（三种主流至少出现一个） |
| `has_final` | 有明确的最终推荐（不是模棱两可） |
| `explainable` | 推荐有据可查（规则 + LLM 摘要都存在） |
| `has_matrix` | 对比矩阵完整（便于用户横向比较） |

### 第三层：冒烟测试

运行方式：`python tests/run_smoke.py`（需要服务运行）

快速验证核心链路是否可通——发一个请求，看能不能拿到 200 响应和完整报告。通常在部署后、演示前执行。

### 第四层：43 项自动验收

运行方式：`python scripts/check_assignment.py --project-root .`

**这是课程大作业的"自检开关"**——逐条对照作业要求检查项目是否符合标准。

| 分类 | 检查项数 | 检查内容 |
|------|---------|---------|
| 核心目录 | 4 | services/frontend/docs/tests 存在 |
| Agent 模块 | 含多个 Agent 文件 | FastAPI 路由、Agent 数量 ≥ 3 |
| 知识库 | 10 风格 | 风格数量 ≥ 10、字段完整 |
| 测试数据集 | 20 用例 | 用例数量 ≥ 20、格式正确 |
| LLM 集成 | 4 配置 + 10 调用点 | LLM API Key 配置、LLM 调用代码 |
| Web API | 23 端点 | API 路由存在、FastAPI/Flask 检测 |
| 可视化 | 前端页面 | HTML/CSS/JS 文件、展示区数量 |

**验收结果输出双格式**：
- `docs/自动验收检查结果.md`（人类可读）
- `docs/自动验收检查结果.json`（机器可读）

### 测试的组合使用

```
开发时        提交前        部署后          演示前
   │            │            │              │
   ▼            ▼            ▼              ▼
pytest       pytest   docker compose   冒烟测试
单元测试     单元测试   回归测试         (快速检查)
            (全量)    (20条用例)        + 自动验收
                                        (43项绿)
```

## 面试官可能怎么问

**Q1: 你怎么确保改了代码之后系统没有坏？**

> 四层验证。开发时跑单元测试（76 条，秒级反馈）。部署后用回归测试跑 20 条用例，检查每个用例的 5 个维度——候选数、主流覆盖、最终推荐、可解释性、对比矩阵。演示前再跑冒烟测试和自动验收（43 项）。这个测试金字塔保证了从"函数正确"到"课程要求满足"的逐层验证。

**Q2: 20 条回归用例是怎么选的？**

> 三个原则。第一是**场景覆盖**——20 条覆盖了即时通讯、电商、金融、大数据、企业系统、特殊场景（医疗/政务/物联网）等 8 大类场景。第二是**风格覆盖**——确保 Event-Driven、Microservices、Layered、CQRS、Pipeline-Filter 等主流风格都能被测试触发。第三是**措辞多样性**——同一场景（如即时通讯）用不同措辞写两条，验证特征提取的鲁棒性。

**Q3: 43 项验收 100% 通过，但有没有"自欺欺人"的检查项？**

> 诚实地说，自动验收验证的是**存在性**而非**质量**。比如"LLM 调用代码存在 10 处"——它能确认代码里有 LLM 调用，但不能验证 LLM 调用是否合理、Prompt 是否有效。这就是为什么还需要回归测试——20 条用例加 5 维验证才是真正验证"系统做得对不对"的。自动验收是"有没有"，回归测试是"好不好"。

## 简历上如何表达

> 建立四层测试体系：76 条单元测试（pytest）覆盖全部 6 个模块；20 条回归用例覆盖 8 大类场景，5 维验证（候选数/主流覆盖/推荐完整性/可解释性/对比矩阵）；43 项自动验收脚本 100% 通过；冒烟测试保障演示前快速验证。

## 本节小结

| 要点 | 一句话 |
|------|--------|
| 单元测试 | 76 条，脱离服务运行，秒级反馈 |
| 回归测试 | 20 条 × 5 维验证，需要服务运行 |
| 冒烟测试 | 部署后快速验证核心链路 |
| 自动验收 | 43 项，验证课程要求符合度 |
| 验收 vs 回归 | 验收验证"有没有"，回归验证"对不对" |
| 测试金字塔 | 底层多（单元）、上层精（端到端）、自检自动化 |
