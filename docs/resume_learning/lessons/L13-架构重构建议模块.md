## 为什么需要重构建议

### 一个"老房子翻新"的类比

搬进一栋老房子，你不会立即推倒重建。你会先检测问题——墙体开裂（单体耦合）、管道老化（扩展瓶颈）、电路过载（性能瓶颈）——然后制定翻新计划：先换管道、再改电路、最后装修。

软件系统的重构也是如此——不是所有系统都从零开始。很多时候用户已经有了一套系统（比如"现有单体电商系统"），他们想知道"怎么改"而不仅仅是"改成什么"。

## 当前项目如何实现

### 架构坏味检测（5 种）

```python
# refactoring_agent/app/main.py:58-84
ARCHITECTURE_SMELLS = {
    "monolith_coupling":       # "单体耦合过重"
    "scaling_bottleneck":      # "扩展瓶颈"
    "slow_release":            # "发布周期过长"
    "legacy_lockin":           # "遗留系统锁定"
    "data_coupling":           # "数据层耦合"
}
```

| 坏味 | 关键词 | 触发条件 |
|------|--------|---------|
| **单体耦合过重** | "单体"、"耦合严重"、"紧耦合" | 关键词命中 |
| **扩展瓶颈** | "性能瓶颈"、"扩展困难" | 关键词命中 **或** high_concurrency + scalability 同时 True |
| **发布周期过长** | "发布周期长"、"团队膨胀" | 关键词命中 |
| **遗留系统锁定** | "遗留系统"、"老系统"、"已有系统" | 关键词命中 |
| **数据层耦合** | "共享数据库"、"数据耦合"、"库存耦合" | 关键词命中 |

**双通道检测**：大部分坏味通过关键词匹配触发；`scaling_bottleneck` 额外支持特征推断——即使需求中没有"性能瓶颈"这个词，但如果同时检测到高并发和可扩展性需求，也可能存在扩展瓶颈。

### 重构模式（5 种）

```python
# refactoring_agent/app/main.py:88-144
REFACTORING_PATTERNS = {
    "strangler_fig":            # 绞杀者模式
    "anti_corruption_layer":    # 防腐层模式
    "modular_monolith_first":   # 模块化单体优先
    "cqrs_refactor":            # CQRS 读写分离
    "event_driven_migration":   # 事件驱动迁移
}
```

每种模式含两个要素：
- **when**：什么情况下适用
- **steps**：4 步渐进操作

### 重点模式：Strangler Fig Pattern（绞杀者）

这个模式是本模块最核心的重构建议，适合"单体 → 微服务"的渐进迁移：

```
绞杀者模式 4 步法：

Step 1: 识别并隔离目标业务边界（如订单模块），建立 API 网关路由规则。
          → 单体继续运行，但网关开始区分"新功能"和"旧功能"

Step 2: 新建独立微服务，逐步将目标功能从单体中迁移出来。
          → 订单服务独立部署，通过网关路由到新服务

Step 3: 通过网关将对应路由指向新服务，单体中该功能逐步退役。
          → 旧代码保留但不调用，作为 fallback

Step 4: 重复以上步骤，逐模块替换，直至单体完全退役。
          → 最终遗留一个"空壳"单体或完全下线
```

**为什么叫"绞杀者"？** 就像绞杀榕——新藤蔓沿着老树干向上攀爬，最终老树被吸收替换。不是砍倒重来，而是逐步替代。

### 规则模板生成

```python
# refactoring_agent/app/main.py:195-257
def build_rule_template(req, smells, patterns, recommended_style, ...):
    refactoring_needed = len(smells) > 0 or any(
        kw in req for kw in ["重构", "迁移", "拆分", "单体", "耦合", "老系统"]
    )

    # 从匹配的重构模式中提取迁移步骤（最多 2 个模式）
    migration_steps = []
    for p in patterns[:2]:
        migration_steps.append(f"【{p['name_zh']}】")
        migration_steps.extend(p.get("steps", []))

    # 根据推荐风格生成针对性风险
    risks = []
    if "Microservices" in recommended_style:
        risks.append("分布式事务一致性风险：拆分后需引入 Saga 或 TCC 模式。")
    if "Event-Driven" in recommended_style:
        risks.append("事件顺序和幂等性风险：消费者需实现幂等处理。")
    ...

    return {
        "refactoring_needed": refactoring_needed,
        "detected_architecture_smells": [...],
        "suggested_patterns": [...],
        "migration_steps": migration_steps,
        "risks": risks,
        "mitigation_suggestions": [...],
    }
```

### 调用方式：非阻塞异步

```python
# api_gateway/app/main.py:192-210
try:
    async with httpx.AsyncClient(timeout=8.0) as ref_client:
        ref_resp = await ref_client.post(f"{REFACTORING_AGENT_URL}/refactor", ...)
        if ref_resp.status_code == 200:
            refactoring_advice = ref_resp.json()
except Exception as e:
    logger.warning(f"Refactoring agent unavailable (non-fatal): {e}")

result["final_report"]["refactoring_advice"] = refactoring_advice
```

重构建议在**主推荐完成后**异步调用，超时 8 秒，失败不阻塞。这是因为它对"推荐"本身不是必需的——它是"进阶建议"。

### 触发逻辑

重构建议**对所有请求都会调用**，但只在需求文本包含重构关键词时才会判定 `refactoring_needed: true`。即使不含重构关键词，refactoring-agent 也会返回 `refactoring_needed: false` 的结果。

## 面试官可能怎么问

**Q1: 重构建议和推荐有什么区别？为什么不合并到一个 Agent？**

> 推荐回答"应该用什么架构"，重构回答"怎么从现在的架构改过去"。它们的输入不同——推荐只需要需求文本，重构还需要当前的架构情况（虽然没有明说，但从"已有系统"、"单体"等关键词推断）。把它们分开是因为职责不同——推荐是核心链路，重构是增值服务。分开后各自独立演进，重构挂了也不影响推荐。

**Q2: Strangler Fig Pattern 的渐进迁移有什么好处？**

> 四个字——"风险可控"。如果一次性重写整个系统，上线那天要么全对要么全错。绞杀者模式每一步只迁移一个模块，出了问题可以快速回退到单体中的旧实现。对于"不能停服"的生产系统，渐进迁移是唯一的可行方案。

**Q3: 重构检测是基于关键词的，会不会误判？**

> 会。比如用户说"我们之前的系统是单体的，现在想做新的"——可能不一定需要重构（新系统可以直接用微服务），但关键词"单体"会触发坏味检测。这是当前基于关键词检测的固有局限。解决方向：引入更精细的意图识别（区分"描述过去"和"寻求重构建议"），或者在检测到关键词后通过 LLM 二次确认。
