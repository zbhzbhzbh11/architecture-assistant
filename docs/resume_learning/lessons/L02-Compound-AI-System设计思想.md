## 为什么需要 Compound AI System

### 从"一根筋"到"组合拳"

想象你去看病。如果你只问一个医生（相当于只问 LLM），他可能凭借经验给你开药——有时候对，有时候错。但如果你去一家好医院，流程是这样的：

1. **护士先量体温血压**（规则引擎：基础指标检查）
2. **化验室做血液检测**（知识图谱：结构化数据检索）
3. **专家会诊综合判断**（LLM：复杂语义分析）
4. **主治医生拍板并写病历**（规则排序 + ADR 生成）

这就是 Compound AI System 的核心理念——**不是让一个模型做所有事，而是让合适的组件做合适的事**。

### 为什么纯 LLM 不够

| 问题 | 纯 LLM 的表现 | Compound AI 的解决方式 |
|------|-------------|---------------------|
| **幻觉** | 可能推荐不存在的架构风格 | 知识库限定候选范围 |
| **不一致** | 同一需求两次问结果可能不同 | 规则引擎输出确定性强 |
| **不可解释** | "我觉得..." | 每步评分标注来源 |
| **成本高** | 每次调用消耗 Token | 缓存 + 规则 only 模式 |
| **单点故障** | LLM 挂了全挂 | 每节点独立降级 |

### 为什么规则引擎放在最底层

因为规则引擎是**唯一永远不降级的模块**。它像房子的地基——即使断电（LLM 不可用）、断网（Neo4j 不可用），房子还能站着。

规则引擎做的事情非常明确：
- 关键词匹配 → 确定性结果
- 专业规则 → 编码了架构领域的最佳实践
- 零外部依赖 → 不依赖任何网络服务

### 为什么 LLM 放在最上层

LLM 做它最擅长的事——语义理解。但它被限制在：
- **只在规则覆盖不足时介入**（≤2 个维度命中，说明需求太模糊或太隐晦）
- **只做增强不做主导**（投票 +1 分，不影响规则引擎的基础分）
- **每次调用都有 fallback**（超时/报错 → 降级到规则模板）

## 面试官可能怎么问

**Q1: 什么是 Compound AI System？用你的项目举例。**

> Compound AI System 的核心思想是"让合适的组件做合适的事"，而不是把一切交给一个 LLM。我的项目就是典型的 Compound AI：
>
> - **规则引擎**做基础特征提取和评分——因为它需要确定性、零延迟、始终可用
> - **知识图谱**做关系推理——因为它需要结构化的知识表示
> - **LLM**做语义补全和摘要生成——因为它擅长处理模糊的自然语言
>
> 最重要的是，这三层是**主从关系而非平等关系**——规则引擎主导，图谱和 LLM 增强。这样即使后两层不可用，核心链路也不受影响。

**Q2: 你怎么判断什么任务给规则、什么给 LLM？**

> 三个标准：
>
> 1. **确定性**：特征提取需要确定性的关键词匹配（"高并发" → high_concurrency = True），这不能交给 LLM——LLM 可能今天说是、明天说不是
> 2. **可解释性**：架构评分必须可追溯。我的报告里每一条推荐理由都标注来源——"标签匹配 +2"、"规则加分 +1"、"LLM 投票 +1"。如果全部交给 LLM，你只能说"因为 LLM 觉得..."
> 3. **成本与可靠性**：LLM 调用有延迟、有成本、可能失败。规则引擎毫秒级响应、零成本、永不下线。把高频、确定性的任务交给规则，把低频、模糊性的任务交给 LLM

**Q3: 如果让你重新设计，你会调整哪些职责分配？**

> 目前 LLM 的特征补全触发条件是"规则命中 ≤ 2 维"，这个阈值是基于直觉设定的。如果有足够的标注数据（需求文本 + 人工标注的特征标签），我会考虑用一个小型的分类模型替代关键词词典，这样在"模糊匹配"和"准确性"之间可能有更好的平衡。

> 设计 Compound AI 系统架构：规则引擎主导（7 条架构规则 + 10 维关键词词典）保证可解释性和确定性，Neo4j 知识图谱提供结构化关系推理，LLM 作为语义增强和投票——三层协同，各司其职。
