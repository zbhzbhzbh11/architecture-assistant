## 为什么需要这个项目

### 一个真实的困境

假设你是一个初入职场的软件工程师，你的团队接了一个新项目——"开发一个支持万人同时在线的即时通讯系统"。

你打开 ChatGPT，输入："我的系统用什么架构好？"

ChatGPT 可能会回答："建议使用微服务架构，因为它具有良好的可扩展性..."

但这个回答有几个问题：
1. **你不知道它为什么这么说**——是拍脑袋还是有依据？
2. **你不知道还有哪些选项**——微服务之外还有事件驱动、分层架构、管道过滤器...
3. **你不知道代价是什么**——每种架构有什么风险？选了微服务之后要注意什么？
4. **下次换一个需求，你还是得再问一次**——因为它没有结构化推理过程

这就是本项目要解决的问题：**把架构推荐从"拍脑袋"变成可追溯、可解释、可复现的推理过程**。

### 课程要求的拉力

本项目来自《软件体系结构》课程大作业，核心要求是构建一个 **LLM + 知识图谱双驱动的架构风格推荐系统**。具体要求覆盖 9 项技术建议：

| # | 技术建议 | 含义 |
|---|---------|------|
| 1 | LLM + 知识图谱双驱动 | 规则引擎、LLM、图谱三者协同 |
| 2 | LangChain / LangGraph Agent 协作 | 使用 LangGraph 编排多个 Agent |
| 3 | Neo4j 知识图谱存储 | 图数据库存储架构知识 |
| 4 | Few-shot Prompt Engineering | 通过示例引导 LLM 输出 |
| 5 | 规则引擎校验 | 规则引擎主导评分，LLM 辅助 |
| 6 | LLM 结果缓存 | 缓存请求结果以节省 Token |
| 7 | ADR 架构决策溯源 | 每次推荐自动生成决策记录 |
| 8 | 架构模式组合推荐 | 支持多风格组合推荐 |
| 9 | 架构重构建议 | 检测架构坏味并推荐重构模式 |

**面试时这句话可以直接用**：

> "这个项目来源于软件体系结构课程的综合性大作业，要求构建一个 LLM + 知识图谱双驱动的架构推荐系统。我不仅完成了全部 9 项技术建议，还额外做了降级设计、组合推荐、重构建议等增强功能，43 项自动验收 100% 通过。"

## 当前项目如何实现

### 一句话定位

> 自然语言需求输入 → 特征提取 → 规则+图谱匹配 → LLM 评估 → 可解释的决策报告

### 和普通 LLM 聊天机器人的区别

| 维度 | 普通 LLM 聊天 | 本项目 |
|------|-------------|--------|
| **推理方式** | 纯 LLM 黑盒生成 | 规则引擎主导 + 图谱推理 + LLM 增强 |
| **可解释性** | "因为我觉得..." | 每条推荐标注命中标签和评分规则 |
| **可靠性** | LLM 挂了就全挂 | LLM 不可用自动降级，核心链路不受影响 |
| **知识来源** | 训练数据中模糊记忆 | 结构化知识库 + 专家规则 |
| **输出格式** | 自由文本 | 结构化报告（推荐+对比矩阵+风险+ADR） |
| **决策追溯** | 无 | 每次推荐自动生成 ADR 决策记录 |

## 面试官可能怎么问

**Q1: 简单介绍一下你做的这个项目。**

> 这是一个 LLM + 知识图谱双驱动的架构风格智能助手。用户输入软件需求描述，系统自动完成特征提取、风格匹配和评估决策，输出包含推荐理由、对比矩阵、风险分析的完整报告。
>
> 系统的核心设计理念是"规则引擎主导 + LLM 增强 + 知识图谱推理"，不是简单地调用 LLM。6 个 FastAPI 微服务通过 LangGraph 编排，LLM 不可用时自动降级保证核心链路高可用。

**Q2: 为什么不直接问 ChatGPT？**

> 四个原因。第一是**可解释性**：我的系统每条推荐都有评分明细——标签命中加几分、规则加分加几分、图谱加分加几分，ChatGPT 做不到这一点。第二是**可靠性**：LLM 服务可能波动或超时，我的系统在每个节点都有降级路径，LLM 不可用时核心推荐链路不受影响。第三是**结构化输出**：我输出的是标准的决策报告——推荐风格、对比矩阵、风险分析、ADR 记录——不是一段自由文本。第四是**可追溯**：每次推荐自动生成 ADR 决策记录。

**Q3: 9 项技术建议你全部完成了吗？**

> 全部完成，43 项自动验收 100% 通过。包括 LLM + 知识图谱双驱动、LangGraph 编排、Neo4j 存储、Few-shot Prompt、规则引擎、LLM 缓存、ADR 溯源、组合推荐、重构建议。每一项都有对应的实现文件和测试覆盖。
