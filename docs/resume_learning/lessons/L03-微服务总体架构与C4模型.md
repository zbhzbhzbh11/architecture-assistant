## 为什么需要微服务架构

### 一个"分诊台"的类比

想象一个医院的挂号处。如果所有病人都在一个窗口排队——挂号、看病、取药、缴费都在同一个柜台——那就是**单体架构**。

而现代医院的做法是：挂号处 → 分诊台 → 各科室。每个科室独立运作，通过统一的导诊系统协同。这就是**微服务架构**。

本系统的"分诊台"就是 API Gateway，"各科室"就是 6 个 Agent 微服务。

### 为什么本项目选微服务

从 [架构设计文档 §2.1](../docs/02-架构设计文档.md) 中提取的 5 个理由：

| 维度 | 分析 |
|------|------|
| **领域职责解耦** | 需求提取（词典规则）、规则匹配（评分计算）、LLM 评估（外部调用）各属不同领域 |
| **异构能力集成** | requirements-agent 是纯计算密集型，evaluation-agent 是 IO 密集型（等 LLM 响应），拆分后可独立优化 |
| **故障隔离** | LLM 调用有外部依赖风险，独立部署 evaluation-agent 后，LLM 波动不影响其他模块 |
| **技术栈灵活** | matching-agent 可选升级为 Neo4j 驱动，其他 Agent 不受影响 |
| **部署独立性** | 每个服务独立构建、独立运行，Docker Compose 按依赖顺序启动 |

### 容器职责速查

| 容器 | 端口 | 核心职责 | 依赖 |
|------|------|---------|------|
| **frontend** | 3000→80 | Web 交互，11 区结果展示 | api-gateway |
| **api-gateway** | 8000 | 请求编排 + 缓存 + 汇总 | 全部 agent |
| **requirements-agent** | 8001 | 关键词匹配 + LLM 补全 | LLM (可选) |
| **matching-agent** | 8002 | 规则评分 + 图谱融合 + 组合 | knowledge-base |
| **evaluation-agent** | 8003 | LLM 投票/摘要 + ADR 生成 | LLM (可选), knowledge-base |
| **knowledge-base** | 8004 | 风格/组合/ADR/反馈 数据服务 | Neo4j (可选) |
| **refactoring-agent** | 8005 | 坏味检测 + 重构模式推荐 | LLM (可选) |
| **neo4j** | 7474/7687 | 图数据库 | 无 |

### 服务间通信：同步 HTTP 的取舍

所有 Agent 间通信都是同步 HTTP（httpx）。这不是"最佳"选择，而是"最合适当前场景"的选择：

| 因素 | 分析 |
|------|------|
| **请求量** | 课程演示场景，QPS ≈ 1，不需要消息队列 |
| **调试** | 同步调用链路清晰，便于排查问题 |
| **延迟** | 单次请求串行 < 5s（含 LLM），在可接受范围 |
| **一致性** | 推荐请求需要完整响应，天然适合请求-响应模式 |

**面试时如果被问**："为什么不用消息队列？"

> 当前场景是一个同步的推荐请求——用户提交需求，等待几秒拿到完整报告。消息队列适合异步场景（如订单处理、通知推送），如果未来需要异步处理大量推荐请求（如批量架构评估），可以在 api-gateway 和 Agent 之间插入消息队列——但架构已预留了这种扩展空间。

## 面试官可能怎么问

**Q1: 你自己的系统用了什么架构？为什么？**

> 微服务架构。我有 6 个独立容器，每个处理不同领域的逻辑——需求提取、风格匹配、LLM 评估、知识库、重构建议。选择微服务的原因：第一，这几个模块的领域职责差异很大；第二，LLM 调用是外部依赖，独立部署可以实现故障隔离；第三，各模块的计算特征不同（CPU 密集 vs IO 密集），独立部署方便优化。

**Q2: 画一下你的系统架构图。**

> （在白板上画出上面的 C4-Container 图，按以下顺序讲解）
>
> 左边是用户和前端，中间是 API Gateway，右边三列分别是三个核心 Agent。API Gateway 是编排层——它用 LangGraph 把三个 Agent 串成一条工作流。matching-agent 下面连接 knowledge-base，knowledge-base 下面有两种存储——Neo4j 图数据库（左边）和 JSON 文件（右边）。refactoring-agent 是独立调用的——在主链路完成后异步触发，失败不阻塞。

## 简历上如何表达

> 基于 FastAPI 构建 6 个独立微服务（需求提取 / 规则匹配 / LLM 评估 / 知识库 / 重构建议），Docker Compose 编排，服务间通过 httpx 同步通信并实现故障隔离；使用 LangGraph StateGraph 编排 Agent 协作流程，LangGraph 不可用时自动回退到手写编排。
