## 为什么需要组合推荐

### 一个"药方配伍"的类比

中医开药不会只开一味药——发汗的配清热的，补气的配活血的。单一药材力量不够，但组合起来疗效倍增。

架构风格也一样。举几个真实场景：

- **电商平台**：订单支付需要强一致事务（Microservices 的 Saga），但双十一流量洪峰需要异步削峰（Event-Driven 的事件总线）→ **Microservices + Event-Driven**
- **银行系统**：需要完整的审计追踪（Event Sourcing），又需要读写分离以提升查询性能（CQRS）→ **CQRS + Event Sourcing**
- **日志分析平台**：数据需要经过多阶段流水线处理（Pipeline-Filter），各阶段间通过消息队列异步触发（Event-Driven）→ **Pipeline-Filter + Event-Driven**

单一风格推荐足够应付大多数场景。但当一个需求同时要求"高并发 + 强一致"或"多团队 + 数据密集"时，组合推荐能给出更精准的方案。

## 当前项目如何实现

### 5 种预定义组合模式

来源文件：[knowledge_base/data/architecture_combinations.json](../services/knowledge_base/data/architecture_combinations.json)

| 组合 | 复杂度 | 核心适用场景 |
|------|--------|------------|
| **Microservices + Event-Driven** | 低 (1) | 高并发分布式系统 |
| **Layered + CQRS** | 低 (1) | 读写不均衡的企业应用 |
| **Pipeline-Filter + Event-Driven** | 中 (2) | 流处理/日志分析平台 |
| **Hexagonal + Microservices** | 中 (2) | 领域驱动的微服务 |
| **CQRS + Event Sourcing** | 高 (3) | 完整审计的金融系统 |

**每种组合模式包含**：
- 组成风格（`styles` 列表）
- 关联特征标签（`tags`）
- 协同效应说明（`synergy` / `synergy_zh`）
- 复杂度惩罚（`complexity_penalty`，1-3 级）
- Mermaid 拓扑图

### 组合评分公式

```python

def score_combination(combo, scored_styles, features, graph_evidence):

    base_sum = sum(c.get("score", 0) for c in components)

    coverage_bonus = len(covered_features) * 1

    # ③ 图谱 CAN_COMBINE_WITH 加分：图谱确认可组合 +2/条
    graph_bonus = ...  # 来自 Neo4j 的 COMPLEMENTS 关系

    # ④ 复杂度惩罚：1-3 分
    penalty = combo.get("complexity_penalty", 1)

    # 最终分
    final_score = base_sum + coverage_bonus + graph_bonus - penalty
```

**举例**：即时通讯场景（高并发 + 实时 + 可扩展）

```
Microservices + Event-Driven:
  base_sum = MS 6分 + EDA 8分 = 14分
  coverage = 高并发+实时+可扩展 = 3分
  graph_bonus = 2分 (Neo4j 确认 COMPLEMENTS 关系)
  penalty = -1分 (低复杂度)
  final = 14 + 3 + 2 - 1 = 18分 ✅ 胜出

Layered + CQRS:
  base_sum = LA 3分 + CQRS 4分 = 7分
  coverage = 2维 = 2分
  graph_bonus = 0分
  penalty = -1分
  final = 7 + 2 + 0 - 1 = 8分
```

### 复杂度惩罚的设计

```
低复杂度 (1): Microservices + EDA, Layered + CQRS
中复杂度 (2): Pipeline-Filter + EDA, Hexagonal + MS
高复杂度 (3): CQRS + Event Sourcing
```

**为什么 CQRS + Event Sourcing 惩罚最高？** 因为引入事件溯源意味着整个系统的状态管理方式彻底改变——你需要事件存储、事件重放机制、CQRS 的双数据库、最终一致性处理。这是架构中最激进的重构之一，必须谨慎推荐。

**惩罚分不是"不推荐"，而是"让推荐更保守"**——CQRS + Event Sourcing 只有在其组成风格本身就非常高分（说明需求确实强匹配）时才能胜出。

### 组合评分的局限（诚实标注）

| 局限 | 说明 |
|------|------|
| **预定义而非动态** | 5 种组合是手写的，不会自动发现新组合 |
| **复杂度惩罚是主观的** | 1-3 级的划分基于经验，没有定量依据 |
| **组合数量有限** | 10 种风格理论上可以有 C(10,2) = 45 种组合，只覆盖了 5 种 |

## 面试官可能怎么问

**Q1: 组合推荐和单一推荐是什么关系？会不会冲突？**

> 不冲突。组合推荐是在单一推荐基础上的增强——先用单一评分算出每个风格的分数，再用这些分数去评组合。报告里两者同时呈现：单一推荐（核心推荐 + 备选）在最前面，组合推荐作为"进阶建议"。用户可以先理解单一风格，再考虑是否采纳组合。

**Q2: 为什么只有 5 种组合？不应该是 45 种吗？**

> 10 种风格理论上可以有 45 种两两组合——但架构不是乐高积木，不是随便两个风格拼在一起就有意义。比如"Pipeline-Filter + Layered"——两者都在自己的领域内工作，组合在一起会产生严重的抽象冲突。5 种组合是我根据架构文献和实际工程经验挑选的"经过验证的组合模式"——它们的协同效应有充分的理论和实践基础。

**Q3: 复杂度惩罚的 1-3 分是怎么定的？**

> 1 分（低）：两种风格有天然的接口边界（如 MS 管理服务边界，EDA 管理服务间通信），组合不会增加太多运维负担。3 分（高）：CQRS + Event Sourcing 涉及事件存储、双数据库、最终一致性处理——带来的技术和运维负担是显著的。这确实是主观判断，但基于业界共识。如果要做更客观的评估，可以引入技术债务量化模型。
