## STAR 面试法速查

| 要素 | 含义 | 时间占比 |
|------|------|---------|
| **S**ituation | 当时的情境和背景是什么？ | 10% |
| **T**ask | 你需要解决什么问题？目标是什么？ | 10% |
| **A**ction | 你具体做了什么？用了什么方法？ | 60% |
| **R**esult | 最终效果如何？学到了什么？ | 20% |

**黄金比例**：S(10%) → T(10%) → A(60%) → R(20%)

**核心原则**：
- A 部分必须用"**我**"来描述自己的贡献，不能用"我们"
- R 部分尽量量化——数字比形容词有说服力
- 每个 STAR 建议控制在 1-2 分钟，追问再展开

## 场景 1：为什么选择微服务架构

### S - 情境

这个项目来自软件体系结构课程大作业，要求构建一个 LLM + 知识图谱双驱动的架构推荐系统。用户输入一段自然语言需求，系统自动完成特征提取、风格匹配、评估决策，最终输出可解释的推荐报告。

### T - 任务

我需要为这个系统选择一个合适的架构风格。关键约束是：需求提取依赖关键词词典（纯计算），风格匹配依赖规则引擎和 Neo4j（数据访问），评估决策依赖外部 LLM API（网络调用）。这三种负载的计算特征完全不同，如果放在一个进程里，CPU 密集的任务会阻塞 IO 密集的任务。

### A - 行动

我选择了微服务架构，并做了四个具体的设计决策：

**1. 按领域职责拆分 6 个独立服务**：requirements-agent 负责特征提取（纯计算），matching-agent 负责规则匹配和知识库查询（数据密集型），evaluation-agent 负责 LLM 调用（IO 密集型），knowledge-base 负责数据存储（数据服务），refactoring-agent 负责重构建议，api-gateway 负责编排。每个服务独立 FastAPI 容器。

**2. 将 LLM 调用隔离在独立服务中**：evaluation-agent 和 requirements-agent 是仅有的两个调用 LLM 的服务。它们各自有自己的降级路径——LLM 不可用时自动切换到规则模板，不会因为 LLM 波动影响其他服务的稳定性。

**3. 引入 API Gateway 作为编排层**：使用 LangGraph StateGraph 建模 Agent 工作流（extract → match → evaluate → trace），LangGraph 不可用时自动回退到手写 httpx 编排。Gateway 不参与业务逻辑，只负责调度。

**4. 通过 Docker Compose 管理服务依赖**：7 个容器通过 depends_on 定义启动顺序，通过环境变量传递服务地址，使得每个服务零硬编码。

### R - 结果

最终的系统架构清晰可维护——每个服务的代码量都在 100-400 行之间，职责单一。最直接的好处体现在故障隔离上：在多次演示中，LLM 服务偶尔超时，但只有 evaluation-agent 的报告文本受影响（降级到规则模板），核心推荐结果不受任何影响。答辩时我直接用 `docker compose ps` 展示了 7 个容器的运行状态，评委一眼就能看到系统结构。

### 面试官追问：微服务是不是杀鸡用牛刀？

> 从请求量看，确实——这个系统的 QPS 是 1，6 个容器看起来有点多。但微服务的价值不只是"扛并发"，更重要的是**领域解耦和故障隔离**。你可以想象如果全部写在一个 FastAPI 应用里：某个 LLM 调用超时导致整个线程阻塞，连关键词提取都做不了。而拆分后，LLM 挂了只影响 evaluation-agent，requirements-agent 和 matching-agent 仍然正常工作。另外，对于课程项目而言，清晰的模块边界本身就很重要——它让代码评审和答辩讲解变得容易。

### 简历对应 Bullet

> 基于 FastAPI 构建 6 个独立微服务，通过 Docker Compose 编排，实现故障隔离和独立部署；使用 LangGraph StateGraph 编排 Agent 工作流，LangGraph 不可用时自动回退手写编排。

## 场景 2：如何设计 Multi-Agent 协作

### S - 情境

系统有三个核心 Agent——需求提取、风格匹配、评估决策——它们之间有严格的先后依赖关系：先提取特征才知道匹配什么，先匹配出候选才知道评估哪个更好。我需要在保持"步骤清晰"和"容错灵活"之间找到平衡。

### T - 任务

设计一个编排机制，满足三个要求：（1）清晰定义每一步的输入输出契约；（2）每一步的耗时和状态可追踪；（3）编排引擎本身不能成为强依赖——如果编排框架出问题，系统还能正常运行。

### A - 行动

我设计了双引擎编排架构：

**1. LangGraph 引擎（优先）**：使用 StateGraph 建模 4 节点工作流。每个节点是一个异步函数，接收全局状态（TypedDict），返回部分更新。LangGraph 自动管理状态传递——节点不需要知道上游是谁、下游是谁。

**2. 手动 httpx 引擎（兜底）**：约 50 行代码实现了与 LangGraph 完全等价的三步串行调用。每个步骤记录耗时和状态到 trace 列表。

**3. TypedDict 状态契约**：定义了 ArchitectureWorkflowState——requirement（输入）、extracted_features（extract 输出）、candidates（match 输出）、final_report（evaluate 输出）、errors 和 trace（可观测）。所有节点读写同一个状态字典，字段拼写由类型系统校验。

**4. 两步 fallback**：启动时如果 `import langgraph` 失败，`build_workflow()` 返回 None，系统标记为 `WORKFLOW_ENGINE = "manual"`。运行时如果 LangGraph 编排抛出异常，`try/except` 捕获后切换到手动编排。

### R - 结果

双引擎设计的直接好处是灵活性。我的本地开发环境没装 langgraph，系统自动使用手动编排——功能完全一致，只是 `workflow_engine` 字段是 "manual" 而不是 "langgraph"。Docker 里装了 langgraph，自动切换到 StateGraph 编排。答辩评委问到"LangGraph 是必需的吗"，我直接演示了注释掉 import 后系统正常运行——workflow_trace 记录完全等价。

### 面试官追问：你的 Agent 会不会自己决定调用顺序？

> 不会。当前的编排是固定顺序的线性 DAG——extract → match → evaluate → trace。Agent 之间没有自主协商或动态路由。这是有意为之——在主推荐链路上，步骤顺序是确定的。但 LangGraph 框架本身支持条件边——如果未来需要根据置信度决定是否走人工审批流程，只需把 `add_edge` 改成 `add_conditional_edges`，节点代码不需要改动。

### 简历对应 Bullet

> 使用 LangGraph StateGraph 建模 4 节点 Multi-Agent 工作流（extract→match→evaluate→trace），TypedDict 定义全局状态契约；LangGraph 不可用时自动回退手写编排，保证零强依赖。

## 场景 3：如何避免 LLM 幻觉

### S - 情境

LLM 有一个众所周知的问题——"幻觉"：它可能会推荐不存在的架构风格、编造不存在的原因、甚至输出不符合 JSON 格式的自由文本。对于一个需要可靠推荐的系统来说，这是绝对不能接受的。

### T - 任务

设计一套机制，让 LLM 在系统中"没法犯错"——即使它产生了不符合预期的输出，也不会污染最终的推荐结果。

### A - 行动

我从四个层面构建了"幻觉防火墙"：

**1. 候选空间限定**：LLM 只能从已有的 10 种架构风格中选择。投票函数 `llm_vote_style()` 的关键代码是：
```python
if text in style_names:   # 只有精确匹配预定义列表才接受
    return text
return None               # 编造的风格名直接丢弃
```

**2. JSON 格式强制 + 容错解析**：Few-shot Prompt 明确要求"返回严格的 JSON 格式，不要输出其他内容"。即使 LLM 多输出了 markdown 代码块包裹（\`\`\`json ... \`\`\`），代码也会自动去除包裹后再解析。解析失败则静默降级。

**3. 规则引擎兜底**：LLM 的输出只是"加分"而非"定论"。规则引擎 `score_style()` 是评分的绝对基础——标签匹配分和规则分构成主体，LLM 投票只能加 1 分（tie-break），无法推翻规则引擎的结论。LLM 摘要失败时，`_fallback_summary()` 用规则理由拼出结构完整的四段式中报告。

**4. Temperature 控制**：投票用 temperature=0.0（需要确定性），摘要用 temperature=0.3（允许一定表达多样性但不能太放飞）。两者都通过显式参数控制，不依赖默认值。

### R - 结果

这四层防护的实际效果是：即使在最极端的情况下——LLM 输出完全不可解析——系统仍然能返回完整的、可读的推荐报告。我专门做了一轮"LLM 全关"测试：注释掉 `.env` 中的 LLM 配置，跑全部 20 条回归用例，结果全部通过 5 维验证。推荐的风格和配 LLM 时一致（因为规则引擎主导评分），只是报告的"自然语言摘要"变成了"规则模板生成的结构化中文"。

### 面试官追问：你的规则引擎本身会不会有偏见？

> 会。比如"高并发 → Event-Driven +1"这条规则——在有些场景下，Microservices 配合消息队列也能很好地处理高并发。规则引擎的"偏见"本质上是架构经验的编码——它反映的是"大多数情况下的最佳实践"，但不排除特殊情况。这就是为什么 LLM 投票和评估摘要仍然有意义——它们可以在规则覆盖不足或规则之间有冲突时提供辅助判断。另外，学习权重机制（基于用户反馈动态调整）也是长期缓解规则偏见的路径。

### 简历对应 Bullet

> 构建四层 LLM 幻觉防护：候选空间限定 + JSON 格式强校验 + 规则引擎兜底 + Temperature 严格管控；LLM 输出不可解析时自动降级，核心推荐不受影响。

## 场景 4：如何从 JSON 知识库升级到 Neo4j 知识图谱

### S - 情境

项目最初的知识库是纯 JSON 文件存储——10 种架构风格，每种风格包含 tags/pros/cons/best_for 等字段。JSON 存储结构简单、零依赖，满足了早期快速迭代的需求。但随着课程要求明确提出"Neo4j 知识图谱存储"，我需要引入图数据库。

### T - 任务

引入 Neo4j 知识图谱，同时满足三个约束：（1）Neo4j 不可用时系统不能崩溃；（2）两种存储的数据必须保持一致；（3）切换存储方式不应该影响上层 Agent 的代码。

### A - 行动

我设计了一套"双后端 + 统一调度"的存储层架构：

**1. 统一接口**：GraphRepository 和 JsonRepository 实现完全相同的方法签名——`get_styles()`、`add_style()`、`get_feedback()`、`add_adr()`、`get_combinations()`、`graph_match()` 等。上层调用者不关心底层是图还是 JSON。

**2. 三级切换策略**：通过 `KNOWLEDGE_BACKEND` 环境变量控制：
- `json`：始终用 JSON，零外部依赖
- `neo4j`：始终用 Neo4j，不可用返回 503
- `auto`：优先 Neo4j，不可用自动 fallback JSON

**3. 统一调度函数** `_repo()`：每次调用先判断 `_prefer_graph()`，优先走 GraphRepository；返回 None（Neo4j 不可用）且不是强制模式时，自动 fallback 到 JsonRepository。

**4. 图模型设计**：6 节点类型（ArchitectureStyle、QualityAttribute、Scenario、Risk、ADR、Feedback）+ 6 关系类型（HAS_QUALITY、SUITABLE_FOR、HAS_RISK、COMPLEMENTS、RECOMMENDS、BASED_ON）。初始化脚本从 JSON 读取数据并在 Neo4j 中创建对应节点和关系。

**5. 图谱推理增强**：matching-agent 的 `blend_scores()` 函数将图谱证据与规则评分融合——每个通过 HAS_QUALITY 匹配到的质量属性 +2 图谱分，但总分上限不超过规则分的 50%。

### R - 结果

这套设计的实际效果是：答辩时我可以先展示 Neo4j 模式下的完整图谱推理（`neo4j://localhost:7474` 浏览器界面），然后切到 `KNOWLEDGE_BACKEND=json` 模式，系统照样正常运行——前后端代码零改动。matching-agent 完全不知道底层存储切换了，因为 `_repo()` 函数透明地处理了 fallback。

### 面试官追问：JSON 和 Neo4j 的数据一致性怎么保证？

> 当前实现的保证方式是"JSON 为主、Neo4j 同步"。初始化时从 JSON 读取数据写入 Neo4j；新增风格时同步写入两边（通过 `_repo()` 调度）。但如果一边成功一边失败（比如 Neo4j 写入超时），当前没有分布式事务保证。这是一个已知局限——在课程项目的规模下（数据量小、写入频率低），这种不一致的概率极低。如果要在生产环境用，可以引入 Outbox 模式或事件溯源来保证最终一致性。

### 简历对应 Bullet

> 设计 Neo4j 知识图谱模型（6 节点 + 6 关系）+ JSON fallback 双后端架构；通过 KNOWLEDGE_BACKEND 环境变量支持 json/neo4j/auto 三种存储模式，Neo4j 不可用时透明切换 JSON。

## 场景 5：如何设计规则引擎 + LLM 混合推理

### S - 情境

架构推荐面临一个核心矛盾：纯规则引擎可解释但僵硬（只能匹配预定义的关键词和规则），纯 LLM 灵活但不可靠（可能幻觉、结果不一致）。我需要找到两者之间的平衡点。

### T - 任务

设计一个混合推理架构，让规则引擎和 LLM 各自发挥优势——规则引擎保证可解释性和确定性，LLM 处理规则覆盖不到的模糊语义。

### A - 行动

我的设计可以概括为"规则主导、LLM 辅助、图谱增强"的三层架构：

**1. 规则引擎为基础（始终执行）**：
- requirements-agent：10 维关键词词典（约 90 个中英文词）+ 6 种否定模式。匹配结果是确定性的。
- matching-agent：`score_style()` 函数——标签匹配 +2/项、7 条硬编码架构规则 +1/条、学习权重 +1（≥2 次确认）。每条规则编码一条架构领域的经验法则。
- evaluation-agent：规则排序 + `_fallback_summary()` 降级摘要。LLM 不可用时依然能输出完整报告。

**2. LLM 为增强（条件触发）**：
- 语义补全：仅在规则命中 ≤ 2 维时触发——因为这说明需求表述模糊，关键词覆盖不足
- 投票：temperature=0.0，只加 1 分——只能做 tie-break，无法推翻规则排序
- 摘要：temperature=0.3，生成自然语言报告——但规则模板是兜底

**3. 知识图谱为桥梁**：
- Neo4j 的 HAS_QUALITY 关系填补了"特征 → 风格"之间规则未覆盖的隐式关联
- 加分上限 50%——让图谱"增强但不主导"评分

**4. 融合策略**：在三层各自评分后，最终决策是"规则排序 + 图谱加分（capped）+ LLM 投票（+1）"。LLM 的权重被刻意压到最低——防止它成为事实上的唯一决策者。

### R - 结果

这个三层架构最直接的价值体现在"审计性"上。答辩时评委问："为什么推荐 Event-Driven 而不是 Microservices？"我直接展示评分明细：标签匹配 3 项 +6 分，规则"高并发有利于事件驱动"+1 分，图谱证据+2 分（capped 到 3 分），总计 10 分。而 Microservices 只有标签匹配 1 项 +2 分，规则 +1 分，总共 3 分。评分完全可追溯——这就是规则引擎主导的价值。

### 面试官追问：如果规则引擎和 LLM 给出完全相反的结论怎么办？

> 规则引擎胜出。这是设计原则——确定性高于模糊性。LLM 投票只加 1 分，在规则引擎已经拉开差距的情况下（比如 Event-Driven 比 Microservices 高 4+ 分），LLM 即使投给 Microservices 也无法改变最终排序。这和自动驾驶的"规则优先于 AI"是同样的理念——当安全规则和 AI 判断冲突时，规则优先。

### 简历对应 Bullet

> 设计三层混合推理引擎：规则引擎主导评分（10 维关键词 + 7 条架构规则），Neo4j 图谱增强推理（加分上限 50%），LLM 语义补全与投票（仅 tie-break）；三层评分完全可追溯。

## 场景 6：如何处理 LLM 不可用和系统降级

### S - 情境

这个系统依赖外部 LLM API——可能是 DeepSeek、通义千问或 OpenAI 兼容的任何服务。外部 API 有三个不可控因素：可能超时、可能返回错误、可能根本没配置。但课程答辩时不能因为"LLM 今天挂了"就演示失败。

### T - 任务

设计全链路的降级策略，确保在任何 LLM 故障场景下，系统的核心推荐功能都能正常运行，输出的报告仍然结构完整、信息准确。

### A - 行动

我逐一审查了全链路所有 LLM 调用点，为每一个设计了独立的 fallback：

**1. requirements-agent 语义补全**：触发条件是"规则命中 ≤ 2 且 LLM 已配置"。如果 LLM 没配或超时，直接返回纯规则特征——可能漏判某些维度，但不会误判。

**2. evaluation-agent LLM 投票**：`llm_vote_style()` 返回 None 时（超时/报错/输出不合法），规则排序结果不变——只是没有 tie-break 加分。

**3. evaluation-agent LLM 摘要**：`llm_summary()` 失败时，调用 `_fallback_summary()`——用规则引擎的命中理由 + 风格的 pros/cons + 风险模板，拼出一份结构完整的四段式中文报告。和 LLM 版本相比，只是表述更"模板化"，信息完全等价。

**4. refactoring-agent LLM 润色**：`llm_polish()` 返回 None 时，直接使用规则模板——迁移步骤和风险描述由 `build_rule_template()` 生成，信息不缺失。

**5. LangGraph 编排**：ImportError 或运行时异常 → 手动 httpx 编排。功能完全等价。

**6. Few-shot Prompt**：ImportError → 零样本 Prompt。质量可能略降，功能不变。

**7. ADR 写入**：try/except 包裹，失败标记 `adr_status: "failed"`——不阻塞推荐。

**8. 重构建议调用**：api-gateway 用 `timeout=8.0` + try/except，失败返回空 `refactoring_advice: {}`。

### R - 结果

最直接的验证：我专门做了一轮"全链路 LLM 关闭"测试——注释掉 `.env` 中的所有 LLM 配置，然后跑全部 20 条回归用例。结果：20 条全部通过 5 维验证（候选数/主流覆盖/推荐完整性/可解释性/对比矩阵）。推荐风格和配 LLM 时完全一致，因为规则引擎主导评分。唯一的变化是报告文字的"自然度"——但这正是设计的预期行为。

答辩评审时，我现场演示了两次推荐：一次配 LLM，一次不配。核心推荐结果一样，评委直接对比了两份报告的差异——确认了降级机制的真实性。

### 面试官追问：所有 LLM 调用点都挂了，你的报告还剩下什么？

> 还剩完整的核心信息：推荐风格（基于规则评分）、备选架构、3 × 3 对比矩阵（得分/优缺点/理由）、针对推荐风格的风险分析（3 种专属模板 + 通用模板）、ADR 决策记录。缺少的是 LLM 的自然语言摘要（换成规则模板的结构化中文）和 LLM 投票加分（权重本就只有 +1 分）。从用户角度看，这份报告可能不够"生动"，但完全够用——该有的信息结构一项不少。

### 简历对应 Bullet

> 全链路 LLM 降级设计：8 个 LLM 调用点均配备独立 fallback 路径；LLM 完全不可用时系统自动切换纯规则模式，核心推荐链路不受影响，20 条回归用例全部通过。

## 场景 7：如何实现 ADR 决策溯源

### S - 情境

架构推荐是一个"决策"行为。如果没有任何记录，一段时间后就没人记得"为什么当时选了 Event-Driven 而不是 Microservices"。这在软件工程中叫"决策失忆"——是架构腐化的重要原因之一。

### T - 任务

实现 ADR（Architecture Decision Record）的自动生成和持久化，记录每次推荐的关键决策信息（选了谁、为什么、还有什么备选、有什么风险）。同时要求写入 ADR 不能影响推荐的响应速度。

### A - 行动

**1. ADR 自动触发**：在 evaluation-agent 的 `evaluate()` 函数中，推荐报告生成完成后，自动向 knowledge-base 发起 ADR 写入请求。一条 ADR 包含：用户需求、提取的特征信号、Top 3 候选风格、最终推荐、备选方案、决策依据（规则理由 + LLM 摘要 + LLM 投票）、风险分析、图谱证据。

**2. ADR ID 设计**：`ADR-YYYYMMDD-NNN` 格式——日期前缀便于按时间检索，三位序号保证同一天内的唯一性。

**3. 双存储**：JSON 文件为主存储（`adr_records.json`），Neo4j 同步为可选增强。在 Neo4j 中，ADR 节点通过 RECOMMENDS 关系连接到 ArchitectureStyle、通过 BASED_ON 关系连接到 QualityAttribute——成为关联需求、风格和属性的枢纽节点。

**4. 非阻塞容错**：ADR 写入用 try/except 包裹，超时 5 秒。写入失败只记录 `adr_status: "failed"`，不抛异常、不影响推荐报告返回。

**5. 查询 API**：`GET /adr?limit=20` 列出最近的决策记录，`GET /adr/{adr_id}` 查看单条详情。

### R - 结果

ADR 的自动生成让每次推荐都有据可查。答辩时我演示了查询历史 ADR 列表——展示了之前演示的几次推荐的完整决策记录。评委可以直接看到"这次推荐的理由和上次有什么不同"——比如"两次都是即时通讯类需求，但措辞不同导致特征提取有细微差异"。这种"可对比的决策历史"是手写 ADR 难以做到的。

### 面试官追问：你的 ADR 和工业界的 ADR 有什么区别？

> 格式上是兼容的——都包含决策、依据、备选、风险。但工业界 ADR 是手写的、面向关键架构变更（频率低、影响大），我的 ADR 是自动生成的、面向每次推荐请求（频率高）。这实际上降低了 ADR 的使用门槛——你不用等到"重大决策"才写，每次推荐都自动留档。缺点是自动生成的 ADR 缺少人工的"上下文补充"——比如"当时团队只有 3 个人所以选了简单的方案"，这些信息无法自动获取。

### 简历对应 Bullet

> 实现 ADR 自动生成与追溯：每次推荐完成后自动写入决策记录（需求/特征/候选/理由/风险），JSON 主存储 + Neo4j 同步，ADR ID 格式 ADR-YYYYMMDD-NNN；写入失败不阻塞推荐。

## 场景 8：如何通过测试证明系统可靠

### S - 情境

课程大作业有一套明确的验收标准——43 项检查，覆盖目录结构、Agent 数量、知识库完整性、测试用例数、LLM 集成、Web API、可视化等 7 个方面。手动逐条检查效率低、容易遗漏。而且改了代码之后，不确定会不会破坏验收标准。

### T - 任务

建立一套自动化测试体系，覆盖四个层次——函数级（单元）、端到端（回归）、快速筛查（冒烟）、课程要求（自动验收）。每一层都应该是自动化的，任何人（包括评委）都可以一键运行。

### A - 行动

我建立了四层测试体系：

**1. 单元测试（76 条，pytest）**：覆盖全部 6 个模块的核心函数。关键测试包括：
- `test_matching.py`：规则引擎评分（验证 score_style 在各种特征组合下的得分）、图谱融合（验证 blend_scores 的 cap 逻辑）、组合推荐评分
- `test_gateway.py`：缓存键生成（验证相同需求产生相同键）、双引擎切换（验证 langgraph 不可用时正确 fallback）
- `test_evaluation.py`：LLM 投票/摘要的降级逻辑、降级摘要的结构完整性
- `test_knowledge.py`：JSON 存储的 CRUD、Neo4j 可用性检测、ADR 读写
- `test_requirements.py`：关键词匹配、否定过滤、LLM 语义补全的触发条件

单元测试不需要外部服务，可以脱离 Docker 独立运行。

**2. 回归测试（20 条用例，5 维验证）**：`tests/run_regression.py` 读入 20 条需求用例，逐条调 api-gateway，检查 5 个维度——候选风格 ≥ 3、主流风格覆盖、有最终推荐、可解释（有规则理由 + LLM 字段）、对比矩阵 ≥ 3 行。20 条用例覆盖即时通讯、电商、金融、大数据、企业系统、医疗、政务、物联网等 8 大类场景。

**3. 冒烟测试**：`tests/run_smoke.py` 快速验证核心链路可通——部署后或演示前执行，30 秒内出结果。

**4. 自动验收（43 项）**：`scripts/check_assignment.py` 自动扫描项目目录，逐条检查课程要求——文件存在性、Agent 数量、API 端点、LLM 配置、测试用例数等。输出 Markdown + JSON 双格式报告。

### R - 结果

四层测试体系的直接价值：每次改代码后，先在本地跑单元测试（秒级反馈）→ 确认没问题后部署 → 跑冒烟测试 → 跑回归测试（分钟级）。修改任何模块都能立刻知道有没有破坏其他模块的行为。最终 43 项自动验收 100% 通过，20 条回归全部通过，76 条单元测试全部通过。

答辩时评委特意问了"你怎么确保系统是正确的"，我直接展示了自动验收脚本的输出——43 个 [OK] 绿色标记。然后展示了回归测试的 20 × 5 维验证矩阵——所有 √。

### 面试官追问：你的测试覆盖是否足够？

> 对于课程项目来说足够——覆盖了全部模块的核心函数和端到端链路。但对于生产系统来说不够——缺少性能测试（没有压测验证高并发下的行为）、缺少边界测试（如超长输入、特殊字符、空输入）、缺少安全测试（如注入攻击）。我清楚这些缺口——它们正是课程项目和生产系统的区别所在。

### 简历对应 Bullet

> 建立四层自动化测试体系：76 条单元测试（pytest）覆盖全部模块，20 条回归用例（5 维验证）覆盖 8 大类场景，冒烟测试保障演示前快速验证，43 项自动验收脚本 100% 通过。
