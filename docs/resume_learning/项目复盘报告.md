# 项目复盘报告 · 架构风格智能助手

> 面向保研/求职面试的深度复盘。每个结论都基于真实代码，每个取舍都有理由。

---

## 1. 项目做了什么

一句话：**输入自然语言软件需求，自动推荐架构风格，输出可解释的决策报告。**

展开说，系统完成了"提取 → 匹配 → 评估 → 溯源"四个环节的闭环：

| 环节 | 做什么 | 用什么 |
|------|--------|--------|
| 提取 | 从自然语言中识别 10 个质量属性信号 | 90 个关键词词典 + Few-shot LLM 补全 |
| 匹配 | 基于信号对 10 种风格打分并融合图谱证据 | 7 条规则 + Neo4j HAS_QUALITY 遍历 |
| 评估 | 生成推荐 + 对比矩阵 + 风险分析 | 规则排序 + LLM 投票/摘要并行 |
| 溯源 | 每次推荐自动生成 ADR 决策记录 | JSON 主存储 + Neo4j 同步 |

额外做了三个增强：**组合推荐**（5 种预定义组合模式）、**重构建议**（5 种坏味检测 + 5 种重构模式）、**全链路降级**（8 个 LLM 调用点每个都有 fallback）。

最终交付：6 个 FastAPI 微服务、Docker Compose 一键启动、76 条单元测试、20 条回归用例、43 项自动验收 100% 通过。

---

## 2. 为什么这个项目符合软件体系结构课程目标

课程的核心命题是"理解架构风格及其质量属性权衡"。这个项目把课本上的概念变成了可运行的推理系统：

| 课程概念 | 项目中的体现 |
|---------|------------|
| 架构风格（Architectural Styles） | 知识库收录 10 种风格，每种含 tags/pros/cons/best_for/topology |
| 质量属性（Quality Attributes） | 10 个维度的特征信号是架构匹配的输入 |
| 质量属性与风格的映射 | 标签关联 + 7 条规则 + Neo4j HAS_QUALITY 关系 |
| 架构决策记录（ADR） | 每次推荐自动生成 ADR-YYYYMMDD-NNN |
| C4 架构模型 | Context 图和 Container 图完整记录自身架构 |
| 微服务架构 | 系统自身采用微服务设计，6 个独立容器 |
| 架构权衡（Trade-off） | 每条推荐附带风险分析和备选方案 |

更重要的是，项目**自身**的架构设计就是对课程内容的实践——用微服务架构构建了一个架构推荐系统。这种"自反性"是答辩中最有说服力的点。

## 3. 最初版本和最终版本的演进过程

### Phase 1：单文件原型（约 1 周）

最初只是一个 FastAPI 应用 + 一个 JSON 文件。`app.py` 里写了关键词提取、if-else 规则匹配、硬编码的风险描述。能跑通 demo，但代码像"意大利面条"——逻辑混在一起，改一个规则要翻好几屏。

**这个阶段学到的**：快速原型有助于验证想法是否可行。不要一开始就设计微服务——先在一个文件里跑通核心链路。

### Phase 2：微服务拆分（约 1 周）

把单文件拆成了 4 个服务：requirements / matching / evaluation / knowledge-base。引入 Docker Compose。拆分后最大的感受是**改一个模块不用通读全部代码**——改 matching 的评分逻辑完全不影响 requirements。

**这个阶段踩的坑**：拆分后首次联调，服务间 URL 硬编码导致 Docker 内和本地跑的地址不一致。后来统一改为环境变量配置。

### Phase 3：图谱与增强（约 1 周）

加入 Neo4j 图数据库、LangGraph 编排、Few-shot Prompt、LLM 缓存、组合推荐、重构建议、ADR。这是功能密度最高的一周。也是在这个阶段，我开始系统地考虑"每个外部依赖都要有 fallback"。

**这个阶段学到的**：功能越多越需要克制。组合推荐和重构建议都是"锦上添花"，但不应该影响核心推荐的质量。所以我设计了异步非阻塞调用——重构挂了不影响推荐。

### Phase 4：测试与文档（约 1 周）

写了 76 条单元测试、20 条回归用例、43 项自动验收脚本。补了架构设计文档、答辩讲稿、PPT。这个阶段最耗时的不是写测试，而是**设计回归测试的 5 个验证维度**——不能只验证"能跑通"，要验证"跑对了"。

**这个阶段学到的**：测试不是"事后补"，而是"设计时就考虑"。如果一开始就定义了"可解释性"的验证标准（必须包含规则理由和 LLM 摘要字段），写回归测试时直接对号入座。

### 演进总览

```
Phase 1          Phase 2          Phase 3           Phase 4
单文件原型    →   微服务拆分   →   图谱与增强    →   测试与文档
──────────────────────────────────────────────────────────
1个FastAPI       4个容器          7个容器            7个容器
1个JSON          4个服务          6个微服务          6个微服务
手动串行         手动编排         双引擎编排         双引擎编排
无测试            无测试           部分测试           76+20+43
```

## 5. 最困难的 5 个设计点

### 困难 1：规则引擎和 LLM 的职责边界

**问题**：评分到底给规则还是给 LLM？规则僵硬但可解释，LLM 灵活但不可靠。

**我的取舍**：规则引擎主导评分（标签 +2、规则 +1、学习 +1），LLM 投票只加 1 分（tie-break）。这意味着 LLM 在任何情况下都无法推翻规则引擎的排序——它只能打破平局。

**学到的**："可解释性优先于智能性"在这个场景下是正确的选择。架构推荐是一个需要审计的决策过程，每分必须有来源。

---

### 困难 2：全链路降级设计

**问题**：系统有 8 个 LLM 调用点。如果 LLM 挂了，系统还能不能正常工作？

**我的取舍**：逐一审查每个调用点，为每一个设计独立的 fallback。最关键的降级是 evaluation-agent 的摘要——LLM 不可用时，`_fallback_summary()` 用规则理由拼一份结构完整的中文报告。虽然不如 LLM 生成的自然，但信息完全等价。

**验证方式**：注释掉 `.env` 中的 LLM 配置，跑 20 条回归用例——全部通过。核心推荐结果和配 LLM 时一致。

**学到的**："永远不要假设外部服务一定可用"——每个 `await client.post()` 后面都应该有一个 `except` 和一份降级方案。

---

### 困难 3：Neo4j 和 JSON 的双后端调度

**问题**：Neo4j 是可选依赖。如何在"有图"和"无图"之间无缝切换，让上层 Agent 完全不感知？

**我的取舍**：GraphRepository 和 JsonRepository 实现完全相同的方法签名。`_repo()` 统一调度函数在 Neo4j 不可用时自动 fallback 到 JSON。上层 Agent 调的是 `_repo("get_styles")`，不关心底层是图还是文件。

**学到的**："接口一致性"是双后端设计的核心。如果两个 Repository 的方法名或返回值不同，上层代码就需要写 if-else——这就失去了"无缝切换"的意义。

---

### 困难 4：LangGraph 引入的价值论证

**问题**：当前主链路是线性串行的——extract → match → evaluate。手写 50 行 httpx 代码完全能搞定。那 LangGraph 的价值在哪？

**我的取舍**：诚实地说——在当前线性流程下，LangGraph 和手动编排没有本质区别。引入它的动机是**框架就绪**。如果未来需要条件路由（低置信度走人工审批）或并行节点（两个独立提取器同时跑），LangGraph 的 `add_conditional_edges` 和并行语义是手写代码难以模拟的。

**学到的**：不是每个技术引入都需要"当前就有巨大价值"。有时候"为未来预留清晰的扩展路径"也是正当理由——但要诚实地说清楚。

---

### 困难 5：回归测试的验证维度设计

**问题**：怎么验证"推荐是对的"？架构推荐没有标准答案——同一个需求在不同约束下可能选不同风格。

**我的取舍**：不验证"推荐了什么风格"，而是验证"推荐过程是否完整和可追溯"。5 个验证维度：候选数 ≥ 3（完整性）、主流风格覆盖（可靠性）、有最终推荐（明确性）、有规则理由和 LLM 字段（可解释性）、对比矩阵 ≥ 3 行（可比较性）。

**学到的**：有些系统不适合验证"输出对不对"，但可以验证"输出是否完整、有据可查"。这是一种务实的不完美。

## 7. 最容易被追问的 10 个问题

### Q1: 你的推荐准确率是多少？

**防守**：没有准确率数据。架构推荐没有标准答案——同一个需求在不同约束下可能选不同风格。我用 20 条回归用例 × 5 维验证来间接证明系统行为稳定，而不是用准确率数字。

### Q2: Neo4j 在 10 种风格规模下真的有必要吗？

**防守**：性能上没有——JSON 查找更快。价值在模型匹配和框架就绪——扩展到 100+ 风格时图优势才会体现。而且我做了 fallback，不用图也能跑。

### Q3: 你的 Agent 是"自主"的吗？

**防守**：不是。当前是固定线性编排——extract → match → evaluate。Agent 之间没有自主协商。LangGraph 框架支持条件路由，但当前场景不需要。

### Q4: 为什么不用机器学习做特征提取？

**防守**：需要标注数据（我没那么多）、可解释性差（关键词匹配是透明的）、对课程项目来说过重。LLM 补全作为模糊性的兜底已经够用。

### Q5: 你的规则引擎有没有偏见？

**防守**：有。"高并发 → Event-Driven +1"在某些场景下可能不适用。缓解方式：LLM 投票提供第二种视角、学习权重动态调整、对比矩阵让用户自己比较。

### Q6: 6 个微服务是不是过度设计？

**防守**：从 QPS 看是过细的。但从职责清晰和故障隔离看是合理的。课程项目的模块边界比性能更重要——而且 Docker Compose 一键启动，运维成本没增加。

### Q7: 你的测试覆盖是否足够？

**防守**：对课程项目够——每条核心函数都有测试。对生产不够——缺压力测试、安全测试、边界测试。我清楚缺口在哪。

### Q8: 为什么不直接用 ChatGPT 做推荐？

**防守**：四个原因——可解释性（评分可追溯）、可靠性（LLM 挂了系统还能用）、结构化输出（不是自由文本）、领域知识编码（规则和知识库比 LLM 记忆可靠）。

### Q9: 你从哪学的这些技术？用了多久？

**防守**：FastAPI 和 Docker 有基础，Neo4j 和 LangGraph 是现学的——官方文档 + 示例项目，各花了一两天。Prompt Engineering 靠反复实验——改了 4-5 版 Few-shot 示例才稳定。

### Q10: 如果继续做，你最想改进什么？

**防守**：知识库的自动化管理——目前新增风格要手动编辑 JSON 和 Neo4j 初始化脚本。我想做一个知识库管理界面，让非技术人员也能添加新风格。

## 8. 当前不足

### 不足 1：知识库规模小且手工维护

10 种风格、90 个关键词、7 条规则——全是手工维护。新增一种风格需要改 3 个文件（JSON + Neo4j 初始化 + 可能需要新增规则）。没有自动化的知识获取管道。

**如果有时间**：设计一个知识库管理界面，或者至少写一个校验脚本——检查 JSON 中新增的风格是否在 Neo4j 初始化脚本中覆盖了。

---

### 不足 2：特征提取依赖关键词而非语义理解

"系统要扛住流量尖峰"——"扛住"和"尖峰"不在词典里，可能漏判高并发。LLM 补全只在规则命中 ≤2 时触发，如果已经命中了 3 个维度，就不会触发补全——即使另外 7 个维度中有隐式的高并发需求。

**如果有时间**：在 LLM 补全中不只判断"是否涉及某维度"，而是"是否漏判了某维度"——不论当前命中数是多少。

---

### 不足 3：学习权重的冷启动

`learned_weights.json` 只有 3 条数据。学习权重机制已经完全实现，但缺乏反馈数据来驱动。≥2 确认的阈值意味着一个新特征-风格关联至少需要两次用户反馈才能生效——在当前数据量下几乎不会触发。

**如果有时间**：降低初始阈值（首次确认即生效），或引入合成反馈（用回归测试的预期结果模拟用户反馈）。

---

### 不足 4：重构检测的粒度

基于需求文本中的关键词（"单体""耦合严重"）做坏味检测，而非对实际代码的静态分析。用户描述历史系统时可能触发误判。

**如果有时间**：引入意图识别——区分"描述过去"（可能需要重构）和"描述现状"（不一定需要重构）。或者干脆把重构建议改为"用户主动触发"而非自动检测。

---

### 不足 5：缺失性能和安全测试

没有压测（最大并发数未知）、没有安全测试（Prompt Injection 防护有限）、没有 CI/CD（测试只能在本地手动跑）。

**如果有时间**：引入 GitHub Actions 做自动化测试。至少加一个 API Key 认证和速率限制。

## 10. 一页纸速记版

> 面试前 5 分钟快速浏览此节。

### 项目一句话
