## 1. 项目目标：解决什么问题

### 1.1 一句话定位

> "自然语言需求输入 → 架构风格推荐 → 可解释决策报告"的完整闭环系统。

用户输入一段软件需求描述（如"开发跨平台即时通讯系统，支持万人同时在线..."），系统自动完成：
- 从需求文本中提取 10 个质量属性维度的特征信号
- 基于规则引擎 + 知识图谱双重评分，推荐 3 种架构风格
- 生成包含推荐理由、对比矩阵、风险建议、ADR 决策记录的完整报告
- 可选：检测架构坏味并给出重构模式建议

### 1.2 解决的痛点

| 痛点 | 系统如何解决 |
|------|------------|
| 架构选型依赖经验，新手不知从何下手 | 10 维特征信号自动提取 + 7 条专业规则自动评分 |
| 架构推荐缺少可解释性 | 每条推荐附命中标签、图谱证据、LLM 摘要、风险矩阵 |
| LLM 服务不稳定导致链路中断 | 全链路 LLM 可选：规则引擎主导，LLM 不可用时自动降级 |
| 架构决策事后难以追溯 | 每次推荐自动生成 ADR 记录，Neo4j + JSON 双存储 |

### 1.3 使用场景（课程预设）

README 预设的 4 个演示输入：
1. **基础推荐**：IM 系统 → Event-Driven Architecture
2. **组合推荐**：电商平台 → Microservices + Event-Driven 组合
3. **重构建议**：已有单体系统 → Strangler Fig Pattern
4. **图谱证据**：日志分析平台 → Pipeline-Filter + Graph Evidence

### 4.2 matching-agent（架构风格匹配）

**文件**：[services/matching_agent/app/main.py](services/matching_agent/app/main.py)（~170 行）
**子模块**：[graph_matcher.py](services/matching_agent/app/graph_matcher.py)（~94 行）、[combo_matcher.py](services/matching_agent/app/combo_matcher.py)（~114 行）
**端口**：8002
**API**：`POST /match`, `GET /health`

**评分体系**（三层融合）：

```
候选风格最终分数 = 规则引擎分 + 图谱证据分(capped) + 学习权重加分
```

| 层次 | 来源 | 说明 |
|------|------|------|
| **L1：规则引擎** | `score_style()` | tags 匹配 +2/项；7 条硬编码专业规则 +1/条 |
| **L2：学习权重** | `learned_weights.json` | 基于历史反馈，确认 ≥2 次的特征-风格关联 +1 |
| **L3：图谱证据** | `graph_matcher.py` → `blend_scores()` | 通过 Neo4j HAS_QUALITY 关系匹配；加分上限 ≤ 规则分的 50% |

**7 条硬编码规则**：

| 规则 | 条件 | 加分 |
|------|------|------|
| 高并发 → Event-Driven | `high_concurrency` 为 True | +1 |
| 多团队 → Microservices | `team_size_large` 为 True | +1 |
| 强一致 → Layered | `strict_consistency` 为 True | +1 |
| 实时 + 数据密集 → Pipeline-Filter | `data_intensive` + `real_time` | +1 |
| 高并发 + 数据密集 → CQRS | `data_intensive` + `high_concurrency` | +1 |
| 高并发 + 强一致 → Microservices | `high_concurrency` + `strict_consistency` | +1 |

**主流优先机制**：
- `MAINSTREAM_STYLES = ["Layered Architecture", "Microservices", "Event-Driven Architecture"]`
- 当无命中信号时，返回主流风格作为基线对比集
- Top 3 优先从主流风格中选出有分数的

**架构组合推荐**：
- 来源数据：[architecture_combinations.json](services/knowledge_base/data/architecture_combinations.json)（5 种组合）
- 评分公式：`组合分 = 组成风格分数和 + 特征覆盖互补加分 + 图谱 CAN_COMBINE_WITH 加分 - 复杂度惩罚`
- 返回 Top 3 组合

**实现程度**：**已实现**。规则引擎始终工作；图谱证据和图谱加分为**可选增强**（Neo4j 不可用时 `fetch_graph_evidence()` 返回 None）；学习权重基于用户反馈积累（当前 learned_weights.json 仅有 3 条数据，为**轻量实现**）。

### 4.3 evaluation-agent（评估与报告）

**文件**：[services/evaluation_agent/app/main.py](services/evaluation_agent/app/main.py)（~393 行）
**端口**：8003
**API**：`POST /evaluate`, `GET /health`

**混合推理机制**：

| 步骤 | 机制 | LLM 依赖 |
|------|------|---------|
| 1. 规则排序 | 按 score 降序 | 不依赖 |
| 2. LLM 投票 | `llm_vote_style()` | 可选，temperature=0.0 |
| 3. LLM 摘要 | `llm_summary()` | 可选，temperature=0.3 |
| 4. 风险分析 | `_dynamic_risks()` | 不依赖（3 种风格有硬编码风险模板） |
| 5. 对比矩阵 | 规则化生成 | 不依赖 |

**LLM 投票机制**：
- 与 LLM 摘要**并行执行**（`asyncio.gather`）
- 投票结果作为轻量 tie-break：若 LLM 选择的风格在候选列表中，该风格 +1 分并标记 `llm_vote_bonus`
- 失败不影响主链路

**LLM 降级摘要**：`_fallback_summary()` 使用规则引擎命中的理由生成结构化中文摘要，保证即使完全没有 LLM 也有可读报告。

**报告输出结构**：

```json
{
  "recommended_style": "...",
  "recommended_style_zh": "...",
  "alternative_styles": ["...", "..."],
  "decision_basis": {
    "rule_engine": ["中文理由"],
    "rule_engine_raw": ["原始理由"],
    "llm_summary": "...",
    "llm_vote": "风格名 或 null"
  },
  "comparison_matrix": [{ "style", "score", "recommendation_type", "pros", "cons", "key_reasons", "topology_mermaid" }],
  "risk_and_suggestions": { "main_risks": [], "suggestions": [] },
  "recommended_combination": { "name", "components", "synergy_zh", ... },
  "adr": { "adr_id", "adr_status" }
}
```

**实现程度**：**已实现**。LLM 投票和摘要均为可选；规则降级保证核心链路高可用。

### 4.7 refactoring-agent（架构重构建议）

**检测机制**：

| 层次 | 机制 | 说明 |
|------|------|------|
| **架构坏味检测** | `detect_smells()` | 5 种坏味 × 关键词匹配 + 特征推断 |
| **重构模式选择** | `select_patterns()` | 5 种重构模式 × 关键词 + 风格匹配 |

**5 种架构坏味**（`ARCHITECTURE_SMELLS`）：
monolith_coupling（单体耦合过重）、scaling_bottleneck（扩展瓶颈）、slow_release（发布周期过长）、legacy_lockin（遗留系统锁定）、data_coupling（数据层耦合）

**5 种重构模式**（`REFACTORING_PATTERNS`）：
Strangler Fig（绞杀者）、Anti-Corruption Layer（防腐层）、Modular Monolith First（模块化单体优先）、CQRS Read/Write Split（读写分离）、Event-Driven Migration（事件驱动迁移）

每种模式含：适用条件（when）+ 4 步操作步骤（steps）

**触发条件**：
- 需求文本含"已有系统/遗留/单体/耦合/重构/迁移/拆分"等关键词
- 或 high_concurrency + scalability 同时命中

**实现程度**：**已实现**。LLM 润色为可选增强（不可用时使用规则模板）。

## 5. 技术亮点

### 5.1 Compound AI System（复合 AI 系统）

**证据文件**：[services/api_gateway/app/main.py](services/api_gateway/app/main.py)、[matching_agent/app/main.py](services/matching_agent/app/main.py)、[evaluation_agent/app/main.py](services/evaluation_agent/app/main.py)

**说明**：系统不是"一个 LLM Call"，而是规则引擎、知识图谱推理、LLM 语义分析的复合架构。6 个微服务各自承担不同职责，通过编排层协同工作。

| 组件 | 技术 | 作用 |
|------|------|------|
| requirements-agent | 关键词词典 + LLM semantic supplement | 特征提取 |
| matching-agent | 规则引擎 + Neo4j graph reasoning | 风格匹配 |
| evaluation-agent | 规则排序 + LLM vote + LLM summary | 决策评估 |

## 8. 可能被面试追问的薄弱点

> 以下标注"诚实区"，面试准备时建议针对性补强。

### 8.1 轻量实现 / 原型级实现

| 薄弱点 | 当前状态 | 被追问风险 | 建议 |
|--------|---------|----------|------|
| **前端** | 单页面 HTML + CSS，无框架 | 中 | 回答"聚焦后端 AI 系统能力，前端仅作展示层" |
| **学习权重** | 仅有 3 条数据（`learned_weights.json`） | 中 | 回答"框架已实现，目前处于冷启动阶段，需要持续积累反馈" |
| **架构组合** | 5 种预定义，非动态生成 | 低 | 回答"当前基于专家经验预定义，未来可扩展为图谱自动发现" |
| **Neo4j 图模型** | 6 节点 + 6 关系，规模较小 | 低 | 回答"与 10 种风格匹配的轻量图模型，符合 80/20 原则" |
| **LangGraph** | 仅实现线性 DAG（串行），未使用条件边 | 中 | 回答"当前主链路为确定性流程，条件路由框架已就绪" |

### 8.2 功能边界（诚实标注）

| 限制 | 说明 | 应对话术 |
|------|------|---------|
| **知识库依赖人工维护** | 新增风格需手动编辑 JSON | "当前聚焦推荐质量，自动化知识获取是后续方向" |
| **LLM 为外部依赖** | 使用 DeepSeek/OpenAI API，非本地模型 | "采用 OpenAI 兼容协议，可替换为任何兼容服务" |
| **无认证机制** | 所有 API 端点开放 | "面向课程答辩和内部工具的定位，生产环境可加 API Key" |
| **单实例部署** | 未配置负载均衡 | "微服务架构已实现服务级拆分，水平扩展可由容器编排层负责" |
| **图谱规模有限** | Neo4j 初始仅含 10 种风格的预设数据 | "图模型已验证，数据量可随知识积累线性扩展" |
| **重构建议依赖关键词** | bad smell 检测基于关键词匹配而非 AST/架构度量 | "当前实现面向自然语言需求，代码级分析需结合静态分析工具" |

### 8.3 架构设计上的权衡（可作为展示"架构思维"的话题）

| 权衡 | 选择 | 理由 |
|------|------|------|
| **规则 vs ML** | 规则引擎主导 | 可解释性优先；LLM 不可用时核心链路不中断 |
| **同步 vs 异步通信** | 同步 HTTP | 课程场景请求量低，同步简化调试和错误追踪 |
| **JSON vs Neo4j** | 双后端自适应 | Neo4j 用于展示图谱推理能力，JSON 保证零外部依赖 |
| **单仓库 vs 多仓库** | 单仓库（Monorepo） | 课程项目规模小，单仓库降低 CI/CD 复杂度 |
| **FastAPI vs Flask** | FastAPI | 需要类型校验（Pydantic）+ 自动 OpenAPI 文档 |
