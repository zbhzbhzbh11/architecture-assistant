## 3. 1 分钟项目介绍（常规面试）

> 我这个项目叫架构风格智能助手，本质上是一个 Compound AI System——它不只是一个调 LLM 的聊天机器人。
>
> **背景**是软件体系结构课程的大作业，要求构建一个 LLM + 知识图谱双驱动的架构推荐系统。我完成了全部 9 项技术建议，43 项自动验收 100% 通过。
>
> **架构上**，系统采用微服务设计，6 个独立 FastAPI 容器通过 Docker Compose 编排。API Gateway 使用 LangGraph StateGraph 做 Agent 工作流编排，LangGraph 不可用时自动回退到手写编排。
>
> **推理上**，核心是三层混合：requirements-agent 用 10 维关键词词典做特征提取，matching-agent 用规则引擎（7 条架构规则）和 Neo4j 知识图谱做风格匹配，evaluation-agent 用规则排序加 LLM 投票/摘要做最终决策。
>
> **工程上**，我特别关注了容错设计——全链路每个 LLM 调用点都有降级路径，LLM 不可用时自动切换规则模式。还实现了 Few-shot Prompt、LLM 缓存（内存/SQLite 双后端）、ADR 自动生成、组合推荐和重构建议。

**表达要点**：
- 三段式：架构 → 推理 → 工程
- "不只是调 LLM 的聊天机器人"是第一句话——建立差异化认知
- 每个技术名词后紧跟一句解释（如"LangGraph StateGraph 做 Agent 工作流编排"）

## 4. 3 分钟项目介绍（深度面试/终面）

> 我来详细介绍一下我的架构风格智能助手项目。
>
> **【背景与动机】** 软件体系结构课程要求构建一个 LLM + 知识图谱双驱动的架构推荐系统。但我从一开始就没想只做一个"调 LLM API 然后输出一段文字"的聊天机器人。我的设计理念是 Compound AI System——让合适的组件做合适的事：规则引擎保证可解释性，知识图谱提供结构化推理，LLM 处理模糊语义。
>
> **【系统架构】** 系统由 6 个 FastAPI 微服务组成，通过 Docker Compose 编排。API Gateway 是编排层——使用 LangGraph StateGraph 建模了 extract → match → evaluate → trace 四个节点的有向无环图。LangGraph 是不可选依赖——不可用时系统自动回退到手写 httpx 编排，两者输出完全一致。
>
> requirements-agent 负责特征提取：核心是 10 维质量属性关键词词典（约 90 个中英文词），配合 6 种否定模式过滤。当规则命中维度 ≤2 时，触发 Few-shot LLM 语义补全。matching-agent 是推理核心：score_style() 函数实现三层评分——标签匹配 +2、7 条架构规则 +1、学习权重 +1（≥2 次反馈确认）。同时通过 Neo4j 的 HAS_QUALITY 关系做图谱推理增强，加分上限为规则分的 50%。evaluation-agent 负责报告生成：规则排序主导，LLM 投票（temperature=0.0）做 tie-break，LLM 摘要（temperature=0.3）生成自然语言报告。投票和摘要是用 asyncio.gather 并行调用的。
>
> **【工程增强】** 我实现了几项工程增强。Few-shot Prompt：需求侧 6 个示例覆盖模糊语义和否定场景，评估侧 3 个示例覆盖三种核心风格，模块不可用时自动降级为零样本。LLM 缓存：内存/SQLite 双后端，缓存键包含 knowledge_version（styles 文件 MD5），知识库更新后自动失效。ADR 自动生成：每次推荐完成后自动写入决策记录，JSON 主存储 + Neo4j 同步，写入失败不阻塞推荐。组合推荐：5 种预定义组合模式，四因素评分公式。重构建议：5 种架构坏味检测 + 5 种重构模式推荐，LLM 可选润色。
>
> **【测试与验收】** 76 条单元测试覆盖全部模块，20 条回归用例覆盖 8 大类场景（每条 5 维验证），43 项自动验收 100% 通过。
>
> **【核心收获】** 通过这个项目，我不仅掌握了微服务架构、LangGraph 编排、Neo4j 图数据库、Prompt Engineering 等具体技术，更重要的是理解了 Compound AI System 的设计哲学——不是让 LLM 做所有事，而是让规则、图谱、LLM 各司其职，并且每一层 LLM 调用都准备好"如果 LLM 挂了"的 B 计划。

**表达要点**：
- 背景→架构→推理→工程→测试→收获，六段式
- 每段之间有明显转折，给面试官消化和打断的空间
- 控制时间在 2:30-3:00，练习到可以自然讲完

## 8. 面试官追问与防守话术

> 当面试官盯住某个技术点追问时，以下的"诚实回答"能让你既不失分、又不被问倒。

### 8.1 追问：你的系统能用于真实项目吗？

**防守策略**：承认定位，强调能力可迁移。

> "目前这个系统的定位是课程原型。10 种架构风格的知识库覆盖了最主流的模式，但从课程要求来说已经足够了。如果要在真实项目中用，需要做三件事：扩充知识库到更多风格和场景、引入真实项目的历史架构选型数据做验证、以及对接实际的架构评审流程。但核心的 Compound AI 架构——规则引擎 + 图谱 + LLM 的组合——是可以直接迁移的。"

### 8.2 追问：你的推荐准确率是多少？

**防守策略**：说明评价体系而非给出虚拟数字。

> "我没有标注数据集来算准确率，因为架构推荐本身就没有'标准答案'——同一个需求在不同约束下可能选择不同的架构。我用的验证方式是 20 条回归用例 × 5 维验证：检查候选数是否 ≥3、主流风格是否覆盖、是否有最终推荐、推荐是否有据可查、对比矩阵是否完整。这 5 个维度目前全部通过。我更关注的是推荐的可解释性——每条推荐都有明确的评分来源，而不是一个黑盒准确率。"

### 8.3 追问：Neo4j 在你的项目中是不是可有可无？

**防守策略**：承认当前数据量下图优势不明显，但强调框架价值。

> "在当前的 10 种风格规模下，JSON 文件和 Neo4j 图数据库的查询效率差异可以忽略。Neo4j 在本项目中的价值更多是框架性的——我验证了图模型设计（6 节点 + 6 关系）、Cypher 查询遍历、以及与 JSON 的自动 fallback 机制。如果知识库扩展到 100+ 风格、1000+ 关联关系，图的优势就会体现出来——多跳关系查询在关系型数据库中需要多表 JOIN，但在图中是原生操作。所以准确地说：Neo4j 在当前规模下是'锦上添花'，但从架构设计上已经为更大规模做好了准备。"

### 8.4 追问：你的 Agent 真的是"自主"的吗？

**防守策略**：诚实说明是固定编排，但框架支持扩展。

> "当前实现是一个确定性的线性工作流——extract → match → evaluate → trace。Agent 之间没有自主协商或动态路由。LangGraph 的 StateGraph 支持条件边和并行节点，框架本身已经就绪——当前没有使用是因为主链路的步骤是确定的。如果后续需要引入'检测到重构需求时路由到重构 Agent'或'根据置信度决定是否需要人工审批'，只需要把 add_edge 改为 add_conditional_edges，节点代码不需要改动。"

### 8.5 追问：你的规则引擎能不能应对新出现的架构风格？

**防守策略**：承认需要人工维护，但强调可扩展性。

> "当前规则引擎的 7 条规则和 10 维关键词词典都需要人工维护。如果出现了一种新的架构风格（比如最近流行的 AI-Native 架构），需要做三件事：在 architecture_styles.json 中新增风格定义（含 tags/pros/cons）、在关键词词典中评估是否需要新的质量属性维度、在 7 条规则中评估是否需要新增规则。这不是自动学习的——这是有意为之，因为架构规则的可靠性比覆盖率更重要。一条错误的自动生成规则可能比没有规则更糟糕。"

### 8.6 追问：你的系统和直接问 ChatGPT 有什么区别？你用 LLM 了为什么还要自己做？

**防守策略**：这是最核心的问题，熟练背诵。

> "四个本质区别。第一，可解释性——我的每条推荐都有评分明细，标签命中加几分、规则加几分、图谱加几分，ChatGPT 做不到。第二，可靠性——我的系统 LLM 不可用时能自动降级，ChatGPT 挂了就是挂了。第三，结构化输出——我输出的是标准决策报告（推荐 + 对比矩阵 + 风险 + ADR），不是自由文本。第四，领域知识——我把 10 种风格的质量属性、适用场景、风险都编码在知识库和规则中，这比 LLM 训练数据中的模糊记忆更可靠。"

### 8.7 追问：你觉得这个项目最大的技术挑战是什么？

**防守策略**：选一个真实的、能展开讲的挑战。

> "最大的挑战是设计降级策略。一开始我想的是'LLM 挂了就报错'——简单直接。但后来意识到这不是好的工程实践。我重新审视了全链路每一个 LLM 调用点——requirements-agent 的语义补全、evaluation-agent 的投票和摘要、refactoring-agent 的润色——然后为每一个都设计了 fallback 路径。最终的效果是：用户不配 LLM，系统照样能跑，输出的报告结构完整、信息准确。这个设计让我深刻理解了'外部依赖的防御性编程'——永远不要假设外部服务一定可用。"
