## 第二部分：15 分钟架构设计专题讲解讲稿

### 章节 1：业务背景与问题定义（1 分钟）

在软件工程实践中，架构选型面临三个痛点：

1. **需求模糊性**：用户描述通常是"要支持高并发""要可靠"，但缺少精确量化指标
2. **依赖个人经验**：同样的需求，不同架构师可能给出不同建议
3. **结论难追溯**："为什么选微服务不选单体？"——往往回答不上来

我们把这个场景建模为一个 AI 辅助决策系统：

```
输入：自然语言需求描述
处理：特征提取 → 规则匹配 → LLM 评估
输出：推荐架构 + 候选对比 + 风险建议 + 决策依据
```

---

### 章节 2：系统总体架构（1 分钟）

*[建议配图：C4-Context 图]*

系统从外到内三层：

- **外部：** 用户（浏览器 / API 调用者）+ LLM 服务（DeepSeek）
- **核心：** 架构风格智能助手（6 个容器）
- **内部：** 前端容器 + API 网关 + 3 个 Agent + 知识库

技术选型：
- FastAPI + Uvicorn：异步、自动文档、轻量
- httpx：异步服务间通信，20s 超时
- Mermaid.js：零依赖拓扑图渲染
- DeepSeek v4-flash：OpenAI 兼容协议，成本可控

---

### 章节 3：微服务划分（1.5 分钟）

*[建议配图：C4-Container 图]*

5 个独立微服务 + 1 个前端容器：

| 服务 | 端口 | 职责 |
|---|---|---|
| api-gateway | 8000 | 统一入口，编排调用，聚合响应 |
| requirements-agent | 8001 | 需求特征提取（10 维度词典 + LLM 补全） |
| matching-agent | 8002 | 规则评分 + Top3 候选生成 |
| evaluation-agent | 8003 | LLM 混合推理 + 最终报告 |
| knowledge-base | 8004 | 架构风格数据 + 案例反馈 |

**划分合理性论证：**
- 需求提取是规则密集型，评估是 LLM 依赖型——异构成分应隔离
- LLM 有外部依赖风险——独立部署后，LLM 波动不影响特征提取和知识库查询
- 后续演化——matching-agent 可升级为 Neo4j 驱动，其他服务无需修改

---

### 章节 4：Agent 协作机制（1.5 分钟）

*[建议配图：UML 时序图]*

我们采用了 **Pipeline-Agent 协作模式**：

```
用户 → Gateway
  Gateway → Requirements Agent POST /extract  → {features}
  Gateway → Matching Agent    POST /match     → {candidates}
  Gateway → Evaluation Agent  POST /evaluate  → {final_report}
  Gateway → 聚合返回
```

**与传统方案对比：**

| 方案 | 优点 | 缺点 |
|---|---|---|
| 单一巨型 Agent | 简单 | 职责不清，故障影响大 |
| 全异步事件驱动 | 松耦合 | 课程周期内实施成本高 |
| **Pipeline-Agent（我们的选择）** | 职责清晰、可独立测试、可替换 | 存在串行延迟 |

**设计原则：**
- 每个 Agent 只解决一类问题
- 上一个输出 = 下一个输入
- LLM 仅在 evaluation-agent 中参与，降低不确定性扩散

---

### 章节 5：LLM 集成方案（1.5 分钟）

*[建议配图：LLM 调用流程图]*

**配置方式：**
- 采用 OpenAI 兼容协议，支持 DeepSeek/通义千问/OpenAI 一键切换
- `.env` 文件配置，Key 不硬编码，自动加载

**两套 Prompt 分工：**

| 功能 | Temperature | 角色 | 要求 |
|---|---|---|---|
| llm_vote_style | 0.0（确定性） | strict architecture judge | 仅返回风格名，不输出多余文字 |
| llm_summary | 0.3（适度表达） | senior software architecture reviewer | 输出结构化中文分析（核心推荐+备选+理由+优缺点） |

**鲁棒性设计——四层防护：**
1. 20s 超时控制
2. try-except 全量捕获
3. 失败返回格式化的降级摘要（规则引擎生成，非空白）
4. LLM 未配置时自动跳过（不影响核心推荐链路）

### 章节 7：混合推理机制（1.5 分钟）

*[建议配图：推理流程图]*

**这是本项目的核心创新点。**

```
规则引擎（确定性）              LLM 评估（语义增强）
─────────────────              ─────────────────
tags 匹配计分 (+2/tag)    →    投票 +1 分 (tie-break)
额外专家规则 (+1)         →    中文分析报告生成
主流风格白名单兜底        →    降级时规则独立运行
─────────────────              ─────────────────
          ↓                           ↓
        规则保下限                LLM 提上限
```

**为什么两者都需要？**

纯规则的问题：泛化弱。"高并发+实时性"匹配事件驱动——但如果用户描述的是"大量用户同时查询"，规则可能无法区分是读压力还是写压力。

纯 LLM 的问题：幻觉、不稳定、不可审计。相同需求两次调用可能得到不同推荐。

协同方案：规则生成确定性候选集和评分，LLM 在规则基础上做语义增强——投票 +1 分是轻量级影响，不会颠覆规则排序。

### 章节 9：异常处理与可靠性（1 分钟）

*[建议配图：异常处理流程图]*

**网关层：**
- 统一 try-except 捕获上游 HTTP 异常 → 标准 502 返回
- `httpx.AsyncClient(trust_env=False)` 避免系统代理干扰

**LLM 层：**
- 20s 超时 → 失败 → 降级为规则摘要
- 未配置 Key → 自动跳过，不影响核心链路

**前端层：**
- 按钮防重复提交（disabled 状态切换）
- 网络失败 → 显示错误信息，不白屏

**容错性成果：**
- 单个 Agent 故障不会导致网关崩溃
- LLM 完全不可用时，系统仍可输出完整的推荐报告（规则引擎独立运行）

### 章节 11：创新点与总结（1 分钟）

**项目创新点：**

1. **规则+LLM 混合推理**：规则保下限、LLM 提上限、三层证据可追溯

## 第三部分：5 分钟问答准备

### Q1：为什么选择微服务架构？

我们按领域职责将系统拆分为 5 个独立微服务。核心考虑是异构成分隔离——需求提取是规则密集型，评估是 LLM 依赖型。将 evaluation-agent 独立部署后，LLM 不可用时不影响特征提取和知识库查询。每个服务有独立的 Dockerfile 和容器，可独立替换和演进。

### Q2：Agent 如何协作？

采用 Pipeline-Agent 模式。API Gateway 作为编排器，依次调用 Requirements Agent → Matching Agent → Evaluation Agent。上一个 Agent 的输出是下一个的输入。每个 Agent 只解决一类问题，职责清晰且可独立测试。

### Q3：为什么需要规则引擎？直接用 LLM 推荐不是更简单？

纯 LLM 有三个问题：不稳定（相同输入可能得到不同推荐）、不可审计（无法解释评分依据）、有幻觉风险（可能推荐不存在的架构）。规则引擎提供确定性的评分基线，LLM 在规则基线上做轻量级语义增强。规则保下限，LLM 提上限。

### Q4：如何处理需求描述的模糊性？

双重策略。第一层：10 维度关键词词典做确定性特征提取，约 100 个中英文关键词。第二层：当规则命中维度 ≤2 时，自动触发 LLM 语义补全（temperature=0.1，返回严格 JSON）。同时包含否定语义过滤——"不需要高并发"不会被错误匹配。

### Q5：如果 LLM 调用失败怎么办？

四层防护：① 20s 超时 ② try-except 全量捕获 ③ 降级为规则引擎生成的格式化摘要（非空白）④ 未配置 Key 时自动跳过。降级后规则引擎仍可独立完成完整推荐。回归测试中从未出现因 LLM 不可用导致推荐链路中断。

### Q6：知识库如何扩展？

两种方式：① POST /styles 动态新增架构风格，即时生效，新风格会参与后续推荐流程；② POST /feedback 记录用户对推荐结果的确认/修正，GET /feedback/stats 统计各风格准确率，积累的反馈数据可用于后续自动权重学习。

### Q7：测试数据集如何设计的？

20 条用例覆盖 10+ 领域：IM 通讯、电商、企业审批、日志分析、银行账务、在线教育、物联网、短视频、政务系统、SaaS、医疗影像、数据同步、外卖、证券交易、客服、科研处理、校园管理、跨境支付。每条用例包含触发不同特征维度的关键词。

### Q8：和直接问 ChatGPT 有什么区别？

本质区别有四个：① 规则引擎保底——候选集来自确定性匹配，而非黑盒生成；② 知识库约束——推荐范围限定在 10 种学术验证的架构风格内；③ 可追溯——三层证据链（特征命中 → 规则评分 → LLM 解释），每步可审计；④ 可扩展——知识库支持动态注入新风格，反馈数据可持续优化。
