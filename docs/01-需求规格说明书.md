# 需求规格说明书

## 1. 引言

### 1.1 文档目的与范围

本文档定义"基于大模型的软件架构风格智能助手"（Architecture Assistant / ArchStyle-Agent）的功能需求、非功能需求、AI 系统特有需求与验收标准。文档作为系统设计、开发、测试与答辩评审的基准依据。

系统目标：接收自然语言描述的软件需求，自动提取非功能特征与约束特征，基于知识库与推理引擎输出至少三种候选架构风格，并生成包含对比矩阵、风险建议与推理依据的最终推荐报告。

### 1.2 文档约定

- FR-XX：功能需求编号（Functional Requirement）
- AIR-XX：AI 系统特有需求编号
- NFR-XX：非功能需求编号
- 验收标准以可观测、可测试方式描述

### 1.3 术语定义

| 术语 | 说明 |
|------|------|
| Agent | 独立运行的微服务，承担特定职责（需求解析、架构匹配、评估决策） |
| 规则引擎 | 基于关键词词典与计分规则的确定性推理组件 |
| LLM | 大语言模型，用于语义补全与推荐解释生成 |
| 混合推理 | 规则引擎为主、LLM 为辅的协同决策模式 |
| 降级（Fallback） | LLM 不可用时自动切换至纯规则模式的容错机制 |
| 知识库 | 存储架构风格元数据（名称、标签、优缺点、拓扑图）的结构化数据源 |

## 2. 项目背景

### 2.1 问题定义

在软件体系结构课程项目场景中，需求描述通常存在模糊性与不确定性。传统人工评审面临以下问题：

1. **需求特征识别效率低**：依赖个人经验判断，缺乏系统化的特征提取方法。
2. **架构候选方案不稳定**：不同评审者可能得出不同结论，缺乏可复现的决策过程。
3. **决策理由不完整**：缺少量化的评分依据和风险分析，结论可追溯性弱。

### 2.2 解决方案概述

本系统通过"规则引擎 + LLM 协同"的双驱动方案，将自然语言需求转化为可解释的架构风格推荐。规则引擎保证推荐下限（确定性、可追溯），LLM 提升推荐上限（语义理解、解释生成），在 LLM 不可用时自动降级为纯规则模式，确保核心链路高可用。

### 2.3 项目范围

- **包含**：需求特征提取、架构风格匹配与评分、候选推荐、对比矩阵生成、风险评估、Web 前端展示、Docker 容器化部署、自动化测试。
- **不包含**：图数据库（Neo4j）知识推理、自动权重学习闭环、ADR 决策追溯日志、生产级高可用集群部署。

### FR-02 需求理解与特征提取

**描述**：系统应从非结构化需求文本中识别关键非功能特征与约束特征，并返回特征命中的关键词证据，支撑后续可解释性追溯。

**输入**：自然语言需求文本

**处理过程**：
1. 需求文本转为小写进行关键词匹配。
2. 遍历 10 个特征维度的关键词词典，命中则标记为 True。
3. 对命中关键词执行否定语义过滤（如"不需要高并发"中"高并发"被否定词"不需要"修饰，不计数）。
4. 如规则命中维度 ≤ 2（特征过于稀疏），且 LLM 已配置，则调用 LLM 做语义补全。LLM 不可用时静默降级，维持规则结果。
5. 返回每个维度的布尔值及命中关键词列表。

**特征维度定义**（10 个维度）：

| 维度 | 英文标识 | 典型关键词（部分） |
|------|----------|-------------------|
| 高并发 | high_concurrency | 高并发、并发、万人、海量用户、秒杀、高吞吐、QPS |
| 实时性 | real_time | 实时、即时、在线、低延迟、毫秒、消息、通知、IM |
| 可靠性 | reliability | 可靠、高可用、容灾、容错、稳定、不丢、一致性 |
| 可扩展性 | scalability | 扩展、弹性、扩容、横向、scale、可伸缩 |
| 复杂业务 | complex_business | 复杂业务、交易、审批、规则、工作流、workflow |
| 强一致性 | strict_consistency | 强一致、事务、金融、账务、一致提交 |
| 部署约束 | deployment_constraint | 本地部署、私有化、边缘、多地域、离线、内网 |
| 数据密集型 | data_intensive | 数据流、ETL、流处理、日志、监控、数据中台、批处理 |
| 多团队协作 | team_size_large | 多团队、跨团队、多人协作、并行开发 |
| 安全性 | security | 安全、加密、认证、鉴权、授权、审计、隔离、防护、合规、脱敏、防篡改、权限、安全隔离、零信任 |

**输出**：
```json
{
  "features": {"high_concurrency": true, "real_time": true, ...},
  "feature_hits": {"high_concurrency": ["高并发", "万人"], "real_time": ["实时"], ...}
}
```

**异常处理**：
- LLM 调用超时（15s）或网络异常时，自动回退为规则结果，不中断流程。
- LLM 返回非 JSON 输出时，尝试解析 markdown 代码块包裹；解析失败则丢弃 LLM 结果。

**对应代码**：

- LLM 语义补全：`llm_semantic_supplement()` (第 80-128 行)

**验收标准**：
- 输出包含 features 布尔映射和 feature_hits 关键词证据。
- 否定语义场景下，被否定词修饰的关键词不被计入。

### FR-03 候选架构推荐

**描述**：系统应基于提取的特征，从知识库中匹配并评分，输出至少 3 种候选架构风格。

**输入**：特征布尔映射 `{"feature_name": true/false, ...}`

**处理过程**：
1. 从知识库获取全部架构风格数据（GET /styles）。
2. 对每种风格计算规则分数：
   - 风格 tags 与特征匹配：每命中 1 项得 2 分。
   - 额外规则加分：事件驱动 + 高并发 +1 分；微服务 + 多团队 +1 分；分层架构 + 强一致性 +1 分。
3. 按分数降序排列。
4. 候选集生成采用"主流优先 + 按分补齐"策略：优先纳入三种主流架构（Layered、Microservices、Event-Driven）；如有剩余空位则按分数从高到低补齐。
5. 如所有风格得分均为 0（无有效特征信号），返回三种主流架构作为基线对比。

**输出**：候选架构列表（Top 3），每项含 style、score、reasons、pros、cons、best_for、topology_mermaid。

**异常处理**：
- 知识库服务不可用时，返回 HTTP 502 及明确错误信息。

**对应代码**：
- [services/matching_agent/app/main.py](services/matching_agent/app/main.py) — 完整实现
- 评分函数：`score_style()` (第 40-68 行)
- 候选生成：`match()` (第 72-107 行)

**验收标准**：
- 每次响应候选数量 = 3。
- 当存在有效特征信号时，主流架构覆盖至少 1 种。
- 当无特征信号时，返回三种主流架构作为基线。

**实现状态**：✅ 已完成

### FR-04 最终决策与报告输出

**描述**：系统应基于候选架构列表，通过规则引擎排名与 LLM 辅助判断，生成最终推荐报告。

**输入**：原始需求文本 + 提取特征 + 候选架构列表

**处理过程**：
1. 将候选按规则分数降序排列。
2. 并行执行两项 LLM 任务（LLM 未配置时跳过）：
   - **LLM 投票**：从候选集中选出最佳风格（temperature=0.0），匹配成功则该风格 +1 分（tie-break）。
   - **LLM 摘要**：生成包含推荐理由、优缺点分析的中文说明文本。
3. 重新按分数排序，确定最终推荐。
4. 生成对比矩阵（每项含 style、score、recommendation_type、pros、cons、key_reasons、topology_mermaid）。
5. 生成风险与缓解建议（3 种核心风格有针对性建议，其余使用通用回退）。
6. 汇总决策依据（规则理由 + LLM 投票结果 + LLM 摘要）。

**输出**：
```json
{
  "recommended_style": "Event-Driven Architecture",
  "alternative_styles": ["Microservices", "Layered Architecture"],
  "decision_basis": {
    "rule_engine": ["高并发场景处理能力强", "实时性需求匹配度高", ...],
    "rule_engine_raw": ["matches feature: high_concurrency", ...],
    "llm_summary": "1. 推荐架构：...\n2. 推荐理由：...\n3. 优缺点分析：...",
    "llm_vote": "Event-Driven Architecture"
  },
  "comparison_matrix": [...],
  "risk_and_suggestions": {
    "main_risks": [...],
    "suggestions": [...]
  }
}
```

**异常处理**：
- LLM 摘要失败 → 返回结构化模板生成的降级摘要（`_fallback_summary()`）。
- LLM 投票失败 → 不加分，维持规则排名。
- 候选列表为空 → 返回空推荐并标记异常。

**对应代码**：
- [services/evaluation_agent/app/main.py](services/evaluation_agent/app/main.py) — 完整实现
- LLM 摘要：`llm_summary()` (第 51-90 行)
- LLM 投票：`llm_vote_style()` (第 120-160 行)
- 降级摘要：`_fallback_summary()` (第 93-117 行)
- 混合推理入口：`evaluate()` (第 250-300 行)

**验收标准**：
- `recommended_style` 不为空。
- `comparison_matrix` 至少 3 行，每行包含 style/score/pros/cons/key_reasons 五项。
- `decision_basis` 包含 rule_engine 和 llm_summary 字段。
- `risk_and_suggestions` 至少包含 1 条风险与 1 条建议。

**实现状态**：✅ 已完成

## 6. AI 系统特有需求分析

### AIR-01 需求描述的模糊性处理

**问题**：用户输入的需求描述可能是模糊的、不完整的，或含有否定语义（如"不需要高并发"）。

**解决方案**：
1. **规则引擎兜底**：10 个维度的关键词词典覆盖常见表述，提供确定性基线。
2. **否定语义过滤**：`filter_negation()` 函数（[services/requirements_agent/app/main.py:49-58](services/requirements_agent/app/main.py#L49-L58)）检查关键词前 6 个字符范围内是否包含否定词（"不需要、不要求、无需、没有、无高、非"），命中则排除该关键词。
3. **LLM 语义补全**：规则命中维度 ≤ 2 时（特征过于稀疏），调用 LLM 进行二次语义分析，以 temperature=0.1 低温度参数降低随机性。LLM 不可用或调用失败时静默降级为规则结果。

**代码证据**：`llm_semantic_supplement()` 函数（[services/requirements_agent/app/main.py:80-128](services/requirements_agent/app/main.py#L80-L128)）

---

### AIR-02 LLM 幻觉控制

**问题**：大语言模型可能生成与知识库不一致的推荐结论，或编造不存在的架构风格。

**解决方案**：
1. **规则优先原则**：候选集 Top 3 完全由规则评分决定，LLM 不参与候选集生成，仅参与评估阶段的投票与摘要。
2. **LLM 投票限定候选范围**：投票 Prompt 明确要求"仅从给定候选列表中选择"，要求返回"仅风格名称，无额外内容"（[services/evaluation_agent/app/main.py:129-134](services/evaluation_agent/app/main.py#L129-L134)）。
3. **投票结果校验**：LLM 返回的风格名必须在候选列表中才被采纳，否则丢弃（[services/evaluation_agent/app/main.py:152-155](services/evaluation_agent/app/main.py#L152-L155)）。
4. **低温度参数**：LLM 投票使用 temperature=0.0，LLM 摘要使用 temperature=0.3，抑制随机生成。

---

### AIR-03 推荐结果可解释性

**问题**：推荐结论需附带可追溯的证据链，支持评审和答辩。

**解决方案**：
系统输出三层证据链，实现从数据到结论的完整追溯：

| 层级 | 证据来源 | 输出字段 | 示例 |
|------|---------|---------|------|
| L1 特征证据 | 关键词命中列表 | `feature_hits` | `"high_concurrency": ["高并发", "万人"]` |
| L2 推理证据 | 规则评分理由 | `decision_basis.rule_engine` | `"高并发场景处理能力强"` |
| L3 语义解释 | LLM 摘要 | `decision_basis.llm_summary` | 推荐理由、优缺点分析文本 |

---

### AIR-04 LLM 调用失败降级

**问题**：外部 LLM 服务可能因网络波动、配额耗尽或配置缺失而不可用。

**解决方案**：三级降级策略

| 降级场景 | 检测方式 | 降级行为 |
|---------|---------|---------|
| LLM 未配置 | `LLM_API_BASE` / `LLM_API_KEY` / `LLM_MODEL` 任一为空 | 跳过所有 LLM 调用，直接进入纯规则模式 |
| LLM 调用超时 | `httpx.AsyncClient(timeout=15.0/20.0/25.0)` | 捕获异常，使用 fallback 摘要 |
| LLM 返回异常 | try/except 包裹 | 日志记录警告，使用 fallback 摘要 |

**降级输出特征**：
- `llm_summary` 返回模板化规则摘要（含推荐理由、优缺点列表）。
- `llm_vote` 返回 `null`。
- 核心推荐链路（提取→匹配→排序）不受影响。

**代码证据**：
- 未配置检测：[services/evaluation_agent/app/main.py:52-54](services/evaluation_agent/app/main.py#L52-L54)
- 超时控制：[services/evaluation_agent/app/main.py:82](services/evaluation_agent/app/main.py#L82) (timeout=25.0)、[services/requirements_agent/app/main.py:108](services/requirements_agent/app/main.py#L108) (timeout=15.0)
- Fallback 逻辑：[services/evaluation_agent/app/main.py:88-90](services/evaluation_agent/app/main.py#L88-L90)、[services/evaluation_agent/app/main.py:93-117](services/evaluation_agent/app/main.py#L93-L117)

---

### AIR-05 Prompt 注入防护

**问题**：用户可能在需求文本中注入 System Prompt 指令，尝试篡改 LLM 行为。

**当前实现**：
- 用户输入仅作为 user message 传递给 LLM，与 system message 分离。
- LLM 调用中 temperature 设为低值（0.0-0.3），降低对注入指令的敏感性。
- LLM 投票输出需精确匹配候选列表中的风格名，拒绝自由文本。

**后续优化**：
- 可增加输入文本的敏感字符过滤层。
- 可对用户输入做最大长度限制，防止上下文窗口溢出攻击。

---

### AIR-06 规则引擎兜底

**问题**：需要在无 LLM 参与时保证推荐质量不低于可接受基线。

**解决方案**：
- 规则评分独立运行，不依赖 LLM。
- 降级摘要 `_fallback_summary()` 可从候选数据中自动构建结构化的推荐理由和优缺点分析。
- 测试验证：纯规则模式下的回归测试各项指标仍为 100%。

**代码证据**：[services/evaluation_agent/app/main.py:93-117](services/evaluation_agent/app/main.py#L93-L117) — `_fallback_summary()` 函数

---

### AIR-07 结果一致性验证

**问题**：同一需求多次输入应产生稳定的推荐结果（可复现性）。

**当前实现**：
- 规则引擎确定的评分函数在相同输入下保证相同输出（确定性）。
- LLM 投票和摘要可能因模型随机性产生轻微差异，但规则分数主导排序，最终推荐受规则约束。
- LLM 投票使用 temperature=0.0 最小化随机性。

**验证机制**：回归测试中 20 条用例批量执行，检查结构一致性指标（top3_rate、matrix_rate 等）。

## 7. 非功能需求

### NFR-01 可用性

- 网关服务应对上游故障进行统一捕获，返回 HTTP 502 而非崩溃。
- 单个 Agent 故障不应导致网关进程退出。
- LLM 不可用时核心推荐链路应保持可用。
- 前端网络异常时展示错误文本并恢复正常交互状态。

**当前实现**：网关 `try/except` 捕获 `httpx.HTTPError` 并返回 502（[services/api_gateway/app/main.py:94-96](services/api_gateway/app/main.py#L94-L96)）。

---

### NFR-02 响应性能

- 单次推荐请求的完整链路（提取→匹配→评估）目标时延 ≤ 15s。
- LLM 调用超时限制：需求提取 15s、LLM 投票 20s、LLM 摘要 25s。
- 网关级总超时：30s（httpx.AsyncClient timeout）。

**实际表现**：回归测试平均时延约 6-10s（受 LLM 网络延迟影响）。纯规则模式（无 LLM）预计 ≤ 2s。

---

### NFR-03 可扩展性

- **知识库扩展**：通过 POST /styles 接口动态新增架构风格，无需重启服务即可生效（matching-agent 每次请求实时从 knowledge-base 拉取数据）。
- **服务扩展**：微服务架构支持对单个 Agent 独立扩容或替换。
- **LLM 模型切换**：通过环境变量 `LLM_MODEL`、`LLM_API_BASE` 切换，支持任何兼容 OpenAI 协议的模型。

---

### NFR-04 可维护性

- 5 个独立微服务，各服务代码量控制在 300 行以内。
- 服务职责单一：需求提取 / 规则匹配 / 评估决策 / 知识库管理 / API 编排。
- 每个服务提供 `/health` 端点用于运维检查。
- 日志统一格式，包含时间戳、日志级别和服务名。

---

### NFR-05 安全与配置管理

- LLM API Key 通过环境变量注入，禁止硬编码。
- `.env` 文件已加入 `.gitignore`，防止凭据泄露。
- 跨域配置（CORS）允许所有来源（调试阶段），生产环境可收紧。
- 对外仅暴露前端端口（3000）和网关端口（8000），内部服务通过 Docker 内部网络通信。

---

### NFR-06 可测试性

- 20 条典型需求场景测试数据覆盖 10+ 业务领域。
- 提供冒烟测试（`run_smoke.py`）、回归测试（`run_regression.py`）、单元测试（pytest，23 条）三层测试体系。
- 单元测试覆盖所有 3 个 Agent + 知识库模块。

## 11. 约束与假设

### 11.1 技术约束

- 后端使用 Python 3.10+ / FastAPI 框架。
- 服务间通信使用 HTTP + JSON（非 gRPC 或消息队列）。
- LLM 通信采用 OpenAI 兼容协议（`/chat/completions` 端点）。

- 知识库使用 JSON 文件持久化（非数据库管理系统）。

### 11.2 假设

1. LLM 服务（DeepSeek/通义千问/OpenAI）的 API 端点遵循 `/chat/completions` 标准路径。
2. 用户输入的需求文本为中文或中英混合文本。
3. 单次推荐请求的并发量不高（课程演示场景，非生产高并发场景）。
4. Docker 运行环境已安装 Docker Desktop 并分配足够内存（≥ 4GB）。

## 14. 当前边界与后续演进

### 当前实现边界

1. 知识库使用 JSON 文件存储，未使用 Neo4j 等图数据库，因此不支持跨风格的关系推理。
2. 案例反馈收集已实现接口（POST/GET /feedback），但尚未实现从反馈数据中自动学习更新权重的闭环机制。
3. 风险与建议针对 3 种核心风格有针对性内容（事件驱动/微服务/分层），其余 7 种使用通用回退模板。
4. 规则权重（tag 匹配 +2 / 额外规则 +1 / LLM 投票 +1）为硬编码值，未抽取为可配置参数。

### 后续演进方向

1. **知识库升级**：将 JSON 文件存储升级为 Neo4j 图数据库，支持风格间关系推理。
2. **自动权重学习**：基于反馈数据，定期分析用户确认/修正分布，自动调整风格 tags 权重。
3. **ADR 决策追溯**：增加架构决策记录（ADR）的自动生成与存储功能。
4. **缓存机制**：对相同需求文本增加请求级缓存，避免重复计算。
5. **规则引擎配置化**：将评分权重、额外规则抽取为配置文件，支持不修改代码的动态调整。
6. **强化风险覆盖**：为全部 10 种架构风格提供针对性的风险与缓解建议。
