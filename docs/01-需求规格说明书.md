## 2. 项目背景与问题定义

### 2.1 问题定义

在软件体系结构课程中，学生需要根据需求描述选择合适的架构风格。传统人工评审面临三个核心问题：

| 问题 | 说明 |
|------|------|
| 特征识别效率低 | 依赖个人经验判断，缺乏系统化的特征提取方法 |
| 候选方案不稳定 | 不同评审者可能得出不同结论，缺乏可复现的决策过程 |
| 决策理由不完整 | 缺少量化的评分依据和风险分析，结论可追溯性弱 |

### 2.2 解决方案概述

本系统通过"规则引擎 + LLM 协同"的双驱动方案解决上述问题：

- **规则引擎** 保证推荐下限（确定性、可追溯、LLM 不可用时仍可用）
- **LLM** 提升推荐上限（语义理解、解释生成、模糊需求处理）
- **知识图谱 (Neo4j)** 提供图结构化的架构知识推理（可选后端，有 JSON fallback）
- **Few-shot Prompt** 约束 LLM 输出格式与候选范围
- **ADR 决策记录** 提供完整的决策溯源链

### 2.3 项目范围

**包含**:
- 自然语言需求特征提取（规则 + LLM 语义补全）
- 架构风格匹配与评分（规则引擎 + 图谱证据融合 + 学习权重）
- 架构组合推荐（5 种预定义组合，含评分公式）
- LLM 辅助评估（投票 + 摘要 + 混合推理）
- 架构重构建议（坏味检测 + 模式推荐 + 迁移方案）
- ADR 架构决策记录自动生成与持久化
- 请求级 LLM 缓存（memory / sqlite 双后端）
- Web 前端可视化（11 个展示区块）
- Docker Compose 容器化部署（8 个容器）
- 三层测试体系（单元 79 条 + 冒烟 20 条 + 回归 20 条）
- 自动验收脚本（15 大类检查）

**不包含**:
- 生产级 Kubernetes 集群部署
- OAuth/OIDC 用户认证
- 多语言需求支持（当前仅中文）
- 实时协作编辑

## 4. 用户角色与使用场景

| 角色 | 说明 | 关注点 |
|------|------|--------|
| 需求输入者 | 输入自然语言需求描述的用户（课程学生/开发者） | 快速获得架构推荐与决策依据 |
| 系统管理员 | 维护知识库和系统配置的人员 | 知识库扩展、LLM 配置、系统健康监控 |

### 典型使用场景

1. **新课设项目启动**: 学生描述软件需求，系统推荐 Top 3 架构风格含对比矩阵。

3. **答辩展示**: 评委查看 ADR 决策记录和知识图谱证据，验证推荐可解释性。
4. **知识库扩展**: 管理员通过 API 新增架构风格或组合方案。
5. **反馈学习**: 用户提交对推荐结果的确认或修正，系统累积学习权重。

## 6. 功能需求

### FR-01 自然语言需求输入与校验

**描述**: 系统应支持通过 Web 页面与 REST API 两种方式接收自然语言需求文本，并对输入做基础校验。

**输入**: 自然语言文本字符串（最短 10 个字符）。

**处理过程**:
1. 前端 `<textarea>` 接收用户输入，通过 HTTP POST 发送至网关。
2. 网关通过 Pydantic 模型 `RecommendRequest` 进行字段校验 (`min_length=10`)。
3. 校验不通过时返回 HTTP 422 及字段级错误提示。
4. 请求期间按钮置为 disabled 状态 ("分析中...")，防止重复提交。

**输出**: 校验通过进入推荐流程；不通过返回错误信息。

**对应模块**: `frontend/index.html`、`services/api_gateway/app/main.py:RecommendRequest`

**验收标准**:
- 合法输入返回 HTTP 200
- 输入不足 10 字符返回 HTTP 422
- 前端网络异常时不崩溃，显示错误提示并恢复按钮状态

**实现状态**: ✅ 已实现

### FR-02 需求特征提取

**描述**: 系统应从非结构化需求文本中识别关键非功能特征与约束特征，并返回特征命中的关键词证据。

**输入**: 自然语言需求文本。

**处理过程**:
1. 需求文本转为小写进行关键词匹配。
2. 遍历 10 个特征维度的关键词词典，命中则标记为 True。
3. 对命中关键词执行否定语义过滤（如"不需要高并发"中"高并发"被否定词修饰，不计入）。
4. 如规则命中维度 ≤ 2（特征过于稀疏）且 LLM 已配置，则调用 LLM 做语义补全。LLM 不可用时静默降级，维持规则结果。
5. 返回每个维度的布尔值及命中关键词列表。

**10 个特征维度**:

| # | 维度 | 英文标识 | 典型关键词 |
|---|------|----------|-----------|
| 1 | 高并发 | high_concurrency | 高并发、并发、万人、海量用户、秒杀、高吞吐 |
| 2 | 实时性 | real_time | 实时、即时、在线、低延迟、消息推送、IM |
| 3 | 可靠性 | reliability | 可靠、高可用、容灾、容错、稳定、不丢消息 |
| 4 | 可扩展性 | scalability | 扩展、弹性、扩容、横向扩展、可伸缩 |
| 5 | 复杂业务 | complex_business | 复杂业务、交易、审批、规则引擎、工作流 |
| 6 | 强一致性 | strict_consistency | 强一致、事务、金融、账务、ACID |
| 7 | 部署约束 | deployment_constraint | 本地部署、私有化、边缘、多地域、离线、内网 |
| 8 | 数据密集型 | data_intensive | 数据流、ETL、流处理、日志、批处理 |
| 9 | 多团队协作 | team_size_large | 多团队、跨团队、多人协作、并行开发 |
| 10 | 安全性 | security | 安全、加密、认证、鉴权、审计、合规、脱敏 |

**输出**:
```json
{
  "features": {"high_concurrency": true, "real_time": true, "security": false, ...},
  "feature_hits": {"high_concurrency": ["高并发", "万人"], "real_time": ["实时"], ...}
}
```

**对应模块**: `services/requirements_agent/app/main.py`

**验收标准**:
- 输出包含 10 维 features 布尔映射和 feature_hits 关键词证据
- 否定语义场景下被否定词修饰的关键词不计入
- LLM 未配置时特征提取仍正常运行

**实现状态**: ✅ 已实现

### FR-03 Few-shot Prompt 语义补全

**描述**: 系统应在 LLM 语义补全阶段使用 Few-shot Prompt 示例约束 LLM 输出格式与语义判断标准。

**输入**: 用户需求文本 + 规则引擎已命中的特征维度。

**处理过程**:
1. Requirements Agent 尝试导入 `common.prompts.requirements_few_shot` 模块（6 个标注示例）。
2. 示例覆盖场景：模糊高并发、否定语义（非实时）、安全合规、数据密集型、强一致性、架构重构倾向。
3. 将示例与用户需求拼接构建完整 Prompt，强制要求 LLM 输出严格 JSON 格式。
4. 若 few-shot 模块不可用，降级为零样本 Prompt（仅 system message + 用户需求）。
5. LLM 返回非 JSON 输出时，尝试解析 markdown 代码块；解析失败则丢弃 LLM 结果。

**Evaluation Agent 对应**: 使用 `common.prompts.evaluation_few_shot` 模块（3 个示例：Event-Driven、Microservices、Layered/CQRS），每个示例含完整的四部分输出（推荐架构、推荐理由、优缺点分析、风险与建议）。

**对应模块**:
- `services/common/prompts/requirements_few_shot.py`（6 个示例）
- `services/common/prompts/evaluation_few_shot.py`（3 个示例）
- `services/requirements_agent/app/main.py:llm_semantic_supplement()`
- `services/evaluation_agent/app/main.py:llm_summary()`

**验收标准**:
- requirements few-shot 含 6 个示例，每个含 user text + expected output
- evaluation few-shot 含 3 个示例，每个含完整的四部分输出结构
- Few-shot 不可用时自动降级为零样本，不中断流程

**实现状态**: ✅ 已实现（9 个标注示例，覆盖两类场景）

### FR-04 知识库查询与管理

**描述**: 系统应提供知识库的查询与扩展接口，支持运行时查询和新增架构风格数据。知识库支持 JSON 文件持久化（默认）和 Neo4j 图数据库（可选）两种后端。

**输入**:
- 查询: `GET /styles`（无参数）
- 新增: `POST /styles`（StylePayload: name、name_zh、tags、best_for、pros、cons、topology_mermaid）

**处理过程**:
1. 根据 `KNOWLEDGE_BACKEND` 环境变量（`json` | `neo4j` | `auto`）调度到对应后端。
2. `auto` 模式下优先 Neo4j，失败自动回退 JSON。
3. JSON 后端读写 `architecture_styles.json` 文件。
4. Neo4j 后端执行 Cypher 查询，返回节点及关系数据。

**输出**: `{"styles": [...]}`（全部 10 种风格），每种含 name、name_zh、tags、best_for、pros、cons、topology_mermaid。

**对应模块**:
- `services/knowledge_base/app/main.py`
- `services/knowledge_base/app/json_repository.py`
- `services/knowledge_base/app/graph_repository.py`

**验收标准**:
- GET /styles 返回风格数量 ≥ 10
- POST /styles 新增风格后再次查询可获取
- Neo4j 不可用时 `auto` 模式自动回退 JSON，不中断服务

**实现状态**: ✅ 已实现（双后端架构，含自动 fallback）

### FR-05 Neo4j 知识图谱查询

**描述**: 系统应支持基于 Neo4j 图数据库的架构知识图谱查询，提供图结构化的特征-风格匹配推理。

**输入**: 活跃特征列表。

**处理过程**:
1. matching-agent 调用 knowledge-base `POST /graph/match` 端点。
2. GraphRepository 在 Neo4j 中查询：每个活跃特征通过 `HAS_QUALITY` 关系匹配 ArchitectureStyle 节点。
3. 同时查询匹配的 Scenario（通过 `SUITABLE_FOR`）、Risk（通过 `HAS_RISK`）和可组合风格（通过 `COMPLEMENTS`）。
4. 每个属性匹配 +2 分。
5. 图谱加分与规则引擎得分通过 `blend_scores()` 融合，上限为规则得分的 50%（防止图谱主导）。
6. Neo4j 不可用时返回 `{"available": false}`，图谱得分为 0，不阻断流程。

**对应模块**:
- `services/knowledge_base/app/graph_repository.py:graph_match()`
- `services/matching_agent/app/graph_matcher.py:fetch_graph_evidence()`, `blend_scores()`
- `services/knowledge_base/init/init_neo4j.py`（初始化脚本，175 行）

**验收标准**:
- Neo4j 运行时返回图谱证据（匹配属性、场景、风险、可组合风格）
- Neo4j 不可用时返回 `available: false`，规则评分不受影响
- 图谱加分不超过规则得分的 50%

**实现状态**: ✅ 已实现（可选后端，完整 fallback，3 个集成测试）

### FR-06 规则引擎评分与候选匹配

**描述**: 系统应基于提取的特征，通过规则引擎对所有架构风格进行评分，并从知识库中匹配候选架构风格。

**输入**: 特征布尔映射 `{"high_concurrency": true, "real_time": true, ...}`。

**处理过程**:
1. 从 knowledge-base (`GET /styles`) 获取全部 10 种架构风格。
2. 对每种风格执行三层评分：
   - **标签基础分**: 风格 tags 与活跃特征匹配，每命中 1 项 +2 分
   - **学习权重加分**: 从 `GET /feedback/weights` 获取学习权重，>= 2 次确认的特征-风格关联 +1 分
   - **特定风格加成规则**: Event-Driven + 高并发 +1；Microservices + 多团队 +1；Layered + 强一致性 +1；Pipeline-Filter + 数据密集型 +1；CQRS + 复杂业务 +1
3. 按分数降序排列。
4. 候选集生成采用"主流优先 + 按分补齐"策略：Layered、Microservices、Event-Driven 始终出现在候选列表中。
5. 如所有风格得分均为 0（无有效特征信号），返回三种主流架构作为基线。
6. 从 knowledge-base `POST /graph/match` 获取图谱证据，通过 `blend_scores()` 融合。

**输出**: 候选架构列表（Top 3），每项含 style、score、pros、cons、best_for、topology_mermaid、key_reasons、graph_evidence。

**对应模块**:
- `services/matching_agent/app/main.py:score_style()`, `match()`
- `services/matching_agent/app/graph_matcher.py`

**验收标准**:
- 每次响应候选数量 = 3
- 当存在有效特征信号时，至少包含 1 种主流架构
- 当无特征信号时，返回三种主流架构作为基线
- 候选含完整的 pros/cons/best_for/topology_mermaid 字段

**实现状态**: ✅ 已实现

### FR-07 LLM 辅助评估

**描述**: 系统应通过 LLM 对候选架构进行辅助评估，包括风格投票和自然语言摘要生成。

**输入**: 原始需求文本 + 提取特征 + 候选架构列表（Top 3）。

**处理过程**:
1. 并行执行两项 LLM 任务（`asyncio.gather`）：
   - **LLM 投票** (`llm_vote_style`): 从候选集中选出最佳风格。temperature=0.0，要求仅返回风格名称。
   - **LLM 摘要** (`llm_summary`): 生成包含推荐理由、优缺点分析的中文说明文本。temperature=0.3。
2. LLM 投票结果校验：返回风格名必须在候选列表中才被采纳。
3. 若 LLM 投票与规则排名第一不同，投票风格 +1 分作为 tie-break（混合推理）。
4. LLM 摘要使用 Few-shot Prompt（3 个示例），不可用时降级为零样本。
5. LLM 完全不可用时（API Key 未配置），跳过所有 LLM 调用，使用 `_fallback_summary()`。

**输出**: `decision_basis` 对象，包含 `rule_engine`（规则理由）、`llm_vote`（LLM 投票结果）、`llm_summary`（LLM 摘要或 fallback 模板）。

**对应模块**: `services/evaluation_agent/app/main.py`

**验收标准**:
- `decision_basis` 包含 rule_engine 和 llm_summary 字段
- LLM 投票结果经过候选列表校验才被采纳
- LLM 不可用时 `llm_vote` 为 null，`llm_summary` 返回模板化摘要

**实现状态**: ✅ 已实现

### FR-08 架构风格 Top 3 推荐与报告输出

**描述**: 系统应基于规则排名与 LLM 辅助评估，生成包含推荐结论、对比矩阵和风险评估的完整推荐报告。

**输入**: 原始需求 + 提取特征 + 候选列表 + LLM 评估结果。

**处理过程**:
1. 按最终分数（规则分数 + 可能的 LLM 投票加分）降序排列。
2. 确定 `recommended_style`（得分最高者）和 `alternative_styles`（其余 2 个）。
3. 构建对比矩阵（3 行 × 6 列：推荐类型、风格名称、得分、关键理由、优点、缺点），含 Mermaid 拓扑图。
4. 动态生成风险与缓解建议（Event-Driven/Microservices/Layered 有预定义风险条目，其余 7 种使用通用回退模板）。
5. 构建推荐组合（若 combination_candidates 非空）。

## 7. AI 系统特有需求分析

### AIR-01 不确定性来源

**问题**: 系统引入 LLM 带来了多种不确定性——模型输出随机性、上下文理解偏差、输出格式不稳定。

**识别的不确定性来源**:

| 来源 | 影响环节 | 严重程度 |
|------|----------|----------|
| 需求描述模糊/不完整 | 特征提取 | 中 |
| LLM 语义理解偏差 | 语义补全 | 中 |
| LLM 幻觉（编造风格名） | 投票与摘要 | 高 |
| LLM 输出格式不稳定 | 语义补全、摘要 | 高 |
| 外部 API 可用性 | 全链路 | 中 |
| 模型温度参数 | 投票、摘要 | 低 |

**应对**: 每个不确定性来源均有对应的需求进行约束（AIR-02 至 AIR-09）。

---

### AIR-02 模糊需求处理

**问题**: 用户输入的需求描述可能是模糊的、不完整的，或含有否定语义（如"不需要高并发"）。

**解决方案**:
1. **规则引擎兜底**: 10 个维度的关键词词典覆盖常见中文表述，提供确定性基线。特征稀疏时（≤ 2 维命中）触发 LLM 补全。
2. **否定语义过滤**: `filter_negation()` 检查关键词前 6 个字符范围内是否包含否定词（"不需要"、"不要求"、"无需"、"没有"、"无高"、"非"），命中则排除该关键词。
3. **Few-shot 引导**: 6 个标注示例中专门包含否定语义场景，训练 LLM 理解中文否定表达。
4. **低温度参数**: LLM 补全使用 temperature=0.1，最大化确定性。

**对应模块**: `services/requirements_agent/app/main.py:keyword_hits()`, `filter_negation()`, `llm_semantic_supplement()`

**验收标准**: 否定语义场景下被否定词修饰的关键词不计入特征。

---

### AIR-03 LLM 幻觉风险控制

**问题**: LLM 可能生成与知识库不一致的推荐结论，或编造不存在的架构风格名称。

**解决方案**:
1. **规则优先原则**: 候选集 Top 3 完全由规则评分决定，LLM 不参与候选集生成。LLM 仅在评估阶段参与投票（从给定候选列表中选择）和摘要（基于给定候选列表生成）。
2. **候选范围限定**: 投票 Prompt 明确要求"仅从给定候选列表中选择"，要求返回"仅风格名称，无额外内容"。
3. **投票结果强制校验**: LLM 返回的风格名必须在候选列表（string match）中才被采纳，否则丢弃。
4. **低温度约束**: 投票使用 temperature=0.0（贪婪解码），摘要使用 temperature=0.3。
5. **Json 输出约束**: Few-shot Prompt 要求输出严格 JSON，便于自动解析和校验。

**对应模块**: `services/evaluation_agent/app/main.py:llm_vote_style()`, `llm_summary()`

**验收标准**: LLM 返回的风格名不在候选列表中时被丢弃，不污染推荐结果。

---

### AIR-04 Prompt 约束设计

**问题**: LLM 的输出格式和质量高度依赖 Prompt 设计。需要确保 Prompt 在不同场景下稳定产出结构化输出。

**解决方案**:
1. **System/User 消息分离**: 用户输入仅作为 user message 传递，与 system message 中的指令和约束分离，降低注入风险。
2. **Few-shot 结构化示例**: 9 个标注示例（6 for requirements + 3 for evaluation），每个示例输出严格遵守目标 JSON schema。
3. **输出格式硬约束**: Prompt 中明确要求"仅返回有效 JSON，不包含任何解释"（requirements）或"输出必须包含以下四个部分"（evaluation）。
4. **非 JSON 输出容错**: 尝试从 markdown 代码块（```json ... ```）中提取；提取失败则丢弃 LLM 结果，使用规则 fallback。
5. **长度限制**: 用户输入不做硬截断（当前），Prompt token 数依赖模型上下文窗口。

**对应模块**:
- `services/common/prompts/requirements_few_shot.py`
- `services/common/prompts/evaluation_few_shot.py`
- `services/requirements_agent/app/main.py:llm_semantic_supplement()`
- `services/evaluation_agent/app/main.py:llm_summary()`

**验收标准**: Few-shot 示例完整覆盖目标输出结构；非 JSON 输出被容错处理。

---

### AIR-05 候选范围限定

**问题**: 若 LLM 完全自由选择架构风格，可能推荐知识库中不存在的风格，导致后续对比矩阵、风险建议等环节无法提供具体数据。

**解决方案**: 采用"闭集推荐"策略：
1. 候选集由规则引擎从知识库 10 种风格中产生（确定性）。
2. LLM 投票时 Prompt 明确定义 `candidates` 数组并要求从中选择。
3. 投票结果经 `_validate_vote()` 校验：返回风格名必须在 `[c["style"] for c in candidates]` 中。
4. 场景-风格映射通过知识库（`best_for` 字段）交叉验证。

**实现状态**: 已实现于 `services/evaluation_agent/app/main.py:llm_vote_style()` 的投票校验逻辑。

---

### AIR-06 LLM 调用降级

**问题**: 外部 LLM 服务可能因网络波动、配额耗尽或配置缺失而不可用。系统必须保证核心推荐链路在 LLM 不可用时仍可正常运行。

**解决方案**: 三级降级策略。

| 降级场景 | 检测方式 | 降级行为 |
|---------|---------|---------|
| LLM 未配置 | `LLM_API_BASE`/`LLM_API_KEY`/`LLM_MODEL` 任一为空 | 跳过所有 LLM 调用，进入纯规则模式 |
| LLM 调用超时 | httpx.AsyncClient timeout（extract: 15s, vote: 20s, summary: 25s） | 捕获异常，使用 fallback 摘要 |
| LLM 返回异常/非 JSON | try/except 包裹 + JSON 解析校验 | 日志记录警告，使用 fallback 摘要 |

**降级输出特征**:
- `llm_summary`: 返回 `_fallback_summary()` 生成的模板化规则摘要（含推荐理由、优缺点结构）
- `llm_vote`: 返回 null
- `features`: 维持规则引擎结果
- 核心推荐链路（提取→匹配→排序）不受影响

**对应模块**:
- `services/evaluation_agent/app/main.py:_fallback_summary()`
- `services/requirements_agent/app/main.py:llm_semantic_supplement()`

**验收标准**: LLM 未配置时系统仍返回含推荐结论、对比矩阵、风险建议的完整响应。

---

### AIR-07 规则引擎校验

**问题**: LLM 输出需要经过确定性规则校验，确保推荐结果不偏离知识库约束。

**解决方案**:
1. **规则分数主导排序**: 候选人按规则分数排名，LLM 投票仅 +1 分作为 tie-break，不翻转 2 分以上差距的排名。
2. **知识库交叉验证**: 推荐的风格必须在 `architecture_styles.json` 中存在，`best_for`、`pros`、`cons` 均来自知识库数据（非 LLM 编造）。
3. **主流风格保底**: Layered、Microservices、Event-Driven 三种主流架构始终出现在候选列表中，确保推荐不偏离业界共识。
4. **场景匹配约束**: matching-agent 的 `score_style()` 包含针对特定场景的硬编码加成规则（如 Event-Driven + 高并发 +1），体现领域最佳实践。

**对应模块**: `services/matching_agent/app/main.py`

**验收标准**: 推荐风格始终在 10 种预定义风格之内，无外部编造。

---

### AIR-08 可解释证据链

**问题**: AI 辅助决策需要提供清晰的推理依据，支持用户复核和评审审查。

**解决方案**: 系统输出三层证据链，实现从数据到结论的完整追溯。

| 层级 | 证据来源 | 输出字段 | 确定性 |
|------|---------|---------|--------|
| L1 特征证据 | 关键词命中列表 | `feature_hits` | 确定性（规则） |
| L2 匹配证据 | 规则评分理由 + 图谱证据 | `key_reasons`, `graph_evidence` | 确定性（规则 + 图谱） |
| L3 语义解释 | LLM 摘要 | `llm_summary` | 概率性（LLM） |
| L4 决策记录 | ADR | `adr.adr_id`, `adr.adr_summary` | 确定性（持久化） |

**前端呈现**: 每一层证据在独立的前端区块中展示（识别特征、图谱证据、推荐摘要、ADR 信息）。

**对应模块**: `services/evaluation_agent/app/main.py`（evidence 构造）, `frontend/index.html`（evidence 渲染）

**验收标准**: 回归测试 `explainability_rate = 100%`。

---

### AIR-09 人类反馈闭环

**问题**: 系统初始推荐可能不完全符合用户预期，需要反馈机制持续优化。

**解决方案**:
1. **反馈收集**: `POST /feedback` 接口接收用户对推荐结果的确认或修正。
2. **自动权重学习**: `_update_learned_weights()` 在每次确认反馈后自动更新特征-风格权重。
3. **权重累积阈值**: 学习权重需 >= 2 次确认才在评分中生效（+1 分），防止单次反馈过度影响。
4. **权重查询**: `GET /feedback/weights` 接口可查看当前学习权重状态。
5. **反馈统计**: `GET /feedback/stats` 提供准确率和风格级统计，用于评估系统表现。

**对应模块**:
- `services/knowledge_base/app/json_repository.py:_update_learned_weights()`
- `services/knowledge_base/data/learned_weights.json`

**验收标准**: 用户确认反馈后学习权重自动更新；>= 2 次确认的权重影响匹配评分。

## 11. 约束与假设

### 11.1 技术约束

| 约束 | 说明 |
|------|------|
| 后端框架 | Python 3.11+ / FastAPI |
| 服务间通信 | HTTP + JSON（非 gRPC 或消息队列） |
| LLM 协议 | OpenAI 兼容协议 (`/chat/completions`) |
| 部署方式 | Docker Compose 单机模式（非 Kubernetes） |
| 知识库主存储 | JSON 文件持久化（Neo4j 为可选增强） |
| 缓存实现 | 进程内缓存（memory/sqlite），非 Redis 等分布式缓存 |
| 前端渲染 | 客户端渲染（CDN 加载 Mermaid.js + marked.js） |
| 语言 | 仅中文需求输入与中文输出 |

### 11.2 设计假设

1. LLM 服务（DeepSeek）的 API 端点遵循 `/chat/completions` 标准路径。
2. 用户输入的需求文本为中文或中英混合文本。
3. 单次推荐请求的并发量不高（课程演示场景，非生产高并发场景）。
4. Docker 运行环境已安装 Docker Desktop 并分配足够内存（≥ 4GB）。
5. Neo4j 为可选组件，JSON 后端可独立支撑全功能运行。
6. langgraph 库为可选依赖，手动编排可独立支撑全功能运行。
