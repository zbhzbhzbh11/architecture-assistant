## 八、运行方式

### 方式一：Docker Compose（推荐，含 Neo4j）

```bash
# 1. 配置 LLM
cp .env.example .env
# 编辑 .env，填入 LLM_API_KEY

# 2. 启动全部服务（8 容器）
docker compose up --build

# 3. 初始化 Neo4j 图数据（首次）
docker compose exec knowledge-base python init/init_neo4j.py

# 4. 访问
# 前端: http://localhost:3000
# Swagger API 文档: http://localhost:8000/docs
# 缓存统计: http://localhost:8000/cache/stats
```

### 方式二：本地 Python（无 Docker，无需 Neo4j）

```bash
# 1. 配置 LLM
cp .env.example .env

# 2. 快速启动
bash start.sh --with-frontend

# 3. 访问
# 前端: http://localhost:3000
# 网关: http://localhost:8000/docs
```

> 无 Docker 模式下 Neo4j 不可用，知识库自动使用 JSON fallback，功能不受影响。

## 十一、注意事项

1. **LLM 配置**: 必须配置 `.env` 中的 `LLM_API_KEY` 才能使用 LLM 功能。不配置 LLM 时系统以纯规则模式运行，核心功能完整。
2. **Neo4j 可选**: Neo4j 仅在 Docker Compose 模式下可用。无 Neo4j 时知识库自动使用 JSON fallback，所有核心功能不受影响。
3. **端口占用**: 确保 8000-8005、3000、7474、7687 端口未被占用。
4. **Docker 内存**: 建议分配 ≥ 4GB 内存给 Docker Desktop。
5. **演示前检查**: 运行 `python scripts/check_assignment.py` 确认 43/43 通过。
6. **`.env` 不入库**: API Key 存储在 `.env` 中，已加入 `.gitignore`，提交时使用 `.env.example` 模板。
7. **测试数据**: `requirements_cases.json` 中 10 号用例（"多团队协作开发SaaS平台，需要模块独立发布"）预期 Microservices + team_size_large + scalability。

## 十二、当前不足与后续改进

| # | 不足 | 影响 | 后续方向 |
|---|------|------|---------|
| 1 | 规则权重硬编码 (+2/+1) | 评分不可动态调优 | 抽取为配置文件或环境变量 |

| 7 | Neo4j 集成测试需运行环境 | CI 中跳过 | 增加 Testcontainers 或 CI Neo4j 服务 |
| 8 | 前端无自动化 E2E | 回归依赖人工 | 引入 Playwright/Cypress |
