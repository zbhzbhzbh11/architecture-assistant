# 答辩讲稿与问答

## 1. 5 分钟系统演示讲稿

### 1.1 开场（约 30 秒）
各位老师好，我们的课题是“基于大模型的软件架构风格智能助手”。系统目标是把自然语言需求自动转成架构建议，并给出可解释的决策依据。今天演示从需求输入到最终推荐的完整闭环。

### 1.2 演示步骤（约 3 分钟）
步骤一：输入需求文本
- 现场输入一条包含并发、实时、扩展性等非功能约束的需求描述。

步骤二：触发推荐
- 前端调用网关接口，网关串联需求解析 Agent、架构匹配 Agent、评估 Agent。

步骤三：查看中间结果
- 展示提取后的结构化特征，说明系统识别到了哪些关键需求信号。
- 展示候选 Top3 架构，说明主流架构覆盖情况。

步骤四：查看最终结果
- 展示最终推荐架构。
- 展示对比矩阵与风险建议，回答“为什么选它、代价是什么”。

步骤五：说明可解释性
- 强调系统不仅给结论，还给规则依据与 LLM 解释，便于评审与追溯。

### 1.3 结果总结（约 1 分钟）
我们在 20 条典型场景上做了回归测试，20 条全部通过，Top3 完整率、主流覆盖率、可解释率均为 100%。这说明系统在课程要求范围内具备稳定可用性与可展示性。

### 1.4 收尾（约 30 秒）
演示完成。下面进入架构设计讲解，重点说明为什么采用微服务 + 多 Agent + LLM 协同的方案，以及如何做工程化权衡。

## 2. 15 分钟架构设计讲解

### 2.1 需求与约束（约 2 分钟）
业务目标：
- 从非结构化需求中提取特征并输出架构建议。
- 候选至少 3 种，且覆盖主流架构。
- 给出最终推荐和可解释依据。

非功能目标：
- 可扩展：可新增架构风格、可扩展规则与案例。
- 高可用：单点故障不应阻断核心推荐链路。
- 可维护：模块职责清晰，便于迭代。
- 可解释：输出过程证据而非黑盒结论。

### 2.2 候选架构方案对比（约 3 分钟）

| 方案 | 可扩展性 | 高可用性 | 维护成本 | 交付风险 | 演进难度 | 适配结论 |
|---|---|---|---|---|---|---|
| 单体应用 + 内部模块 | 中 | 中 | 低 | 低 | 中 | 适合快速原型，不利于后续能力拆分 |
| 微服务 + 多 Agent 协作 | 高 | 高 | 中 | 中 | 中 | 最匹配课程目标与能力演进 |
| 事件驱动 + 异步编排为主 | 高 | 高 | 高 | 高 | 高 | 适合大规模生产，课程周期内实施成本偏高 |

结论：
- 课程场景下选择微服务 + 多 Agent 最平衡，既满足可扩展和可解释，也控制了交付复杂度。

### 2.3 最终架构设计（约 4 分钟）
核心组件：
- API Gateway：统一入口与编排。
- Requirements Agent：提取需求特征。
- Matching Agent：基于知识库与规则计算候选架构。
- Evaluation Agent：融合规则结果与 LLM 说明，输出最终建议。
- Knowledge Base：维护架构风格属性、优缺点与适配场景。
- Frontend：展示推荐结果、对比矩阵与拓扑说明。

协作链路：
- 用户输入需求 -> 网关 -> 需求解析 -> 架构匹配 -> 决策评估 -> 返回结果。
- 返回内容包含特征、候选、最终推荐、解释与风险建议。

设计原则：
- 职责单一：每个 Agent 只做一类问题。
- 可替换：模型供应商可替换，规则可调整。
- 可追溯：关键输出都能映射到中间证据。

### 2.4 关键技术决策（约 3 分钟）
为什么要规则 + LLM 协同：
- 纯规则可控但泛化弱。
- 纯 LLM 灵活但稳定性与可审计性不足。
- 协同方案兼顾工程可控和语义理解能力。

为什么要知识库模块：
- 把架构风格信息从代码中解耦，支持持续扩展。
- 便于后续做案例学习与权重优化。

为什么保留降级机制：
- 当 LLM 波动或不可用时，规则引擎仍可输出可用结果，保证系统可用性下限。

### 2.5 测试与验证（约 2 分钟）
测试策略：
- 使用 20 条典型场景做端到端回归。
- 同时检查正确性、完整性、解释性和稳定性。

结果：
- 通过率 100%。
- Top3 完整率、主流覆盖率、可解释率、矩阵产出率均为 100%。
- 平均时延 6108.16 ms，满足课程展示需求。

## 3. 5 分钟问答题库（高频问题）

### 3.1 为什么不用纯 LLM 直接推荐？
回答：
- 纯 LLM 在稳定性、可重复性与可审计性方面存在波动。
- 我们采用规则作为可控底座，LLM 负责语义补强与说明生成，既保证工程可控，又保留智能能力。

### 3.2 为什么用微服务而不是单体？
回答：
- 需求解析、匹配、评估属于不同演进节奏与职责边界。
- 微服务便于独立扩展和替换，后续接入新模型或新规则时影响面更小。

### 3.3 你们的可解释性如何证明？
回答：
- 系统输出三类证据：特征命中、候选与评分依据、最终推荐与对比矩阵。
- 回归测试中可解释率 100%，每条样例都能追溯推荐理由。

### 3.4 平均时延 6 秒会不会太高？
回答：
- 对课程演示和离线评审是可接受的。
- 若面向高实时生产场景，需要做缓存、并行化、异步化和模型调用优化，我们已给出具体优化路径。

### 3.5 如何保证知识库可持续演进？
回答：
- 已提供可扩展接口，可新增风格与属性。
- 后续可引入案例学习、权重学习和图数据库形态，逐步从静态知识走向数据驱动。

### 3.6 如果 LLM 服务不可用怎么办？
回答：
- 系统具备降级路径：规则引擎可独立完成候选与推荐，LLM 仅影响增强说明，不阻断核心业务结果。

### 3.7 该方案的失败模式有哪些？
回答：
- 失败模式一：特征抽取偏差导致候选偏移。
- 失败模式二：外部模型波动导致说明不稳定。
- 失败模式三：复杂场景下响应时延抖动。
- 对策：扩充词典与测试集、引入监控告警、增加缓存与异步策略。

### 3.8 后续迭代计划是什么？
回答：
- 第一阶段：性能优化与监控补齐。
- 第二阶段：知识库图谱化与案例学习。
- 第三阶段：支持多模型路由与成本治理。
