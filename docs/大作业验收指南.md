## 1. 项目简介

本项目是一个**基于大模型的软件架构风格智能助手**，核心能力闭环为：

```
用户自然语言需求 → 特征提取 → 候选架构匹配 → LLM 混合评估 → 最终推荐 + 对比矩阵 + 拓扑图 + 风险建议
```

- **后端：** Python 3.11+ / FastAPI / Uvicorn，5 个独立微服务
- **前端：** 原生 HTML/CSS/JS + Mermaid.js 拓扑图渲染
- **LLM：** DeepSeek v4-flash（OpenAI 兼容协议，可替换为通义千问/OpenAI）
- **知识库：** JSON 结构，10 种架构风格，含属性/场景/优缺点/拓扑图定义
- **测试：** 20 条场景用例，回归测试 100% 通过

## 5. 15 分钟架构设计讲解提纲

### 5.1 业务背景（1 分钟）

- 软件架构选型依赖个人经验，缺乏系统化工具
- 需求描述存在模糊性和不确定性
- 需要一个可解释、可追溯的架构推荐辅助系统

### 5.2 总体架构（2 分钟）

- **C4-Context：** 用户 → 架构风格智能助手 → LLM 服务（DeepSeek）
- **C4-Container：** 前端容器 + API 网关 + 需求解析 Agent + 架构匹配 Agent + 评估生成 Agent + 知识库服务
- **技术选型理由：** FastAPI（轻量/异步/自动文档）、httpx（异步服务间通信）、Mermaid.js（零依赖拓扑图）

### 5.3 微服务划分（2 分钟）

- 按领域职责拆分 5 个微服务：需求提取、规则匹配、LLM 评估、知识库管理、网关编排
- 划分合理性：① 异构成分隔离（规则引擎 vs LLM 调用）② 独立演进（更换模型/规则不影响其他服务）③ 故障隔离（LLM 波动不影响知识库查询和特征提取）

### 5.4 Agent 协作机制（2 分钟）

- **Pipeline-Agent 模式：** 每个 Agent 仅关注自己的领域逻辑
- **协作时序：** Gateway 依次调用 Extract → Match → Evaluate，上一个输出 = 下一个输入
- **特点：** 职责单一、可替换、可独立测试、LLM 仅在评估阶段参与以降低不确定性扩散

### 5.5 LLM 集成方案（1 分钟）

- OpenAI 兼容协议，支持 DeepSeek/通义千问/OpenAI 一键切换
- **提示词工程：** 两套 Prompt（投票用 t=0 追求确定性 / 说明用 t=0.2 保留表达空间）
- **鲁棒性设计：** 20s 超时、异常捕获、未配置/失败自动降级为纯规则模式
- 环境变量注入（Key 不硬编码），.env 自动加载

### 5.6 知识库设计（1 分钟）

- 10 种架构风格 × 5 个字段（name/tags/best_for/pros/cons/topology_mermaid）
- 支持 POST /styles 动态扩展
- JSON 结构化存储，文档已说明可升级至 Neo4j 图数据库

### 5.7 混合推理机制（1 分钟）

- **规则引擎：** 标签匹配（+2/tag）+ 额外专家规则（+1），生成确定性候选排序
- **LLM 增强：** 投票 +1 分 tie-break + 中文说明生成
- **协同价值：** 规则保下限（确定性/可审计），LLM 提上限（语义理解/自然语言解释）

### 5.8 可视化设计（1 分钟）

- 对比矩阵：5 列 HTML 动态表格，字段来自后端 API
- 拓扑图：Mermaid.js 渲染，10 种风格各有专属定义，前端从 matrix 数据动态读取
- 特征卡片：显示命中的需求维度和关键词证据

### 5.9 异常处理和缓存（1 分钟）

- 网关统一捕获上游异常，标准 HTTP 502/500 返回
- LLM 调用超时 20s + 失败降级文案
- 前端按钮防重复提交 + 错误分支处理
- 已实现 `trust_env=False` 绕过系统代理干扰
- 已规划：请求级缓存、LLM 调用并行化

### 5.10 测试验证（1 分钟）

- 20 条典型需求场景（覆盖 10+ 领域）
- 回归测试 20/20 通过，5 项指标 100%
- 平均响应时延 ~10s（含 LLM 两次串行调用）
- pytest 单元测试 + 冒烟测试 + 自动验收检查脚本

- 案例学习为接口预留级别（有反馈收集/统计，未实现自动权重更新）

## 7. 老师可能提问与回答

### Q1：为什么选择微服务架构？

**回答：** 需求解析、规则匹配和 LLM 评估属于不同演进节奏和职责边界。将 LLM 调用独立为 evaluation-agent 可以隔离外部依赖风险（LLM 波动不影响特征提取和知识库查询）。每个模块可独立替换——例如 matching-agent 将来升级为 Neo4j 驱动时，其他服务无需修改。

### Q2：Agent 如何协作？

**回答：** 采用 **Pipeline-Agent 模式**。API Gateway 作为编排器，依次调用 requirements-agent（提取特征）→ matching-agent（匹配候选）→ evaluation-agent（评估决策）。上一个 Agent 的输出 = 下一个 Agent 的输入。这种模式职责单一、链路清晰、可独立测试。

### Q3：LLM 在系统中起什么作用？

**回答：** LLM 在 evaluation-agent 中承担两个角色：① **投票角色**（llm_vote_style）：从候选架构中选出最佳方案，结果以 +1 分轻量加权参与最终排序；② **解释角色**（llm_summary）：生成中文推荐说明，提升可解释性。LLM 不参与候选集生成（候选由规则引擎确定），以控制不确定性。

### Q4：为什么还需要规则引擎？只用 LLM 推荐不是更简单？

**回答：** 纯 LLM 推荐存在三个问题：① **稳定性**：相同需求多次调用可能得到不同推荐；② **可审计性**：无法解释"为什么给 Event-Driven 打 7 分"；③ **幻觉风险**：可能推荐不存在的架构风格。规则引擎提供确定性的评分基线，LLM 在此基线上做语义增强，两者互补。

### Q5：如何保证推荐结果可靠？

**回答：** 三层保障：① **规则兜底**：候选集由标签匹配和专家规则决定，不依赖 LLM；② **主流白名单**：Layered/Microservices/Event-Driven 强制优先，避免冷门风格；③ **LLM 失败降级**：LLM 不可用时自动回退纯规则模式，核心推荐链路不中断。回归测试 20/20 全部通过验证了可靠性。

### Q6：知识库如何扩展和进化？

**回答：** 三个层次：① **静态扩展**：`POST /styles` 接口直接新增架构风格（含 name/tags/best_for/pros/cons/topology_mermaid），新增后即时生效。② **案例学习闭环**：`POST /feedback` 收集用户对推荐结果的确认或修正，系统自动从需求文本中提取特征维度（`_extract_features_from_requirement()`），更新特征-风格关联权重到 `learned_weights.json`。matching-agent 每次评分时拉取学习权重，对 ≥2 次确认的特征-风格组合给予 "learned boost" +1 分，实现"反馈越多，推荐越准"的知识进化闭环。③ **统计监控**：`GET /feedback/stats` 查看各风格准确率和确认分布，`GET /feedback/weights` 查看当前学习权重。

### Q7：为什么推荐这个最终架构？（以即时通讯系统为例）

**回答：** 该需求触发了高并发（万人在线）、实时性（消息实时）、可靠性、可扩展性四个特征。Event-Driven Architecture 标签直接覆盖 4 个特征，得分最高（7 分）。额外规则"高并发有利于事件驱动"再加 1 分。LLM 投票虽选择了 Microservices，但规则评分主导排序，因此最终推荐 Event-Driven。**关键设计原则：规则决定下限，LLM 提供参考。**

### Q8：如何处理需求描述的模糊性？

**回答：** 双重策略：① **规则优先**：通过关键词词典（10 维度 × 约 100 个关键词）做第一轮确定性提取；② **LLM 补全**：当规则命中维度 ≤2 时（需求描述过于简略），自动触发 LLM 语义补全（temperature=0.1，要求返回严格 JSON）。需求规格说明书中 AIR-01/AIR-02 详细说明了不确定性处理方案。

### Q9：如果 LLM 调用失败怎么办？

**回答：** 系统设计了完整的降级路径：① 20s 超时后自动放弃；② try-except 捕获所有异常；③ llm_summary 返回"LLM not configured, fallback to rule-based explanation."；④ llm_vote_style 返回 None（不参与加分）。降级后规则引擎仍可独立完成候选推荐和排序，核心业务不受影响。

### Q10：测试数据集如何设计？

**回答：** 20 条用例覆盖 10+ 领域（IM 通讯、电商、审批、日志、银行、教育、物联网、短视频、政务、SaaS、医疗、数据同步、外卖、证券、客服、科研、校园、支付），每个用例包含触发不同特征维度的关键词。回归测试自动检查 5 项指标：Top3 完整率、主流覆盖率、推荐产出率、可解释率、矩阵产出率。

### Q11：对比矩阵的评分依据是什么？

**回答：** 评分 = 标签匹配分 + 额外规则分 + LLM 投票分。标签匹配：style.tags 与 features 的交集，每命中一个 +2 分；额外规则：3 条专家经验（如"高并发→事件驱动 +1"）；LLM 投票：+1 分轻量 tie-break。评分可追溯——每条理由都记录在 key_reasons 中。

### Q12：拓扑图如何生成？

**回答：** 知识库中每种架构风格预定义了 `topology_mermaid` 字段（Mermaid 图语法字符串）。该字段随 matching-agent 的评分结果一路传递到 evaluation-agent 的 comparison_matrix，前端 `buildTopology()` 从矩阵中找到推荐风格对应的 Mermaid 定义，由 Mermaid.js 渲染为 SVG。

### Q13：系统有哪些局限？

**回答：** ① 知识进化算法当前为基于计数累加的简单线性加分（≥2 阈值触发），尚未实现衰减、归一化等高级学习机制——算法框架已预留升级接口，可平滑升级为贝叶斯权重更新或 TF-IDF 加权；② 知识库为 JSON 存储，非 Neo4j 图数据库（已预留扩展路径并在设计文档中说明）；③ 需求解析的 LLM 语义补全仅在规则命中少时触发，正常情况仍以关键词为主；④ 平均时延 ~10s，适合课程演示和离线评审场景；⑤ Mermaid 渲染依赖外部 CDN，离线环境不可用。

### Q14：后续如何改进？

**回答：** 短期：请求级缓存减少 LLM 重复调用、LLM 两次调用并行化减少 ~40% 时延、补充单元测试。中期：知识库升级 Neo4j 支持关系推理、规则权重配置化。长期：基于反馈数据自动学习标签权重、架构决策溯源（ADR）、多模型路由和成本治理。

### Q15：本项目和普通 ChatGPT 问答有什么区别？

**回答：** ChatGPT 是通用对话模型，架构推荐可能存在幻觉且无法解释评分依据。本系统有四个本质区别：① **规则引擎保底**——候选集来自确定性的标签匹配，而非黑盒生成；② **知识库约束**——推荐范围限定在 10 种经过学术验证的架构风格内；③ **可追溯**——三层证据链（特征命中 → 规则评分 → LLM 解释），每步都可审计；④ **可扩展**——知识库支持动态注入新风格，反馈数据可持续优化权重。

|---|---|
| 为什么没用 Neo4j？ | "当前阶段使用 JSON 结构化存储满足了 10 种风格的管理和扩展需求。架构设计文档已明确说明 JSON → Neo4j 的升级路径：将 tags 关系转为图边，best_for 转为属性约束，支持关系推理。后续升级只需修改 knowledge-base 服务的数据层，不影响对外接口。" |
| 案例学习为什么算法比较简单？ | "当前实现的是完整的知识进化闭环——从反馈收集、特征提取、权重更新到评分生效四个环节都已打通。算法选择了基于计数累加 + 阈值触发的轻量方案，原因是：① ≥2 的阈值避免了单次偶然反馈的噪声干扰；② 跨特征加分（learned boost 不依赖于风格已有标签）比仅更新风格标签权重更灵活；③ 框架支持平滑升级——将 `_update_learned_weights` 中的 +1 计数替换为贝叶斯公式或 TF-IDF 加权，无需改动其他模块。" |
