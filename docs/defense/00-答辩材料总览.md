## 1. 项目一句话介绍

**架构风格智能助手** — 输入自然语言需求描述，经过"规则引擎 + 知识图谱 + LLM"三层推理，输出 Top 3 架构风格推荐及完整决策溯源报告。

---

## 2. 系统核心目标

| # | 目标 | 实现路径 |
|---|------|---------|
| G1 | 自然语言理解 | 10 维关键词词典 + LLM 语义补全（Few-shot） |
| G2 | 多源推理融合 | 规则引擎（确定性）+ Neo4j 图谱（关系推理）+ LLM（语义理解） |
| G3 | 可解释决策 | 四层证据链：特征证据 → 匹配证据 → 语义解释 → ADR 决策记录 |
| G4 | 高可用降级 | LLM / Neo4j / LangGraph 三级独立降级，核心链路不中断 |
| G5 | 知识可进化 | 用户反馈 → 自动学习权重 → 评分修正的闭环 |

## 5. Agent 列表（4 个业务 Agent）

| Agent | 端点 | 推理方法 | LLM 角色 | 不可用时的降级 |
|-------|------|---------|---------|--------------|
| **Requirements Agent** | POST /extract | LLM 优先 + 词典溯源 | 语义补全（命中 ≤2 时触发） | 纯规则结果 |
| **Matching Agent** | POST /match | 规则引擎 + 图谱融合 + 学习权重 | 不参与 | 无图谱得分（JSON 模式） |
| **Evaluation Agent** | POST /evaluate | 规则排名 + LLM 投票 tie-break + LLM 摘要 | 投票（限定候选集）+ 摘要生成 | 模板化摘要 + 无投票加分 |
| **Refactoring Agent** | POST /refactor | 规则引擎主导 | 可选润色 | 纯规则模板 |

**Agent 协作模式**: Pipeline-Agent 顺序编排（extract → match → evaluate），Refactoring Agent 为非阻塞异步调用。

## 9. 规则引擎与混合推理

### 9.1 评分公式（确定性，始终运行）

```
总分 = 标签基础分（tag ∈ features → +2/项）
     + 学习权重分（feature-style 关联确认 ≥2 次 → +1/项）
     + 特定风格加成（6 条硬编码规则 → +1/条）
```

**6 条特定加成规则**:
- Event-Driven + high_concurrency → +1
- Microservices + team_size_large → +1
- Layered + strict_consistency → +1
- Pipeline-Filter + data_intensive + real_time → +1
- CQRS + data_intensive + high_concurrency → +1
- Microservices + high_concurrency + strict_consistency → +1

### 9.2 候选筛选策略

- **主流优先**: Layered / Microservices / Event-Driven 始终出现在 Top 3 中
- **按分补齐**: 剩余空位按分数降序填充
- **零信号基线**: 全部得分为 0 时返回三种主流架构作为对比基线

### 9.3 混合推理层次

```
Layer 1: 规则引擎评分        [始终运行，确定性]
Layer 2: 知识图谱推理        [Neo4j 可用时运行，上限 50%]
Layer 3: LLM 投票/摘要       [LLM 可用时运行，+1 tie-break]
```

**核心理念**: "规则保证下限，图谱增强关系，LLM 提升上限"。三层独立运行、松耦合。

## 10. Few-shot、缓存、ADR、组合推荐、重构建议

### 10.1 Few-shot Prompt

| 模块 | 文件 | 示例数 | 覆盖场景 | Fallback |
|------|------|--------|---------|---------|
| Requirements | `requirements_few_shot.py` (87行) | 6 | 模糊高并发、否定语义、安全合规、数据密集、强一致、重构 | 零样本 Prompt |
| Evaluation | `evaluation_few_shot.py` (119行) | 3 | Event-Driven、Microservices、Layered+CQRS | 零样本 Prompt |

**设计要点**: 每个示例含完整输入输出对，Fallback 链为 ImportError → 内联零样本 Prompt。

### 10.2 LLM 缓存

| 维度 | 实现 |
|------|------|
| **粒度** | 请求级（以 requirement + model + knowledge_version 为键） |
| **键生成** | SHA256(requirement + LLM_MODEL + MD5(storage_file)[:8]) |
| **后端** | memory（默认，TTL 3600s，线程安全）或 sqlite（跨重启持久化） |
| **自动失效** | knowledge_version 基于 `architecture_styles.json` 内容 MD5，知识库更新后旧缓存自动失效 |
| **管理 API** | `GET /cache/stats`（命中率统计）、`POST /cache/clear` |
| **前端展示** | `cache_hit: true` → 绿色徽章"缓存命中"；`false` → 蓝色徽章"实时计算" |
| **定位** | 轻量实现（内存/SQLite，非 Redis），适合课程项目单机场景 |

### 10.3 ADR 架构决策记录

| 维度 | 实现 |
|------|------|
| **生成时机** | 每次 `POST /evaluate` 完成后自动生成 |
| **ID 格式** | `ADR-YYYYMMDD-NNN`（如 ADR-20260513-001） |
| **存储** | JSON 文件持久化（始终可用）+ Neo4j 可选同步（ADR 节点 + RECOMMENDS/BASED_ON 关系） |
| **API** | `POST /adr`（创建）、`GET /adr?limit=N`（列表）、`GET /adr/{id}`（详情） |
| **容错** | ADR 写入失败 → `adr_status = "failed"`，不阻塞推荐主流程 |
| **前端** | ADR 信息卡片显示 ID、状态、摘要，可点击查询 |

### 10.4 架构组合推荐

| 维度 | 实现 |
|------|------|
| **组合数据** | `architecture_combinations.json`：5 种预定义组合 |
| **评分公式** | Σ 组件风格分数 + 特征覆盖互补加分 + 图谱 COMPLEMENTS 加分 − 复杂度惩罚 |
| **5 种组合** | Microservices+Event-Driven / Layered+CQRS / Pipeline-Filter+Event-Driven / Hexagonal+Microservices / CQRS+Event Sourcing |
| **输出** | 推荐组合含名称、分数、组件、协同效应、复杂度、Mermaid 拓扑图 |
| **前端** | 组合推荐卡片展示组件列表、协同描述、风险、备选方案 |

### 10.5 架构重构建议

| 维度 | 实现 |
|------|------|
| **坏味检测** | 5 种：单体耦合、扩展瓶颈、发布缓慢、数据层耦合、遗留系统（基于关键词匹配 + 特征推断） |
| **重构模式** | 5 种：绞杀者模式、防腐层、模块化单体优先、CQRS 迁移、事件驱动迁移 |
| **迁移方案** | ≥ 3 个阶段步骤 + 风险提示 + 缓解措施（规则模板生成） |
| **LLM 角色** | 可选润色步骤描述和风险说明，保持结构不变；不可用时使用纯规则模板 |
| **调用方式** | API Gateway 非阻塞异步调用，超时 8s，失败不影响主流程 |

## 12. 答辩时最应该突出的 5 个亮点

### 亮点 1: 三层驱动架构的设计哲学

**"规则保证下限，图谱增强关系，LLM 提升上限。"**

- 规则引擎提供确定性基线，在任何情况下都有基于知识库的、可追溯的推荐
- Neo4j 知识图谱通过关系遍历发现隐含的架构关联（如 CQRS COMPLEMENTS Event-Driven）
- LLM 在规则和图谱基础上提供语义级辅助判断
- **每层独立可降级**：LLM 不可用 → 纯规则推荐仍 100% 可用；Neo4j 不可用 → JSON fallback 全功能完整

**这是课程核心概念"混合推理"的工程落地范例。**

### 亮点 2: 完整的 AI 工程化设计

不只是"调一个 LLM API"，而是完整的工程化闭环：

- **Few-shot Prompt Engineering**: 9 个标注示例 + 零样本 fallback
- **LLM 输出校验**: 投票结果必须在候选列表内才采纳，非 JSON 输出容错解析
- **三级降级策略**: LLM 超时/不可用/幻觉 → 自动回退规则基线
- **请求级缓存**: memory/SQLite 双后端，知识库版本感知，自动失效
- **ADR 决策溯源**: 每次推荐自动生成架构决策记录，可查询可审计

### 亮点 3: Neo4j 知识图谱 + JSON Fallback 的务实架构

- Neo4j 5.26 图数据库建模 4 类节点 + 4 类关系（+ ADR 扩展），Cypher 查询实现图结构推理
- 同时提供 JSON 文件后端零外部依赖运行
- `auto` 模式自动探测：Neo4j 可用 → 图推理增强；不可用 → 静默回退 JSON
- 图谱加分上限 50% 融合策略确保规则引擎的主导地位

**体现"过度工程 vs 务实权衡"的判断力。**

### 亮点 4: 4 个独立 Agent + LangGraph 双引擎编排

- 4 个业务 Agent 职责清晰：提取（规则+LLM）、匹配（规则+图谱）、评估（LLM投票/摘要）、重构（坏味检测）
- **双引擎设计**: LangGraph StateGraph 优先；langgraph 未安装或运行时异常 → 自动回退手动编排
- 两套编排代码共存，功能等价，方便对比调试

### 亮点 5: 全链路可观测性与证据链

- **四层证据链**: L1 特征关键词 → L2 规则理由+图谱证据 → L3 LLM 语义解释 → L4 ADR 决策记录
- **前端 11 个独立展示区**: 矩阵、拓扑图、图谱证据、组合、重构、ADR、工作流追踪、缓存状态
- **各个环节可监控**: `/health`、`/cache/stats`、`/graph/status`、`/feedback/stats`
- **全面测试覆盖**: 79 单元 + 20 冒烟 + 20 回归 + 43 自动验收，全部 100% 通过

## 13. 当前项目需要谨慎表述的地方

### 13.1 轻量实现（功能完整但非生产级）

| 项 | 当前实现 | 生产级替代 |
|----|---------|-----------|

| **LangGraph** | 可选编排引擎。手动编排可独立支撑全功能运行 | "系统实现了双引擎设计，LangGraph 为可选引擎。手动编排在功能上完全等价，可独立运行" |
| **LLM** | 可选智能增强。纯规则模式可独立支撑核心推荐 | "LLM 为可选增强层。纯规则模式下推荐、矩阵、风险建议功能完整，回归测试 100% 通过" |

### 13.3 Fallback 实现（有完整降级，不缺失）

| 场景 | Fallback 行为 | 表述建议 |
|------|-------------|---------|
| LLM 未配置/不可用 | 跳过所有 LLM 调用，使用规则模板摘要 | "全功能模式下 LLM 提供语义增强；LLM 不可用时自动降级为纯规则模式，核心推荐链路不受影响" |
| Neo4j 不可用 | 自动回退 JSON 文件后端 | "系统在 Neo4j 不可用（如未安装 Docker 或资源不足）时自动回退 JSON 后端，不影响核心功能" |
| LangGraph 不可用 | 自动回退手动编排 | "LangGraph 编排引擎不可用时，系统自动切换至内置的手动编排，用户无感知" |
| ADR 写入失败 | adr_status="failed"，推荐主流程继续 | "ADR 写入采用非阻塞设计，失败不中断推荐主流程" |

### 13.4 硬编码与覆盖不全（已知局限，非缺陷）

| 局限 | 说明 |
|------|------|
| **规则权重硬编码** | +2/+1 分值为经验设定值，未通过数据驱动方法（如网格搜索）自动调优 |
| **风险建议覆盖** | 3 种主流架构（Event-Driven/Microservices/Layered）有针对性风控条目，其余 7 种使用通用回退模板 |
| **学习权重算法** | 简单计数累加，无衰减机制和归一化处理 |
| **测试覆盖** | 3 条 Neo4j 集成测试需 Neo4j 运行环境（CI 中自动跳过）；前端测试为手动验证，无自动化 E2E |
| **多语言支持** | 当前仅支持中文需求输入，英文关键词词典已部分覆盖但未完整适配 |
| **用户认证** | 无 OAuth/JWT 用户认证，反馈接口无来源区分 |

**答辩时话术**: "这些是已知的工程权衡——在有限时间内优先保证核心机制的完整性，而非追求每个维度的极致覆盖。每个局限都有明确的后续改进路径。"
