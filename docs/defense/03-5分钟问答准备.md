## 一、项目定位（3 题）

### Q1: 你的系统和普通 ChatGPT 问答有什么区别？

**老师为什么问**: 验证你是否理解"AI 系统"与"调 API"的本质区别。这是区分高分和及格的关键问题。

**推荐回答**:

> 三个本质区别。
>
> 第一，**确定性不同**。你问 ChatGPT"我的系统用什么架构"，它可能推荐一个不存在的风格名。我们的系统候选集由规则引擎从知识库 10 种预定义风格中产生，LLM 只能从这个候选列表里投票——这是"闭集推荐"，不是"开集生成"。
>
> 第二，**可追溯性不同**。ChatGPT 的输出是一个黑盒。我们的系统输出四层证据链——特征关键词、规则评分公式、图谱匹配证据、LLM 语义解释——每一步都有据可查，还有 ADR 决策记录持久化存储。
>
> 第三，**可用性不同**。ChatGPT 挂了你就没招了。我们系统 LLM 不配置照样跑——规则引擎提供确定性基线，纯规则模式回归测试 20/20 通过。

**加分点**: 提到"闭集推荐 vs 开集生成"这个对比，展示你对 LLM 局限性的工程化理解。

**避坑提醒**: 不要说"ChatGPT 不好"——要说"场景不同"。ChatGPT 适合开放式对话，我们的场景需要确定性推荐和可追溯决策。

**证据**: `matching_agent/app/main.py` 的 `MAINSTREAM_STYLES` 和 `score_style()` 函数；`evaluation_agent/app/main.py` 的 `llm_vote_style()` 投票校验逻辑。

### Q2: 为什么说它是 Compound AI System？

**老师为什么问**: "Compound AI System"是课程可能涉及的前沿概念，老师想看你是否理解这个概念，以及是否真正按这个范式设计了系统。

**推荐回答**:

> Compound AI System 的核心思想是"用多个专用组件协同工作，而不是依赖单一模型解决所有问题"。
>
> 我们的系统体现了这个范式的三个关键特征——
>
> 第一，**组件专业化**。规则引擎负责确定性匹配，Neo4j 负责图结构关系推理，LLM 只负责语义理解和自然语言生成。每个组件做自己最擅长的事。
>
> 第二，**组件可组合**。三层推理通过 `blend_scores()` 函数融合——规则分是基础，图谱分做增强（上限 50%），LLM 投票做 tie-break（+1 分）。融合逻辑是显式的、可审计的，不是模型内部的隐式权重。
>
> 第三，**组件可降级**。任一层不可用，系统退化为下层能力。LLM 不可用时规则引擎独立运行，核心推荐链路不受影响。

**加分点**: 用 `blend_scores()` 的实际融合公式说明"可组合"——规则分 + min(图谱分, 规则分//2) + LLM 投票加分。

**避坑提醒**: 不要只说概念定义。必须落到你系统中的具体实现——规则引擎、Neo4j、LLM 分别对应什么代码。

**证据**: `matching_agent/app/graph_matcher.py:blend_scores()`；`evaluation_agent/app/main.py` 的混合推理逻辑；`docs/02-架构设计文档.md` 第 7 节。

### Q3: 这个项目解决了软件体系结构中的什么问题？

**老师为什么问**: 验证你的项目有明确的问题定义，而不是"为了用技术而用技术"。

**推荐回答**:

> 解决了软件体系结构课程中的一个核心痛点：**如何从自然语言需求描述出发，系统化地、可复现地推荐合适的架构风格。**
>
> 具体三个问题——
>
> 一是**特征识别效率低**。学生看一段需求描述，靠个人经验判断该用什么架构。我们通过 LLM 独立语义分析 + 词典溯源，把特征提取系统化了。
>
> 二是**候选方案不稳定**。同一个人不同时间、不同人同一需求，可能得出不同结论。我们的规则引擎评分公式是确定性的——相同输入一定得到相同分数。
>
> 三是**决策理由不完整**。传统做法是"我觉得微服务合适"，但说不出量化依据。我们输出包含评分明细、关键词证据、图谱匹配记录和 ADR 决策记录的四层证据链。

**加分点**: 对齐课程教学目标——"架构风格选型"是软件体系结构的核心教学内容。

**避坑提醒**: 不要说"解决了所有软件架构问题"。限定范围——解决的是"架构风格推荐"这个子问题。

**证据**: `docs/01-需求规格说明书.md` 第 2 节"项目背景与问题定义"。

## 二、微服务架构（4 题）

---

### Q4: 为什么采用微服务，而不是单体？

**老师为什么问**: 这是软件体系结构课程的**核心考查点**。老师想听你论证架构决策背后的驱动因素，而非背诵微服务的优点列表。

**推荐回答**:

> 五个具体理由，每个都有对应的架构驱动因素——
>
> 第一，**领域解耦**。特征提取是 LLM 分析（网络 I/O 密集），架构评估是 LLM 调用（网络 I/O 密集），知识库是数据存取（I/O 密集）。三种不同的技术特征，混在一个进程里边界模糊。
>
> 第二，**故障隔离**。LLM 调用有外部依赖风险。在单体里，一次 LLM 超时可能导致整个请求失败。在我们的微服务架构中，evaluation-agent 的 LLM 超时只影响摘要质量，requirements-agent 和 matching-agent 的结果不受影响。
>
> 第三，**异构集成**。只有 matching-agent 需要 Neo4j Python driver，其他 3 个 Agent 不需要。微服务让依赖按需加载，不污染其他服务的镜像。
>
> 第四，**独立部署**。每个服务有独立 Dockerfile，可以独立构建、独立重启。代码量也控制了——最大单文件 393 行，平均约 200 行，便于代码走查。
>
> 第五，**技术灵活**。所有配置通过环境变量注入——LLM 模型切换、缓存后端切换、知识库后端切换都不需要改代码。

**加分点**: 每个理由都对应一个具体的架构驱动因素（而非泛泛的"微服务优点"）；提到"单文件最大 393 行"这个具体数字。

**避坑提醒**: 承认代价——HTTP 通信有额外延迟，缺少消息队列的异步解耦，不适用于超高并发场景。但课程演示场景完全够用。

**证据**: `docker-compose.yml`（8 个 service 定义）；`docs/02-架构设计文档.md` 第 5 节微服务划分表。

### Q5: API Gateway 的作用是什么？

**老师为什么问**: 网关是微服务架构的标准模式，老师想看你是否正确理解了它的职责边界。

**推荐回答**:

> Gateway 是系统的唯一对外入口，承担四个核心职责——
>
> 第一，**请求校验**。通过 Pydantic `RecommendRequest` 模型验证输入（`min_length=10`），非法输入直接返回 422，不会进入编排链路。
>
> 第二，**缓存检查**。请求到达后先查缓存——命中直接返回，跳过所有 Agent 调用。缓存键基于 `SHA256(requirement + model + knowledge_version)`。
>
> 第三，**双引擎编排**。优先使用 LangGraph StateGraph（4 节点顺序执行），LangGraph 不可用时自动回退手动编排（httpx 顺序调用）。两套编排功能完全等价。
>
> 第四，**非阻塞增强**。推荐完成后异步调用 refactoring-agent 获取重构建议，超时 8s，失败不影响主流程。

**加分点**: 提到"双引擎编排"——不只是代理转发，而是有编排逻辑；提到"非阻塞调用"——重构建议不影响主链路。

**避坑提醒**: 不要说网关做"负载均衡"或"限流"——当前系统没有实现这些。

**证据**: `services/api_gateway/app/main.py` (226行)；`services/api_gateway/app/langchain_workflow.py` (166行)。

### Q6: 服务拆分是否过度设计？8 个容器对一个课程项目太多了吧？

**老师为什么问**: 这是最可能被追问的问题。老师可能在测试你是否为了堆技术而拆分，还是每个拆分都有合理理由。

**推荐回答**:

> 这个问题的答案是"取决于你用什么标准衡量"。
>
> 如果用"生产环境 QPS"衡量，8 个容器对这个量级来说确实过度。
>
> 但如果用"架构设计教学价值"衡量——每个服务恰好展示了一个微服务的典型特征：requirements-agent 展示了"规则引擎 + LLM 协同"，matching-agent 展示了"评分算法 + 外部数据源集成"，evaluation-agent 展示了"LLM 调用故障隔离"，knowledge-base 展示了"多后端策略模式"，refactoring-agent 展示了"非阻塞异步服务"。
>
> 而且每个服务的代码量非常小——最大 393 行，平均约 200 行。这不是把一个大服务拆成了 8 个大服务，而是把本就在逻辑上独立的模块物理隔离了。
>
> 另外，Neo4j 是**可选**的——`KNOWLEDGE_BACKEND=json` 时它完全不参与系统运行。你完全可以只用 7 个容器甚至去掉 Docker 直接用 Python 跑，核心功能完整。

**加分点**: 承认"对生产环境来说过度"这个事实，展示你有工程判断力；接着论证"对教学场景来说合理"。

**避坑提醒**: 绝对不要辩解"8 个不多，微服务都这样"——这会显得你没有成本意识。正确策略是承认代价，解释场景。

**证据**: `docker-compose.yml`；各 Agent `main.py` 行数统计；`KNOWLEDGE_BACKEND` 环境变量说明。

### Q7: 某个 Agent 挂了怎么办？

**老师为什么问**: 考察你对故障隔离和降级的理解——这是微服务的重要设计考量。

**推荐回答**:

> 分三种情况——
>
> 如果是 **requirements-agent、matching-agent 或 evaluation-agent** 挂了：Gateway 通过 `try/except httpx.HTTPError` 捕获异常，返回 HTTP 502。这三个是核心链路的必须节点，缺一不可。
>
> 如果是 **refactoring-agent** 挂了：Gateway 捕获异常后，`refactoring_advice` 设为空对象 `{}`，主推荐结果正常返回。前端检测到空对象后自动隐藏重构建议区块——用户感知不到异常。
>
> 如果是 **knowledge-base** 挂了：这是最严重的情况——因为 matching-agent 和 evaluation-agent 都依赖它获取风格数据和写入 ADR。当前实现会向上传播异常，最终返回 502。
>
> 改进方向是给 knowledge-base 加只读缓存——即使存储层故障，仍可从缓存返回风格数据。

**加分点**: 区分了必须节点（核心链路）和非必须节点（重构建议）的不同处理策略。

**避坑提醒**: 不要过度承诺"任何服务挂了系统都能跑"——三个核心 Agent 挂了确实不行。诚实说明当前局限和改进方向。

**证据**: `services/api_gateway/app/main.py` 中的 `try/except` 逻辑；`langchain_workflow.py` 中的节点异常处理。

## 三、智能体协作（4 题）

---

### Q8: 为什么使用 Multi-Agent？

**老师为什么问**: 验证 Agent 拆分有合理的架构驱动因素，不是跟风 AI Agent 概念。

**推荐回答**:

> Multi-Agent 在这个系统里不是一个 AI 概念，而是一个**软件工程决策**。
>
> 三个驱动因素——
>
> 第一，**异构技术栈**。requirements-agent 主要是 CPU 密集的字符串匹配，evaluation-agent 主要是网络 I/O 密集的 LLM 调用，matching-agent 需要 Neo4j Python driver。把三种完全不同的计算特征放在同一个进程里，资源调度会互相干扰。
>
> 第二，**独立故障域**。LLM 调用是系统中最不稳定的环节。把 LLM 调用隔离在 evaluation-agent 里，它的超时或异常不会影响 requirements-agent 的关键词匹配和 matching-agent 的规则评分。
>
> 第三，**职责单一可测试**。每个 Agent 只做一件事，对应一个测试文件——`test_requirements.py` 测特征提取，`test_matching.py` 测评分匹配，`test_evaluation.py` 测评估决策。如果是单体，测试边界会模糊。

**加分点**: 从"异构计算特征"角度论证，而不只是说"解耦"这个泛词。

**避坑提醒**: 不要把 Multi-Agent 说成"每个 Agent 都有自主决策能力"——我们的 Agent 是 Pipeline 模式下按固定顺序执行的，不是自主协商的。

**证据**: 各 `services/*/app/main.py`；`tests/unit/` 下 6 个测试文件；`docs/02-架构设计文档.md` 第 5 节。

### Q9: 为什么使用 LangGraph？

**老师为什么问**: LangGraph 是课程技术建议之一，老师想听你对其价值的理解，而非"因为建议用所以用"。

**推荐回答**:

> LangGraph 带来了三个工程价值——
>
> 第一，**显式状态管理**。`ArchitectureWorkflowState` TypedDict 定义了工作流中传递的全部字段——requirement 进来，经过 extract → match → evaluate → trace，每一步的输出写入 state，下游节点直接读取。手动编排中这些字段是散落在函数参数里的隐式约定。
>
> 第二，**标准化追踪**。每个节点自动记录耗时和状态到 `trace` 数组，前端工作流追踪区块直接渲染——不需要在手动编排中手工打点。
>
> 第三，**可扩展性**。如果未来需要加条件分支（比如"检测到重构信号 → 触发 refactor 节点"），在 StateGraph 中加一条 `add_conditional_edges` 即可——手动编排需要改一大段 if-else。
>
> 但 LangGraph 在系统里是**可选引擎**——`build_workflow()` 在导入失败时返回 None，Gateway 自动切换到手动编排。两套编排功能完全等价。这个设计确保了"框架不可用不影响核心功能"。

**加分点**: 明确说"LangGraph 是可选引擎"；说明条件分支的扩展场景（展示前瞻思考）。

**避坑提醒**: 不要只说"LangGraph 好"——要说明它比手动编排好在哪里（状态管理、追踪、可扩展），同时承认可以不用它。

**证据**: `services/api_gateway/app/langchain_workflow.py:build_workflow()`；`workflow_state.py` (36行)。

### Q10: 你的 Agent 是完全自主 Agent 吗？

**老师为什么问**: 区分"完全自主 Agent"和"Workflow-based Agent"是课程可能涉及的概念辨析。

**推荐回答**:

> 不是完全自主 Agent，是 **Workflow-based Multi-Agent**。
>
> 区别在于谁决定"下一步做什么"——
>
> 完全自主 Agent 中，每个 Agent 自己决定下一步调用哪个 Agent、传什么参数，这是通过 LLM 推理来实现的。好处是灵活，代价是增加了 LLM 的不确定性——Agent 可能在不需要的时候调用其他 Agent，甚至陷入循环。
>
> 我们的 Workflow-based 模式中，编排逻辑是**确定性的**——一定是先 extract、再 match、再 evaluate，不存在动态路由。理由是：架构推荐这个场景的流程是固定的，不需要动态决策。
>
> 这种选择是有意为之——用确定性编排换来可靠性。如果场景是开放式的"用户问什么就做什么"，那完全自主 Agent 更合适；但对架构推荐这种 Pipeline 场景，Workflow-based 就够了。

**加分点**: 展示你理解两种模式的适用场景差异，而非盲目选择一种。

**避坑提醒**: 不要说"完全自主 Agent 不好"——说"场景不匹配"。

**证据**: `langchain_workflow.py` 中 4 个节点的固定顺序：`START → extract → match → evaluate → trace → END`。

---

### Q11: 为什么保留 manual fallback？

**老师为什么问**: 考察你是否理解"框架依赖风险"和"降级设计"。

**推荐回答**:

> 三个原因——
>
> 第一，**降低环境依赖**。LangGraph 依赖 `langgraph` 和 `langchain-core` 两个包。如果评委的电脑上没装这两个包（或者版本不兼容），没有 manual fallback 系统就无法运行。保留手动编排后，系统在零额外依赖下即可运行全部功能。
>
> 第二，**运行时容错**。即使 LangGraph 安装了，运行时也可能出异常——比如 StateGraph 编译失败。manual fallback 作为 try/except 的兜底，确保单次请求不受框架异常影响。
>
> 第三，**可对比调试**。两套编排功能等价，通过对比两者的 `workflow_trace` 耗时，可以验证 LangGraph 没有引入额外开销。这在答辩中也是一个好的对比展示点。

**加分点**: 提到"可对比调试"这个实用价值；把降级设计上升到"不因框架不可用而丧失核心功能"的原则高度。

**避坑提醒**: 不要说"manual fallback 是临时方案"——它是永久设计，不是 TODO。

**证据**: `api_gateway/app/main.py` 中 `_manual_orchestrate()` 函数；`langchain_workflow.py:build_workflow()` 返回 None 的逻辑。

## 四、LLM 集成（5 题）

---

### Q12: LLM 在系统里承担什么职责？

**老师为什么问**: 考察 LLM 的职责边界是否清晰——这是 AI 系统工程化的核心问题。

**推荐回答**:

> LLM 只承担**三项辅助职责**，不参与核心决策——
>
> 第一，**语义补全**（requirements-agent）。当规则引擎关键词命中太少（≤2 维）时，LLM 帮助推断隐含特征。temperature=0.1，超时 15s。失败则维持规则结果。
>
> 第二，**风格投票**（evaluation-agent）。从规则引擎产出的候选列表中选一个最佳风格。temperature=0.0（贪婪解码），超时 20s。投票结果必须精确匹配候选列表中已有的风格名，否则丢弃。
>
> 第三，**摘要生成**（evaluation-agent）。用自然语言解释推荐理由、分析优缺点。temperature=0.3，超时 25s。失败则使用规则模板。
>
> LLM 明确**不参与**的环节：候选集生成（由规则引擎确定）、风格评分（由评分公式计算）、知识库数据（来自 JSON 文件或 Neo4j）。

**加分点**: 明确说 LLM 不参与什么——这比说 LLM 参与什么更能体现职责边界意识。

**避坑提醒**: 不要说"LLM 辅助决策"这种模糊表述——要说清楚辅助在哪个环节、怎么辅助、失败怎么办。

**证据**: `requirements_agent/app/main.py:llm_analyze()`；`evaluation_agent/app/main.py:llm_vote_style()`, `llm_summary()`。

### Q13: 为什么不让 LLM 直接推荐架构？

**老师为什么问**: 这是对 LLM 幻觉风险的核心追问。

**推荐回答**:

> 如果不加限制让 LLM 直接推荐，会出现两个致命问题——
>
> 第一，**幻觉**。LLM 可能推荐知识库中不存在的架构风格，比如编造一个"Micro-Event-Driven Architecture"。后续的对比矩阵、风险建议、拓扑图都无从生成——因为这些数据来自知识库，不在知识库里的风格没有任何元数据。
>
> 第二，**不可复现**。相同的需求描述，LLM 在不同时间可能给出不同的推荐——这是概率模型的固有特性。对于架构评审场景，决策的稳定性比"偶尔惊艳"更重要。
>
> 我们的方案是**"规则定候选，LLM 做选择"**——候选集由规则引擎从知识库 10 种风格中确定性产生，LLM 只能在这个闭集内投票。这既保留了 LLM 的语义理解能力，又杜绝了幻觉干扰核心推荐。

**加分点**: 用"闭集"和"开集"的对比说明设计意图；提到"不可复现"这个评审场景的核心约束。

**避坑提醒**: 不要说"LLM 不可靠所以我们不用它"——我们用了，但是加了约束。正确表述是"限制使用范围"。

**证据**: `matching_agent/app/main.py:score_style()` 确定候选集；`evaluation_agent/app/main.py` 投票校验逻辑。

### Q14: 如何防止 LLM 幻觉？

**老师为什么问**: LLM 幻觉是 AI 系统工程化的核心挑战，老师想看你有没有实际的防御措施。

**推荐回答**:

> 四道防线，从粗到细——
>
> **防线 1: LLM 不参与候选集生成。** 候选 Top 3 完全由规则引擎 `score_style()` 确定，LLM 无权推荐规则未产出的风格。这是最根本的防线——从架构上杜绝了 LLM 推荐不存在风格的可能。
>
> **防线 2: 闭集投票 + 强制校验。** Prompt 明确要求"仅从以下候选列表中选择"。LLM 返回的风格名要经过 `if returned_name in [c["style"] for c in candidates]` 校验——不匹配就直接丢弃，不加分。这行校验在 `evaluation_agent/app/main.py` 中。
>
> **防线 3: 低温度参数。** 投票使用 temperature=0.0，即贪婪解码——模型每次都选择概率最高的 token，最大化确定性。摘要使用 temperature=0.3，在确定性和语言自然度之间取平衡。
>
> **防线 4: Few-shot 格式化约束。** 9 个标注示例严格约束了 LLM 的输出格式。所有示例输出都是严格 JSON，训练 LLM 输出结构化数据而非自由文本。LLM 返回非 JSON 时尝试从 markdown 代码块提取，提取失败则丢弃。

**加分点**: 四道防线从架构层 → 校验层 → 参数层 → Prompt 层，展示分层防御思维。

**避坑提醒**: 不要说"完全防止幻觉"——只能说"将幻觉影响限制在摘要质量层面"。

**证据**: `evaluation_agent/app/main.py:llm_vote_style()` 投票校验；`requirements_few_shot.py` (6 示例)；`evaluation_few_shot.py` (3 示例)。

### Q15: LLM 不可用怎么办？

**老师为什么问**: 降级设计是 AI 系统可靠性的核心考查点。

**推荐回答**:

> 系统实现了三级降级——
>
> **完全不可用（API Key 未配置）**: 系统启动时检测到 `LLM_API_KEY` 为空，所有 LLM 调用点自动跳过。特征提取用纯规则结果，投票返回 null，摘要用 `_fallback_summary()` 生成模板化文本。核心推荐链路——候选 Top 3、对比矩阵、风险建议——完全不受影响。回归测试 20/20 验证了纯规则模式的完整性。
>
> **部分不可用（超时或异常）**: 每处 LLM 调用有独立超时（15s/20s/25s）和 try/except。单次调用失败只影响那个环节——比如投票超时，摘要仍正常生成。
>
> **输出异常（非 JSON 或无效名）**: 投票名不在候选列表 → 丢弃。返回非 JSON → 尝试容错解析 → 失败则丢弃。

**加分点**: 区分了三种不可用场景（未配置/超时/异常输出），而非笼统说"降级"。

**避坑提醒**: 不要说 LLM 降级后"体验一样好"——LLM 摘要是自然语言，规则模板是格式化文本，后者读起来更机械。这是诚实的代价说明。

**证据**: `evaluation_agent/app/main.py:_fallback_summary()`；`requirements_agent/app/main.py` LLM 调用条件判断；回归测试 100% 通过率的报告。

### Q16: Few-shot Prompt 起什么作用？

**老师为什么问**: Few-shot Prompt 是课程技术建议之一，老师想看你是否理解其工程价值。

**推荐回答**:

> Few-shot Prompt 解决的是 LLM 的**输出格式不稳定**问题。
>
> 我们设计了 9 个标注示例——6 个给需求特征提取，3 个给评估摘要。每个示例都是完整的"输入 → 期望输出"对。
>
> 具体作用——
>
> 第一，**格式约束**。需求提取的 6 个示例输出都是严格 JSON `{feature: true/false}`，LLM 通过示例学习到这个格式约定。
>
> 第二，**语义教学**。示例 1（"日活百万，双十一暴增"）教 LLM 在没有"高并发"关键词时也要推断出 `high_concurrency: true`；示例 2（"不需要实时，批量处理即可"）教 LLM 理解中文否定语义。
>
> 第三，**降级兼容**。Few-shot 模块通过 `try/except ImportError` 导入——如果文件被误删或模块不可用，自动降级为零样本 Prompt（内联在 Agent 代码中），功能不中断。

**加分点**: 具体说明示例 1 和示例 2 的教学目的，展示你知道每个示例为什么这么写。

**避坑提醒**: 不要说"Few-shot 让 LLM 更聪明"——说"约束输出格式和语义判断标准"。

**证据**: `services/common/prompts/requirements_few_shot.py` (87行, 6 示例)；`services/common/prompts/evaluation_few_shot.py` (119行, 3 示例)。

### Q17: 为什么用 Neo4j？

**老师为什么问**: 数据库选型是架构设计的核心决策。

**推荐回答**:

> 选 Neo4j 是因为架构知识天然是**图结构**的——架构风格之间有关系（COMPLEMENTS 互补），风格和质量属性之间有关系（HAS_QUALITY 具备），风格和风险之间有关系（HAS_RISK 存在风险）。
>
> 如果用关系数据库存这些数据，查询"哪些风格具备高并发属性且与 Event-Driven 互补"需要多次 JOIN。在 Neo4j 中，这只是一条 Cypher 查询的图遍历。
>
> 但我们做了一个关键的务实决策——Neo4j 是**可选后端**。系统同时支持 JSON 文件后端，通过 `KNOWLEDGE_BACKEND` 环境变量切换。Neo4j 不可用时自动回退 JSON。这意味着评委不需要安装 Docker 和 Neo4j 也能体验全部功能。

**加分点**: 承认 Neo4j 是可选后端，不是必需品——展示务实判断力。

**避坑提醒**: 不要说"因为课程建议用 Neo4j 所以用了"——要说"架构知识的图结构特征天然适合图数据库"。

**证据**: `services/knowledge_base/app/graph_repository.py` (388行)；`services/knowledge_base/init/init_neo4j.py` (175行)；`KNOWLEDGE_BACKEND` 环境变量文档。

### Q18: Neo4j 相比 JSON 的增量价值是什么？

**老师为什么问**: 如果 JSON 也能跑全部功能，那 Neo4j 的价值在哪？老师想听图遍历的具体例子。

**推荐回答**:

> JSON 模式只能做**属性匹配**——"这个风格的 tags 里有 high_concurrency，所以 +2 分"。这是 1 跳查询。
>
> Neo4j 模式能做**关系驱动的多跳推理**——
>
> 具体例子：用户需求是"日志分析平台"。规则引擎识别出 `data_intensive` 和 `real_time` 两个特征。JSON 模式分别匹配 Pipeline-Filter（适合数据密集型）和 Event-Driven（适合实时性），各自独立评分。
>
> Neo4j 模式多走一步：它通过 `COMPLEMENTS` 关系发现了 Pipeline-Filter 和 Event-Driven 之间存在互补关系——Pipeline-Filter 负责数据处理流水线，Event-Driven 负责实时事件分发。这个关联在 JSON 的 tags 数组里是不显式存在的。
>
> 图谱加分上限为规则得分的 50%——确保图谱是"增强"而非"替代"。

**加分点**: 用具体场景说明多跳推理；提到 50% 上限的设计意图。

**避坑提醒**: 不要暗示 JSON 模式"不够好"——JSON 模式是全功能模式，Neo4j 是增强。

**证据**: `graph_matcher.py:blend_scores()` 的 50% 上限逻辑；`graph_repository.py` 中的 COMPLEMENTS 查询。

---

### Q19: 为什么还保留 JSON fallback？

**老师为什么问**: 考察降级设计的完整性——是否真正做到了"无 Neo4j 也能跑"。

**推荐回答**:

> 三个理由——
>
> 第一，**降低使用门槛**。不是所有人的电脑上都装了 Docker，也不是所有 Docker 环境都有 4GB 以上内存跑 Neo4j。JSON 模式零外部依赖——Python 标准库的 `json` 模块就能跑。
>
> 第二，**故障容错**。Neo4j 容器可能因为内存不足启动失败，也可能在演示过程中意外挂掉。`auto` 模式下系统自动探测——Neo4j 可用就用，不可用就回退 JSON，用户无感知。
>
> 第三，**开发效率**。本地开发时不需要每次调试都启动 Neo4j。JSON 文件改完直接生效，反馈循环极短。
>
> JSON fallback 不是"临时方案"——它是系统的**正式后端**，与 Neo4j 后端地位平等。双后端通过统一的 `_repo()` 调度函数切换，调用方不感知。

**加分点**: 说明 JSON 不是临时方案而是正式后端；提到 `auto` 模式的自动探测。

**避坑提醒**: 不要说"Neo4j 太重了所以用 JSON 保底"——正确表述是"两种后端各有适用场景"。

**证据**: `knowledge_base/app/main.py` 中的 `_repo()` 调度函数；`KNOWLEDGE_BACKEND` 的 json/neo4j/auto 三种模式。

### Q20: 规则引擎如何保证推荐可靠？

**老师为什么问**: 规则引擎是"规则保证下限"的关键，老师想听具体的评分机制。

**推荐回答**:

> 规则引擎通过四层机制保证可靠性——
>
> 第一，**确定性评分**。`score_style()` 是纯函数——相同输入永远得到相同分数。评分公式是 `标签匹配 +2 + 学习权重 +1 + 特定规则 +1`，没有随机数，没有概率。
>
> 第二，**知识库闭环**。所有候选风格的 pros/cons/best_for/topology_mermaid 都来自 `architecture_styles.json`，不是 LLM 编造的。推荐结论不可能出现知识库之外的内容。
>
> 第三，**主流保底**。Layered / Microservices / Event-Driven 三种业界公认的主流架构**始终**出现在 Top 3 中——无论特征匹配结果如何。这确保推荐不偏离业界共识。
>
> 第四，**可审计**。每个评分组成可以逐项审计——"Event-Driven 为什么得 9 分？因为高并发 +2、实时 +2、可靠 +2、扩展 +2、特定规则 +1"。没有不可解释的环节。

**加分点**: 强调"确定性"和"可审计"——这是 AI 辅助决策系统区别于纯 LLM 方案的本质特征。

**避坑提醒**: 不要说"规则引擎绝对正确"——规则权重是经验值，不是数学真理。正确说法是"规则引擎保证推荐结果在知识库约束范围内有据可查"。

**证据**: `matching_agent/app/main.py:score_style()` 函数；`architecture_styles.json`。

### Q21: 规则是不是太简单了？

**老师为什么问**: 老师可能觉得 +2/+1 的评分太简单，想看你是否意识到这个局限以及改进思路。

**推荐回答**:

> 当前规则确实简单——这是有意识的权衡，不是能力不足。
>
> 简单的价值在于**可解释性**。评委看到 +2 来自"标签匹配 high_concurrency"，+1 来自"Event-Driven 特定规则加成"，可以逐项理解和质疑。如果用复杂的加权公式或机器学习模型，评分就成了新的黑盒。
>
> 但简单不等于没有改进空间。三个明确的改进方向——
>
> 第一，**权重数据驱动**。当前 +2/+1 是经验值，可以通过网格搜索或贝叶斯优化在测试数据集上调优。
>
> 第二，**特征权重差异化**。当前所有标签匹配都是 +2。实际上"强一致性"可能比"可扩展性"对架构选择的影响更大——应该有不同的权重。
>
> 第三，**负向规则**。当前规则只加分不减分。可以加入"冲突惩罚"——比如"Serverless 遇上强一致性"应该减分，因为 Serverless 不太适合强一致性场景。
>
> 这三点我们在架构设计文档的"当前不足"部分已经记录了。

**加分点**: 承认简单，但论证简单的价值（可解释性）；提出三个具体改进方向（不是空谈）。

**避坑提醒**: 绝对不要辩解"简单就够了"。正确策略是"当前选择有理由，未来改进有方向"。

**证据**: `matching_agent/app/main.py:score_style()` 评分逻辑；`docs/02-架构设计文档.md` 第 23 节"架构取舍与不足"。

### Q22: LLM 缓存如何设计？

**老师为什么问**: 缓存设计体现了对 LLM 调用成本和响应延迟的工程考量。

**推荐回答**:

> 缓存设计有三个要点——
>
> 第一，**缓存粒度是请求级**。以整个推荐结果作为缓存单元，缓存键是 `SHA256(requirement + model + knowledge_version)`。这意味着知识库内容一旦变化（MD5 变了），所有旧缓存自动失效——避免推荐过时的架构信息。
>
> 第二，**双后端设计**。memory 模式使用内存字典 + TTL（默认 3600s），线程安全，进程重启丢失。sqlite 模式使用 SQLite 持久化存储，跨重启持久化，定期清理过期条目。通过 `CACHE_BACKEND` 环境变量切换。
>
> 第三，**前端可视化**。响应中的 `cache_hit` 字段驱动前端展示——绿色徽章"缓存命中"或蓝色徽章"实时计算"。还有 `/cache/stats` API 提供命中率统计。
>
> 当前缓存是**轻量实现**——用的是进程内字典和 SQLite，不是 Redis。对于课程项目的单机演示场景完全够用。生产环境可以替换为 Redis，接口设计已预留扩展性。

**加分点**: 提到 `knowledge_version` 自动失效机制——确保缓存一致性；承认轻量实现但说明可替换。

**避坑提醒**: 不要说"我们实现了分布式缓存"——只实现了内存/SQLite。

**证据**: `services/common/cache/simple_cache.py` (96行)；`services/common/cache/sqlite_cache.py` (155行)；`services/common/cache/hash_utils.py` (51行)。

### Q23: ADR 决策溯源有什么用？

**老师为什么问**: ADR 是课程技术建议之一，老师想听它在系统中的实际价值。

**推荐回答**:

> ADR 在这个系统中有三个角色——

**加分点**: 提到"非阻塞设计"——ADR 是重要但不阻塞的功能。

**避坑提醒**: 不要说 ADR 符合什么工业标准——就是课程项目中的决策记录机制。

### Q24: 组合架构推荐如何避免乱组合？

**老师为什么问**: 如果不加约束，可能会推荐不合理的架构组合（比如 Serverless + CQRS）。

**推荐回答**:

> 避免乱组合靠三重约束——
>
> 第一，**预定义组合库**。我们只推荐 `architecture_combinations.json` 中预定义的 5 种组合，不允许自由组合。每种组合都经过了人工审核——确保两个风格的组合在工程上有合理性。
>
> 第二，**复杂度惩罚**。每个组合有 `complexity_penalty` 字段（1-3 级），从组合分数中扣除。比如 CQRS + Event Sourcing 的复杂度是 3 级，会扣 3 分——这个组合功能强但工程复杂度高，不适合所有场景。
>
> 第三，**特征覆盖互补验证**。评分时检查组合是否覆盖了用户需求中规则引擎未覆盖的特征维度，每个额外覆盖 +1 分。只推荐"真正有互补价值"的组合，而非"分数高就推"。

**加分点**: 提到"人工审核的预定义组合"而非自由组合——展示对组合推荐风险的认识。

**避坑提醒**: 不要暗示组合推荐是 AI 自动发现的——是人工预定义 + 评分排序。

### Q25: 架构重构建议是否可靠？

**老师为什么问**: 重构建议涉及对现有系统的修改，可靠性非常重要。

**推荐回答**:

**加分点**: 强调"宁可漏报不可误报"的原则；承认覆盖度有限。

**避坑提醒**: 不要说"重构建议可以替代架构师判断"——它是辅助建议，不是权威诊断。

## 八、3 个最可能被老师追问的问题

---

### 追问 1: "你刚才说规则保证下限，那你倒是说说规则引擎在什么情况下会给出错误推荐？"

**推荐回答**:

> 两种情况——
>
> 第一，**特征提取不准确**。关键词匹配是粗粒度的——用户说"这个系统对一致性要求很高"但没说"强一致性"或"事务"，规则引擎可能漏掉 `strict_consistency` 特征，从而影响评分。这种情况下 LLM 语义补全可以补救——但如果 LLM 也没配置，就确实会漏。
>
> 第二，**非功能特征之间存在冲突**时，规则引擎只是把分数加起来，不做冲突消解。比如用户同时要求"强一致性"和"高并发"——规则引擎会给同时支持这两个特征的风格分别加分，但不会告诉你这两个目标在工程上存在 trade-off。
>
> 但"错误推荐"有一个保护机制——三种主流架构始终在 Top 3 中。即使评分完全跑偏，评委也能看到 Layered / Microservices / Event-Driven 作为对照基线。

---

### 追问 2: "你的系统学习权重是怎么更新的？这个算法有什么问题？"

**推荐回答**:

> 当前的学习权重更新很简单——
>
> 用户提交确认反馈 → `_update_learned_weights()` 从需求文本中用词典提取特征 → 每个特征到推荐风格的关联计数 +1 → 持久化到 `learned_weights.json`。
>
> 三个明确的问题——
>
> 第一，**无衰减**。所有反馈的权重永远累积，不区分新旧。旧的反馈可能已不适用于当前知识库。
> 第二，**无归一化**。不同特征的计数不可比——"高并发"可能被确认了 10 次，"安全性"只被确认了 1 次，不代表安全性不重要，只是场景分布不均。
> 第三，**无协同过滤**。当前算法只看特征到风格的一阶关联，没有利用"相似用户有相似偏好"的信息。
>
> 改进方向是引入时间衰减因子和特征级归一化。

---

### 追问 3: "如果让你把这个系统部署到生产环境，你最优先改什么？"

**推荐回答**:

> 按优先级排序——
>
> 第一优先：**缓存升级**。从内存/SQLite 升级到 Redis，支持分布式缓存和更细粒度的 LLM 调用级缓存。
>
> 第二优先：**异步解耦**。在 Gateway 和 Agent 之间加消息队列（如 RabbitMQ），避免同步 HTTP 调用的阻塞和故障传播。
>
> 第三优先：**认证授权**。加上 OAuth2/JWT 用户认证，反馈接口需要区分来源。
>
> 第四优先：**规则权重优化**。用 A/B 测试或离线评估来调优 +2/+1 的权重值。
>
> 这四点的改动成本从低到高排列——缓存升级只需改配置和替换存储层，异步解耦则需要改通信架构。

## 九、3 个最容易扣分的问题

---

### 扣分 1: "你用了这么多技术，核心创新到底是什么？"

**危险回答**: "我们整合了 LLM、Neo4j、LangGraph、Few-shot、ADR 等先进技术……"（这是在罗列技术，不是在讲创新）

**安全回答**:

> 核心创新不是用了哪些技术，而是**三层推理的架构设计**——
>
> 规则引擎提供确定性基线、Neo4j 提供图关系推理、LLM 提供语义理解——这三层不是简单叠加，而是通过 `blend_scores()` 的显式融合公式（规则分 + min(图谱分, 规则分//2) + LLM 投票加分）协同工作。
>
> 这个设计解决了"AI 辅助架构推荐"的核心矛盾——既要 LLM 的语义理解能力，又要防止 LLM 的幻觉污染推荐结果。解法是让 LLM 只在最末端的评估环节参与，且只能在规则确定的候选集内投票。
>
> 如果用一句话概括创新点：**在一个 AI 系统里，让确定性规则保护概率性模型的边界。**

---

### 扣分 2: "你的项目和 GitHub 上随便一个 LLM 架构推荐项目有什么区别？"

**危险回答**: "我们的更好，因为……"（贬低别人抬高自己容易引发反感）

**安全回答**:

> 三个关键区别——
>
> 第一，大多数 LLM 架构推荐项目是"输入需求 → LLM 输出推荐"的两步黑盒。我们的系统是"规则 → 图谱 → LLM"的三步白盒，每步输出可独立审计。
>
> 第二，大多数项目没有降级设计——LLM 不可用整个系统瘫痪。我们设计了三级降级，纯规则模式回归测试 20/20 通过。
>
> 第三，大多数项目把"推荐"作为终点。我们把推荐当作起点——后面还有 ADR 决策记录、反馈学习权重、重构建议、组合推荐，形成了一个从"推荐"到"验证"到"进化"的闭环。
>
> 当然，我们的系统也有局限——工程规模是课程项目级别的，没有做生产级的性能优化和分布式部署。

---

### 扣分 3: "如果你有更多时间，你会怎么改进？"

**危险回答**: "加更多功能，接入更多 LLM，支持英文，加上 K8s……"（泛泛堆功能，没有优先级判断）

**安全回答**:

> 我会把时间花在**当前系统中最薄弱的环节**——不是加新功能，而是加固现有功能的正确性。
>
> 具体来说——
>
> 第一，**做一次严格的离线评估**。用 20 条测试数据的 `expected_style` 标注，计算纯规则模式下 Top-1 准确率。当前回归测试只验证"有推荐"和"推荐在候选集中"，没有验证"推荐的确实是最合适的那个"。有了准确率基线，才能数据驱动地调优规则权重。
>
> 第二，**补全风险建议覆盖**。当前 3 种主流架构有针对性风险条目，其余 7 种用通用模板。给所有 10 种风格补齐针对性建议。

> 第三，**加一层"冲突检测"规则**。当前评分只加分不减分。加入负向规则——比如"Serverless 遇上强一致性 → -2"——让推荐更精准。
>
> 这三个改进全部不涉及新架构、新服务、新依赖——是在当前架构框架内提升正确性和覆盖率。

## 十、3 个主动承认不足但不失分的回答

---

### 承认不足 1: LLM 缓存是轻量实现

> 我们的 LLM 缓存用的是进程内 dictionary 和 SQLite，不是 Redis 这样的分布式缓存。这个选择是有意为之——课程项目是单机场景，引入 Redis 多一个外部依赖。但如果部署到生产环境，Redis 是更合适的方案——支持分布式、支持更细粒度的 TTL 控制、支持更大的缓存容量。当前的内存/SQLite 实现接口设计已经和缓存逻辑解耦，替换为 Redis 只需实现相同的 get/set/stats 接口，不需要改调用方代码。

### 承认不足 2: 规则权重是硬编码

> +2/+1 的权重值是经验设定，不是通过数据驱动方法（如网格搜索或贝叶斯优化）得出的最优值。选择硬编码的理由是"可解释"——评委看到 +2 来自"标签匹配"，+1 来自"特定规则"，可以逐项理解。机器学习优化后的权重可能更准确，但可解释性会下降。这是一个"可解释 vs 准确"的经典权衡。改进方向是用离线评估数据调优权重，同时保留当前权重的文档作为"可解释基线"。

### 承认不足 3: 前端无自动化测试

> 前端 11 个展示区块的验证目前是手动完成的——浏览器打开、逐个检查渲染是否正常。没有引入 Selenium 或 Playwright 做自动化 E2E 测试。原因有两个——一是 Mermaid.js 渲染是异步的，自动化等待逻辑比较复杂；二是前端是原生 HTML（289 行），逻辑相对简单，手动验证的边际收益在这个规模下还算合理。但在生产环境中，至少有对核心路径（输入 → 点击 → 渲染）的自动化 E2E 测试。

## 十一、1 分钟最终总结话术

> 最后用一分钟总结我们这个项目的核心设计哲学——
>
> 我们构建的不是一个"问 LLM 推荐什么架构"的聊天机器人，而是一个 **Compound AI System**。
>
> 它的核心特征是三句话：**规则保证下限，图谱增强关系，LLM 提升上限。**
>
> 规则引擎提供确定性基线——在任何情况下，10 种预定义风格、6 条评分规则、3 种主流保底，推荐始终在知识库约束范围内。
>
> Neo4j 知识图谱提供关系推理增强——通过图遍历发现隐含的架构关联，但加分上限 50%，确保图谱是增强而非替代。
>
> LLM 提供语义理解和自然语言解释——但只参与末端评估，候选集由规则确定，投票结果强制校验。四道防线将 LLM 的不确定性限制在"摘要质量"层面。
>
> 这三层是**独立可降级**的——LLM 不可用、Neo4j 不可用、LangGraph 不可用时，核心推荐链路不中断。回归测试 20 条用例在纯规则模式下 100% 通过。
>
