# 演示输入用例

> 版本: 1.0
> 日期: 2026-05-13
> 用途: 答辩 5 分钟演示的输入材料 + 备选展示
> 原则: 预期结果基于规则引擎评分公式的确定性分析；LLM 投票可能导致 tie-break 微调

---

## 评分公式速查

在阅读每个用例的预期结果之前，先回顾系统的评分规则：

```
总分 = 标签基础分 (tag ∈ features → +2/项)
     + 学习权重分 (≥2次确认的 feature-style 关联 → +1/项)
     + 特定规则加成 (7条硬编码规则 → +1/条)

特定规则:
  Event-Driven + high_concurrency                   → +1
  Microservices + team_size_large                   → +1
  Layered + strict_consistency                      → +1
  Pipeline-Filter + data_intensive + real_time      → +1
  CQRS + data_intensive + high_concurrency          → +1
  Microservices + high_concurrency + strict_consistency → +1

主流保底:
  Layered / Microservices / Event-Driven 始终出现在 Top 3 中
```

> **注意**: 以下计算假设学习权重为初始状态（全部为 0）。如果已累积反馈权重，评分可能略有不同。

## 用例 1: 即时通讯系统 ⭐ 主演示

### 输入

```
开发跨平台即时通讯系统，支持万人同时在线，消息实时可靠，后续扩展视频通话。
```

### 预期触发特征

| 维度 | 命中 | 关键词证据 |
|------|------|-----------|
| high_concurrency | ✅ | 万人 |
| real_time | ✅ | 实时、消息（在上下文中的实时语义） |
| reliability | ✅ | 可靠 |
| scalability | ✅ | 扩展 |

### 预期推荐架构

| 排名 | 风格 | 评分演算 |
|------|------|---------|
| **1** | **Event-Driven Architecture** | high_concurrency(+2) + real_time(+2) + scalability(+2) + reliability(+2) + 特定规则(+1) = **9** |
| 2 | Microservices | high_concurrency(+2) + scalability(+2) + reliability(+2) = **6** |
| 3 | Layered Architecture | scalability(+2) + reliability(+2) = **4**（主流保底确保出现） |

> 注: Space-Based Architecture 理论上得分 6（high_concurrency+2, scalability+2, real_time+2），与 Microservices 并列。主流保底策略确保 Microservices 和 Layered 进入 Top 3。

### 预期组合推荐

**Microservices + Event-Driven Architecture**（复杂度 1）

微服务负责服务边界和独立部署，事件驱动解耦服务间通信——特别适合高并发、实时性要求高的 IM 场景。

### 是否触发重构建议

**否**。输入描述的是新系统需求，不含"单体"、"遗留"、"拆分"、"耦合"等重构信号。

### 演示价值

这是**最稳健的主演示用例**。特征清晰（4 个维度）、评分差异明显（9-6-4）、组合推荐有说服力、风险建议有针对性条目（Event-Driven 是 3 种有针对性的风格之一）。

### 如果结果偏差，现场怎么解释

| 偏差情况 | 解释 |
|---------|------|
| 首选变成 Microservices | "LLM 投票给 Microservices 加了 +1 分 tie-break。Event-Driven 和 Microservices 原始规则分差距是 3 分，但如果 LLM 投票了 Microservices，它仍然是强有力的候选。这正好体现了混合推理——LLM 在规则基础上提供第二意见。" |
| Top 3 中包含 Space-Based | "Space-Based Architecture 确实是一种适合高并发实时场景的架构。它没有出现在我们预期中是因为主流保底策略让 Microservices 和 Layered 优先，但出现也不意外——它的 tags 完美匹配了这个场景的特征。" |
| 特征提取少了某维度 | "特征提取依赖关键词词典。如果某维度未被命中，可能是关键词表述不在词典中（如'视频通话'未触发 scalability）。这正是 LLM 语义补全存在的价值——补充关键词遗漏的特征。" |

## 用例 2: 电商平台

### 输入

```
构建电商平台，订单支付库存需要强一致事务，双11高峰要抗压，多团队并行开发。
```

### 预期触发特征

| 维度 | 命中 | 关键词证据 |
|------|------|-----------|
| strict_consistency | ✅ | 强一致、事务 |
| high_concurrency | ✅ | 高峰、抗压 |
| scalability | ✅ | 抗压（推断性命中，取决于词典覆盖） |
| team_size_large | ✅ | 多团队、并行开发 |

### 预期推荐架构

| 排名 | 风格 | 评分演算 |
|------|------|---------|
| **1** | **Microservices** | team_size_large(+2) + high_concurrency(+2) + scalability(+2) + 特定规则(team_size_large+1) + 特定规则(high_concurrency+strict_consistency+1) = **8** |
| 2 | Event-Driven Architecture | high_concurrency(+2) + scalability(+2) + 特定规则(+1) = **5** |
| 3 | Layered Architecture | strict_consistency(+2) + scalability(+2) + 特定规则(strict_consistency+1) = **5** |

> 注: CQRS 也可能得 4-6 分（取决于 strict_consistency + high_concurrency 组合是否命中 CQRS 特定规则）。如果 CQRS 得分高于 5，它可能进入 Top 3。

### 预期组合推荐

**Microservices + Event-Driven Architecture**（复杂度 1）

微服务满足多团队并行开发和独立部署需求，事件驱动满足高并发下的异步解耦需求。双 11 场景特别适合这种组合。

### 是否触发重构建议

**否**。虽然描述了高性能要求，但输入是新系统构建而非遗留系统改造。

### 演示价值

展示**多特征维度同时命中**时的评分演算——特别是两条特定规则同时触发（team_size_large +1 和 high_concurrency+strict_consistency +1）。适合作为"如果我换一个需求，系统会怎样响应"的快速演示。

### 如果结果偏差，现场怎么解释

| 偏差情况 | 解释 |
|---------|------|
| CQRS 进入 Top 3 | "CQRS 的 tags 包含 strict_consistency、high_concurrency 和 data_intensive。电商场景有数据密集特征（订单数据），CQRS 出现在候选列表是合理的——它特别适合读写分离的高并发场景。" |
| scalability 未被命中 | "scalability 的关键词词典包含'扩展'、'弹性'等，但不一定包含'抗压'。如果未被命中，这是关键词匹配的覆盖盲区，LLM 语义补全可以填补。" |

## 用例 3: 银行核心账务系统

### 输入

```
银行核心账务系统，要求高可靠和强一致，审计可追溯。
```

### 预期触发特征

| 维度 | 命中 | 关键词证据 |
|------|------|-----------|
| reliability | ✅ | 高可靠 |
| strict_consistency | ✅ | 强一致 |
| security | ✅ | 审计 |

### 预期推荐架构

| 排名 | 风格 | 评分演算 |
|------|------|---------|
| **1** | **Layered Architecture** | strict_consistency(+2) + reliability(+2) + security(+2) + 特定规则(strict_consistency+1) = **7** |
| 2 | Microservices | reliability(+2) + security(+2) = **4** |
| 3 | Event-Driven Architecture | reliability(+2) = **2**（主流保底确保出现） |

> 注: Client-Server 得分约 4（reliability+2, security+2），与 Microservices 并列。推荐优先级取决于排序时的稳定策略。

### 预期组合推荐

**Layered Architecture + CQRS**（复杂度 1）

分层架构满足账务系统的业务逻辑组织需求，CQRS 满足查询（审计追溯）和写入（交易处理）的分离需求。

### 是否触发重构建议

**否**。

### 演示价值

展示**强一致性 + 安全合规**场景——这是 Layered Architecture 最经典的适用场景。与用例 1（Event-Driven 主导）形成鲜明对比，展示系统对不同特征组合的差异化推荐能力。同时展示了 security 维度的关键词"审计"触发。

### 如果结果偏差，现场怎么解释

| 偏差情况 | 解释 |
|---------|------|
| Microservices 排第一 | "Microservices 的 reliability 和 security 标签各 +2 分，加上可能的 LLM 投票加分，在某些条件下可能超过 Layered。但两者差距很小——说明对于这种特征组合，两种架构都有合理的一面。" |
| security 命中覆盖不足 | "security 维度的关键词词典包含'加密'、'认证'、'鉴权'、'审计'、'合规'、'脱敏'等 15 个词。'审计'是确定性的命中。" |

## 用例 4: 日志分析平台

### 输入

```
日志分析平台，每秒采集百万条日志并实时告警。
```

### 预期触发特征

| 维度 | 命中 | 关键词证据 |
|------|------|-----------|
| data_intensive | ✅ | 日志、采集、百万条 |
| real_time | ✅ | 实时 |
| high_concurrency | ✅ | 每秒、百万 |

### 预期推荐架构

| 排名 | 风格 | 评分演算 |
|------|------|---------|
| **1** | **Event-Driven Architecture** | high_concurrency(+2) + real_time(+2) + data_intensive(+2) + 特定规则(+1) = **7** |
| 2 | Pipeline-Filter | data_intensive(+2) + real_time(+2) + 特定规则(data_intensive+real_time+1) = **5** |
| 3 | Microservices | high_concurrency(+2) = **2**（主流保底确保出现） |

> 注: CQRS 理论上也可能得 5 分（data_intensive+2, high_concurrency+2, 特定规则+1），但被主流保底推至 Top 3 之外。所有主流架构得分较低时，基线对比价值凸显——"你看，Microservices 在这个场景下确实不如 Event-Driven"。

### 预期组合推荐

**Pipeline-Filter + Event-Driven Architecture**（复杂度 2）

管道-过滤器负责日志处理的 ETL 流水线，事件驱动负责实时告警的异步分发。复杂度 2 比微服务+事件驱动的组合略高——这个惩罚反映了流处理+事件驱动同时引入的工程复杂度。

### 是否触发重构建议

**否**。

### 演示价值

这是展示**图谱证据**的最佳用例。如果 Neo4j 已初始化，`POST /graph/match` 会返回 Pipeline-Filter 和 Event-Driven 之间的 COMPLEMENTS 关系（如果图谱中定义了这一关系）。同时展示复杂度惩罚机制——复杂度 2 的组合比复杂度 1 的组合多扣 1 分。

### 如果结果偏差，现场怎么解释

| 偏差情况 | 解释 |
|---------|------|
| 图谱证据区显示"无图谱证据" | "Neo4j 未启动或未初始化，系统自动回退 JSON 后端。图谱虽是最佳展示项，但不是推荐结果的必需品——规则评分已经正确识别了 Pipeline-Filter 和 Event-Driven。" |
| Pipeline-Filter 进入 Top 3 替代了 Microservices | "Space-Based Architecture 和 CQRS 在这个场景下得分也可能较高。Top 3 的具体构成取决于各风格的标签覆盖。无论如何，Event-Driven 作为首选是稳定的。" |

## 用例 5: 旧单体系统重构 ⭐ 重构演示

### 输入

```
已有单体电商系统，订单库存耦合严重，性能瓶颈明显，希望拆分为微服务。
```

### 预期触发特征

| 维度 | 命中 | 关键词证据 |
|------|------|-----------|
| scalability | ✅ | 拆分 |
| team_size_large | ✅ | 拆分（暗示多团队并行） |

> 注: 这个输入的特征命中较少（2 维）。如果 LLM 已配置且触发语义补全（命中 ≤2），LLM 可能补充 high_concurrency 和 reliability 等特征。

### 预期推荐架构

| 排名 | 风格 | 评分演算（无 LLM 补全） |
|------|------|---------|
| **1** | **Microservices** | scalability(+2) + team_size_large(+2) + 特定规则(team_size_large+1) = **5** |
| 2 | Event-Driven Architecture | scalability(+2) = **2**（主流保底） |
| 3 | Layered Architecture | scalability(+2) = **2**（主流保底） |

### 预期重构建议

| 字段 | 预期值 |
|------|--------|
| refactoring_needed | ✅ true |
| detected_architecture_smells | monolith_coupling（单体耦合）、scaling_bottleneck（扩展瓶颈） |
| suggested_patterns | Strangler Fig（绞杀者模式） |
| migration_steps | ≥ 3 个阶段的渐进式迁移步骤 |
| risks | 含风险提示 |
| mitigation_suggestions | 含缓解措施 |

### 是否触发重构建议

**是**——这是最佳的重构演示用例。输入中的"单体"、"耦合"、"拆分"三个关键词强烈触发坏味检测。

### 演示价值

展示**重构建议的完整链路**：坏味检测 → 模式选择 → 迁移步骤 → 风险提示 → 缓解措施。前端"架构重构建议"区块会显示红色"需重构"徽章、坏味列表、建议模式（绞杀者模式）、可展开的迁移步骤。

### 如果结果偏差，现场怎么解释

| 偏差情况 | 解释 |
|---------|------|
| 重构建议未触发 | "refactoring-agent 的坏味检测基于关键词匹配。如果'单体'、'耦合'等词未被识别到，系统可能判定 refactoring_needed=false。这说明当前检测规则的覆盖度有限——重构信号检测是我们明确记录的改进方向。" |
| 绞杀者模式未推荐 | "系统可能推荐防腐层模式作为替代。这两种模式都适用于遗留系统改造——绞杀者侧重逐步替换，防腐层侧重隔离保护。选择取决于对需求文本的具体解析。" |
| 迁移步骤不够详细 | "迁移步骤由规则模板生成。当前模板提供 3+ 阶段的框架性步骤。如果需要更具体的步骤，LLM 润色可以增强——前提是 LLM 已配置且 refactoring-agent 触发了润色逻辑。" |

## 用例 6: 在线教育平台

### 输入

```
在线教育平台，支持万人同时听课，课程直播低延迟，互动消息及时推送。
```

### 预期触发特征

| 维度 | 命中 | 关键词证据 |
|------|------|-----------|
| high_concurrency | ✅ | 万人、同时 |
| real_time | ✅ | 直播、低延迟、及时推送 |

> 注: 特征命中较少（2 维）。如果 LLM 已配置，可能触发语义补全，补充 scalability 和 reliability。

### 预期推荐架构

| 排名 | 风格 | 评分演算（无 LLM 补全） |
|------|------|---------|
| **1** | **Event-Driven Architecture** | high_concurrency(+2) + real_time(+2) + 特定规则(+1) = **5** |
| 2 | Microservices | high_concurrency(+2) = **2**（主流保底） |
| 3 | Layered Architecture | 无标签命中 = **0**（主流保底基线） |

> 注: 当特征稀疏时，Top 2 和 Top 3 的分数差距较大——这恰好展示了主流保底策略的价值：即使只有 Event-Driven 明显合适，评委也能看到另外两种主流架构作为对比基线。

### 预期组合推荐

**Microservices + Event-Driven Architecture**（复杂度 1）

教育平台的课程管理（微服务边界）和实时互动（事件驱动）是天然的组合场景。

### 是否触发重构建议

**否**。

### 演示价值

展示**特征稀疏场景**下的系统行为——当只有 2 个维度命中时，评分差异更加鲜明。这也是展示 LLM 语义补全价值的场景——如果 LLM 已配置，可以看到补全后的特征数量和补全前的差异。

### 如果结果偏差，现场怎么解释

| 偏差情况 | 解释 |
|---------|------|
| 特征补全后推荐变化 | "LLM 补全可能新增 scalability 特征——这会提升 Microservices 和 Layered 的得分。这是 LLM 语义理解的正面价值：从'万人同时听课'推断出可扩展性需求。" |
| Layered 得 0 分 | "Layered 在这个场景确实不是最优选择——但我们的主流保底策略确保它出现在候选列表中，让用户有一个明确的对比基线：Event-Driven 5 分 vs Layered 0 分，解释为什么分层架构不适合这个场景。" |

## 用例 7: 医疗影像处理系统

### 输入

```
医疗影像处理系统，图像处理链路长，适合分阶段流水线，患者数据需要隐私保护和合规审计。
```

### 预期触发特征

| 维度 | 命中 | 关键词证据 |
|------|------|-----------|
| data_intensive | ✅ | 图像处理、影像 |
| security | ✅ | 隐私保护、合规审计 |
| complex_business | ✅ | 分阶段（可能触发，取决于词典覆盖） |

> 注: "流水线"和"分阶段"可能触发 Pipeline-Filter 相关的特征推理。"隐私保护"和"合规审计"明确触发 security。

### 预期推荐架构

**可能推荐** (特征覆盖稀疏，优先级依赖主流保底策略):

| 风格 | 概率 | 理由 |
|------|------|------|
| **Event-Driven Architecture** | 高 | data_intensive(+2)，主流保底 |
| Pipeline-Filter | 中 | data_intensive(+2)，但不一定进入 Top 3（被主流保底挤掉） |
| Layered Architecture | 中 | security(+2)，主流保底 |
| Microservices | 中 | security(+2)，主流保底 |

> 注: 这是推荐结果最不确定的用例之一——特征相对稀疏且没有触发强烈的特定规则。**不建议作为主演示用例**，但适合 Q&A 环节作为"边界案例"展示。

### 预期组合推荐

如果有意义，可能推荐 **Pipeline-Filter + Event-Driven Architecture**。但可能性较低——取决于特征命中是否足够。

### 是否触发重构建议

**否**。

### 演示价值

展示**系统在特征稀疏时的诚实表现**——不会强行给出高分推荐，而是通过主流保底策略给出一个"虽然没有明显最佳选择，但三种主流架构可供参考"的对比基线。适合回答"如果用户输入很模糊怎么办"的提问。

### 如果结果偏差，现场怎么解释

> "医疗影像场景的特征信号相对稀疏——'流水线'确实暗示了 Pipeline-Filter，但如果不命中 data_intensive 词典中的关键词，评分会比较保守。这体现了系统的设计原则——宁可给出保守的候选列表，也不强行推荐不合适的架构。LLM 语义补全可以增强对隐含特征的识别——如果 LLM 已配置，可以展示补全前后的差异。"

## 用例 8: 企业数据中台

### 输入

```
企业数据中台，ETL任务复杂，要求可扩展和可观测，支持批处理和流处理，多数据源接入。
```

### 预期触发特征

| 维度 | 命中 | 关键词证据 |
|------|------|-----------|
| data_intensive | ✅ | ETL、数据中台、批处理、流处理、数据源 |
| scalability | ✅ | 可扩展 |
| complex_business | ✅ | 数据中台（可能命中复杂业务词典） |

> 注: 这是 data_intensive 特征信号最强的用例之一——"ETL"、"批处理"、"流处理"、"数据源"四个关键词密集命中。

### 预期推荐架构

**可能推荐**:

| 风格 | 概率 | 理由 |
|------|------|------|
| **Event-Driven Architecture** | 高 | data_intensive(+2) + scalability(+2) = 4，主流保底 |
| Pipeline-Filter | 中-高 | data_intensive(+2) + scalability(+2) = 4，但不一定进入 Top 3 |
| Microservices | 中 | scalability(+2)，主流保底 |
| Layered Architecture | 中 | complex_business(+2) + scalability(+2) = 4（如果 complex_business 命中） |

### 预期组合推荐

**Pipeline-Filter + Event-Driven Architecture**（复杂度 2）

管道-过滤器处理 ETL 流水线，事件驱动处理流处理场景的异步事件。复杂度 2 反映了同时管理批处理和流处理管道的工程成本。

### 是否触发重构建议

**否**。

### 演示价值

展示**数据密集型场景的最强信号**。"ETL"、"批处理"、"流处理"、"数据源"密集命中 data_intensive 词典——展示特征提取的词典覆盖能力。同时也是 Pipeline-Filter + Event-Driven 组合推荐的最佳场景之一。

### 如果结果偏差，现场怎么解释

| 偏差情况 | 解释 |
|---------|------|
| Pipeline-Filter 不在 Top 3 | "Pipeline-Filter 得分约 4-5 分，但如果 Layered 也被 complex_business 和 scalability 各 +2，它与 Pipeline-Filter 分数持平。主流保底策略优先确保 Layered 出现在候选列表中。这是主流保底的代价——可能挤掉得分相当的非主流选择。我们设计文档中已记录了这一点。" |
| complex_business 未被命中 | "'数据中台'是一个相对新的概念，可能不在 complex_business 词典中。这展示了关键词词典的覆盖局限性——也是 LLM 语义补全存在的价值。" |
