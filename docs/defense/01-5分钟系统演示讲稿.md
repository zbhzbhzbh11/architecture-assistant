## 2. 演示前准备

### 2.1 环境启动（演示前 10 分钟完成）

```bash
# 进入项目目录
cd architecture-assistant

# 1. 确认 .env 已配置 LLM（可选，不配置则纯规则模式运行）
cat .env
# 应包含: LLM_API_BASE / LLM_API_KEY / LLM_MODEL

# 2. 启动全部服务（首次构建约 2-3 分钟）
docker compose up --build -d

# 3. 等待所有容器 healthy
docker compose ps
# 确认 8 个容器状态均为 running / healthy

# 4. 初始化 Neo4j 知识图谱（首次）
docker compose exec knowledge-base python init/init_neo4j.py

# 5. 验证各服务健康
curl http://localhost:8000/health
# 预期: {"status":"ok","service":"api-gateway","workflow_engine":"langgraph|manual","cache_backend":"memory"}
```

### 2.2 关键地址

| 地址 | 用途 |
|------|------|
| `http://localhost:3000` | 前端界面（演示主窗口） |
| `http://localhost:8000/docs` | Swagger API 文档（可展示微服务接口） |
| `http://localhost:7474` | Neo4j Browser（如 Neo4j 运行，可展示图数据库） |
| `http://localhost:8000/cache/stats` | 缓存统计（可展示缓存命中率） |

### 2.3 演示前检查命令

```bash
# 快速冒烟：发一个测试请求验证全链路
curl -s -X POST http://localhost:8000/api/v1/recommend \
  -H "Content-Type: application/json" \
  -d '{"requirement":"测试系统，需要高并发和实时响应"}' \
  | python -c "import sys,json; d=json.load(sys.stdin); print('OK' if d.get('final_report') else 'FAIL')"
# 预期输出: OK

# 缓存统计
curl -s http://localhost:8000/cache/stats | python -m json.tool
```

### 2.4 常见问题排查

| 问题 | 原因 | 解决 |
|------|------|------|
| `docker compose up` 报端口冲突 | 8000/3000 等端口被占用 | `netstat -ano \| findstr 8000` 查占用进程并关闭 |
| 前端页面空白 | nginx 未正确代理或 CORS 问题 | 检查浏览器控制台 Network 面板 |
| 响应很慢（>15s） | LLM API 网络延迟或超时 | 可临时注释 `.env` 中的 LLM 配置走纯规则模式（<2s） |
| Neo4j 初始化报连接拒绝 | Neo4j 容器尚未完全启动 | 等待 30s 后重试 `docker compose exec ...` |
| `langgraph` 未安装 | 当前环境缺少 langgraph 包 | 属于正常状态——系统自动回退手动编排，`workflow_engine: "manual"` |

## 4. 主演示用例逐步讲稿

> **使用输入**：`开发跨平台即时通讯系统，支持万人同时在线，消息实时可靠，后续扩展视频通话。`
>
> 以下为逐秒操作与逐字稿，[] 内为操作动作，其余为口述内容。总时长严格控制在 5 分钟。

---

### 0:00-0:30 | 打开系统并介绍输入

**[操作] 浏览器打开 `http://localhost:3000`，页面标题"架构风格智能助手"可见。**

> 各位老师好，我今天演示的是"基于大模型的软件架构风格智能助手"。
>
> 这个系统的核心能力是：你输入一段自然语言描述的系统需求，它会自动分析你的非功能特征，匹配知识库中的 10 种架构风格，给出 Top 3 推荐，并附带完整的决策依据。
>
> 现在我输入一个典型的即时通讯系统需求——
>
> "开发跨平台即时通讯系统，支持万人同时在线，消息实时可靠，后续扩展视频通话。"

**[操作] 将上述文本粘贴到 textarea 中。**

> 大家注意，这个需求提到了几个关键信息：万人同时在线说明高并发、消息实时说明需要低延迟、可靠说明不能丢消息、扩展视频通话说明系统需要弹性扩容。

**[操作] 点击"开始推荐"按钮，按钮变为 disabled 状态"分析中..."。**

---

### 0:30-1:20 | 展示需求特征提取

**[操作] 等待响应返回（约 2-10 秒，取决于 LLM 网络延迟），页面自动展示结果。**

> 好，结果已经返回了。首先我们看到页面最上方的**状态栏**——
>
> 左侧显示"实时计算"，说明这是一次全新的分析请求，没有命中缓存。
>
> 右侧显示编排引擎类型，如果显示"langgraph"说明使用了 LangGraph 工作流编排；如果显示"manual"说明当前环境未安装 langgraph，系统自动回退到了内置的手动编排——两种模式功能完全等价。

**[操作] 鼠标指向"识别特征"区块（页面第三个大区块）。**

> 接下来看**特征提取**的结果。系统识别出了 4 个关键维度：
>
> - **高并发**（命中了"万人"关键词）
> - **实时性**（命中了"实时"和"消息"）
> - **可靠性**（命中了"可靠"）
> - **可扩展性**（命中了"扩展"）
>
> 这 4 个维度是我们的 requirements-agent 通过 10 维关键词词典匹配出来的。每个特征标签旁边还有命中的具体关键词，做到了特征来源可追溯。
>
> 如果用户需求描述比较模糊、关键词命中很少，系统还会调用 LLM 来做语义补全，这个我们后面可以单独演示。

---

### 1:20-2:10 | 展示 Top 3 架构推荐

**[操作] 滚动到"架构对比矩阵"区块（页面第五个大区块，一张 6 列表格）。**

> 接下来是核心部分——**架构对比矩阵**。系统给出了 3 个候选架构，按得分从高到低排列：
>
> 第一名是**事件驱动架构（Event-Driven Architecture）**，得分 9 分。关键理由是：高并发 +2、实时性 +2、可靠性 +2、可扩展性 +2，再加上 Event-Driven 对高并发的额外规则加成 +1。
>
> 第二名是**微服务架构（Microservices）**，得分 6 分。高并发、可扩展、可靠各 +2。
>
> 第三名是**分层架构（Layered Architecture）**，得分 3 分。
>
> 大家注意一个设计细节：Layered、Microservices、Event-Driven 这三种主流架构**始终会出现在候选列表中**——这是我们设计的"主流架构保底策略"，确保推荐不偏离业界共识。

**[操作] 鼠标依次指向每一行的"优点"和"缺点"列。**

> 每一行都附带了该风格的优点和缺点分析，这些数据来自知识库中的 `architecture_styles.json`，不是 LLM 编造的。

### 2:10-3:00 | 展示最终推荐理由和对比矩阵

**[操作] 滚动回页面顶部"推荐结果"区块。**

> 回到**推荐摘要**部分。系统给出了核心推荐结论——
>
> 核心推荐是**事件驱动架构**，备选是微服务和分层架构。
>
> 再看下方的 LLM 摘要区域——如果当前配置了 LLM，这里显示的是 LLM 生成的详细推荐理由，包含推荐依据、优缺点分析和风险建议，使用 Few-shot Prompt 约束输出格式（3 个标注示例）。
>
> 如果没有配置 LLM，这里会显示规则模板生成的摘要，结构一样完整，包含推荐理由、优点、缺点——这是我们的**三级降级策略**：LLM 不可用时核心推荐链路不受影响。

**[操作] 滚动到"工作流追踪"区块（页面第四个区块）。**

> 这是**工作流追踪**——可以看到整个推荐过程经历了几个阶段：extract（特征提取）、match（架构匹配）、evaluate（评估决策）。每个阶段的耗时都有记录，方便定位性能瓶颈。

### 3:00-3:40 | 展示 Mermaid 拓扑图

**[操作] 滚动到"推荐拓扑图"区块（页面第六个区块）。**

> 这里是**Mermaid 拓扑图**——用可视化的方式展示了推荐架构的组件关系。

> 事件驱动架构的拓扑图显示了：事件生产者 → 事件总线 → 事件消费者 的经典结构。每种架构风格在知识库中都预置了对应的 Mermaid 拓扑图定义，共 10 张。

### 3:40-4:20 | 展示组合推荐 / ADR / 重构建议 / 图谱证据

**[操作] 依次滚动展示以下区块。**

**图谱证据（区块 7）**：

> 这是**知识图谱证据**。如果 Neo4j 已初始化，这里会显示通过 Cypher 查询匹配到的质量属性、适用场景、关联风险和可组合风格。每个属性匹配贡献 +2 分，上限为规则得分的 50%。
>
> 如果 Neo4j 未启动，这里会显示"无图谱证据"——这是我们设计的 JSON fallback 机制在起作用，不影响核心推荐。

**组合推荐（区块 8）**：

> 这是**架构组合推荐**。对于这个即时通讯场景，系统推荐"微服务 + 事件驱动"的组合方案。微服务负责服务边界和独立部署，事件驱动解耦服务间通信。这种组合特别适合高并发、实时性要求高的系统。

**重构建议（区块 9）**：

> 当前这个需求不涉及遗留系统重构，所以重构建议区块为空——这是正常的，只有输入包含"单体"、"耦合"、"拆分"等关键词时，refactoring-agent 才会检测到架构坏味并给出建议。我们后面用备选输入演示。

**ADR 决策记录（区块 10）**：

> 这是**架构决策记录**。每次推荐完成后，系统会自动生成一条 ADR，包含决策 ID、推荐结论和完整决策链。可以通过 API 接口 `/adr/{adr_id}` 随时追溯查询。

### 4:20-5:00 | 总结系统完整链路

> 好，最后总结一下我们刚刚走过的完整链路——
>
> 用户输入自然语言需求 → **api-gateway** 校验并编排 →
> **requirements-agent** 提取 10 维非功能特征 →
> **matching-agent** 规则评分 + 图谱融合，产出 Top 3 候选 →
> **evaluation-agent** LLM 投票 + 摘要，生成对比矩阵和风险评估 →
> ADR 自动写入 knowledge-base，整个决策链可追溯。
>
> 整个过程涉及 6 个后端微服务的协作，但用户只需前端一个操作就完成了。
>
> 核心设计理念是三句话：**规则保证下限，图谱增强关系，LLM 提升上限**。每一层都是独立可降级的，LLM 不可用、Neo4j 不可用、LangGraph 不可用时——核心推荐链路都不受影响。
>
> 以上是我的演示，欢迎各位老师提问。

## 6. 演示失败应急方案

### 6.1 前端打不开（`http://localhost:3000` 无法访问）

**症状**: 浏览器显示"无法连接"或空白页。

**应急步骤**:
1. 检查 Docker 容器状态：`docker compose ps`（确认 frontend 容器为 running）
2. 如容器未启动：`docker compose up -d frontend`
3. 如端口被占用：`docker compose down && docker compose up -d`
4. **兜底方案**: 直接用 curl 调用 API 展示结果，跳过前端：
   ```bash
   curl -s -X POST http://localhost:8000/api/v1/recommend \
     -H "Content-Type: application/json" \
     -d '{"requirement":"开发跨平台即时通讯系统，支持万人同时在线，消息实时可靠，后续扩展视频通话。"}' \
     | python -m json.tool
   ```
   然后口头解释各字段含义，同时展示 Swagger 文档 `http://localhost:8000/docs`。

### 6.2 LLM 不可用

**症状**: 响应速度很快（<2s），但推荐摘要比较模板化，llm_vote 为空。

**应急话术**:
> "当前 LLM 服务未配置或暂时不可达，系统自动降级为纯规则模式。大家可以看到——推荐结论、对比矩阵、风险建议这些核心功能完全正常。LLM 摘要换成了规则模板生成的文本，结构一样完整。这正是我们三级降级策略的设计目标：LLM 增强体验，但不绑定生存。"

**不需做任何操作**——纯规则模式本身就是正常功能，回归测试 20/20 通过。

### 6.3 Neo4j 不可用

**症状**: 图谱证据区块显示"无图谱证据"。

**应急话术**:
> "当前 Neo4j 图数据库未运行，知识库自动回退到了 JSON 文件后端。大家可以看到图谱证据区显示了'无图谱证据'，但推荐流程完全不受影响——因为我们设计了 JSON + Neo4j 双后端架构，Neo4j 是可选增强，不是必需依赖。"

**如需现场补救**（非必需）:
```bash
docker compose up -d neo4j
# 等待约 20s 后初始化
docker compose exec knowledge-base python init/init_neo4j.py
# 重新请求即可看到图谱证据
```

### 6.4 Docker 服务没启动

**症状**: `curl http://localhost:8000/health` 无响应。

**应急步骤**:
1. `docker compose down`（清理残留）
2. `docker compose up -d`（重新启动）
3. `docker compose ps`（确认 8 个容器都在运行）
4. 如果 Docker Desktop 本身未启动，先启动 Docker Desktop。

**极端兜底**: 如果 Docker 完全不可用，展示关键代码片段（`services/` 下各 Agent 的 `main.py`）和架构图（`docs/02-架构设计文档.md` 中的 C4 图），口头说明系统设计。

### 6.5 推荐结果不符合预期

**症状**: 推荐的首选架构不是 Event-Driven，或者得分和演示稿不一致。

**原因**: 如果 LLM 已配置，LLM 投票可能导致 +1 分 tie-break 改变排名。纯规则模式下推荐稳定。

**应急方案**:
1. **不重要**: 只要 Top 3 包含 Event-Driven / Microservices / Layered 中的至少一种，且推荐有据可查，就属于正常结果。
2. **话术**: "推荐结果受规则评分、学习权重、LLM 投票三者综合影响。当前结果体现了混合推理的特点——规则排名为主、LLM 投票做微调。"
3. **必要时**: 清空学习权重（`learned_weights.json` 重置为 `{}`）和缓存（`POST /cache/clear`）后重新请求。

答辩当天逐项勾选：

### 环境准备
- [ ] Docker Desktop 已启动，分配内存 ≥ 4GB
- [ ] `.env` 文件中 LLM 配置已验证（或决定走纯规则模式）
- [ ] `docker compose up --build -d` 执行成功
- [ ] `docker compose ps` 显示 8 个容器全部 running
- [ ] `curl http://localhost:8000/health` 返回 200
- [ ] Neo4j 知识图谱已初始化（如需要图谱证据展示）
- [ ] `python scripts/check_assignment.py --project-root .` 全部通过（作为备选展示材料）
- [ ] `curl -s -X POST http://localhost:8000/api/v1/recommend -H "Content-Type: application/json" -d '{"requirement":"测试"}'` 返回 HTTP 422（证明校验层正常工作）

### 浏览器准备
- [ ] 浏览器打开 `http://localhost:3000`，页面正常显示
- [ ] 浏览器打开 `http://localhost:8000/docs`，Swagger 文档正常显示
- [ ] F12 打开开发者工具 → Network 面板，清除历史记录
- [ ] 主演示输入文本已复制到剪贴板
- [ ] 备选输入文本已准备好（3 个）

### 演示节奏
- [ ] 已至少完整彩排 1 次（掐表 5 分钟）
- [ ] 逐字稿中关键段落已标注
- [ ] 知道每个区块在页面的什么位置（不需要现场找）
- [ ] 已确认滚动方向（11 个区块是纵向排列的，逐一滚动即可）

### 应急准备
- [ ] curl 命令已准备好（前端崩溃时备用）
- [ ] Swagger 文档页面已加载（API 直调展示备用）
- [ ] 架构设计文档（含 C4 图）已打开备用
- [ ] 知道如何快速重启服务（`docker compose restart`）
