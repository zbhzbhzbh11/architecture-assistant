## 第 5 页：Agent 协作机制

**标题：** Pipeline-Agent 协作机制

**内容：**
```
时序图：
  Gateway → Req.Agent  POST /extract  → {features}
          → Mat.Agent  POST /match    → {candidates}
          → Eval.Agent POST /evaluate → {final_report}
```
三个 Agent 的输入输出：
| Agent | 输入 | 输出 |
|---|---|---|
| Requirements | 需求文本 | 10 维特征 + 命中关键词 |
| Matching | 特征 | Top3 候选 + 评分理由 |
| Evaluation | 需求+特征+候选 | 推荐 + 矩阵 + 风险 |

**建议配图：** UML 时序图

**讲解要点：**
- Pipeline 模式：职责单一、可独立测试、可替换
- LLM 仅在 evaluation-agent 参与，降低不确定性扩散

## 第 6 页：LLM 集成方案

**标题：** LLM 集成与鲁棒性设计

**内容：**
```
两套 Prompt：
  llm_vote_style:  t=0.0 → "仅返回风格名"
  llm_summary:     t=0.3 → "结构化中文分析报告"

四层防护：
  [1] 20s 超时控制
  [2] try-except 全量捕获
  [3] 降级为规则引擎格式化摘要
  [4] 未配置 Key 时自动跳过

配置：OpenAI 兼容协议 → 支持 DeepSeek/通义千问/OpenAI 一键切换
```

**建议配图：** LLM 调用流程图 + 降级分支

**讲解要点：**
- 为什么 temperature 分档：投票要确定性，说明要表达力
- .env 自动加载，Key 不硬编码
- 降级后规则引擎可独立完成推荐

## 第 7 页：架构知识库设计

**标题：** 架构知识库设计

**内容：**
```
10 种架构风格 × 6 个字段：
  name / tags / best_for / pros / cons / topology_mermaid

扩展机制：
  POST /styles     → 动态新增风格，即时生效
  POST /feedback   → 记录用户确认/修正
  GET /feedback/stats → 统计各风格准确率

演进路径：JSON → Neo4j 图数据库
```

**建议配图：** 知识库 JSON 截图 + 风格列表

**讲解要点：**
- 恰好 10 种（满足作业 ≥10 要求）
- 每种风格的 topology_mermaid 字段使拓扑图数据驱动
- 反馈数据为后续自动权重学习积累素材

## 第 8 页：混合推理机制

**标题：** 规则引擎 + LLM 混合推理（核心创新点）

**内容：**
```
    规则引擎（确定性）          LLM 增强（语义）
    ────────────────          ──────────────
    tags 匹配    +2/tag       投票加分   +1
    专家规则     +1           中文说明   生成
    主流白名单   兜底          失败降级   规则独立运行
    ════════════════          ══════════════
       规则保下限                LLM 提上限
```

**建议配图：** 双引擎协同流程图

**讲解要点：**
- 为什么两者都要：纯规则泛化弱，纯 LLM 不稳定
- LLM 投票只 +1 分，不会颠覆规则排序
- 这是项目最大创新点

## 第 9 页：架构推荐流程

**标题：** 从需求到推荐的全链路

**内容：**
```
输入：自然语言需求
  ↓
[Step 1] 关键词匹配（10 维度 × ~100 词）
  ↓ 命中 ≤2？ → 触发 LLM 语义补全
[Step 2] 标签评分（10 风格逐一打分）
  ↓ 额外规则 + 主流白名单 → Top3
[Step 3] 混合推理（规则排序 + LLM 投票 + LLM 说明）
  ↓
输出：recommended_style + comparison_matrix + risk_and_suggestions + topology_mermaid
```

**建议配图：** 端到端数据流图（标注每个步骤的输出字段）

**讲解要点：**
- 每条字段的来源和流向
- 三步之间通过 API Gateway 串联
- 对比矩阵每行附 topology_mermaid 供前端渲染

## 第 13 页：异常处理与可靠性

**标题：** 异常处理与可靠性设计

**内容：**
```
网关层：统一 HttpException → 502 返回
       trust_env=False 避免代理干扰
LLM 层：20s 超时 → try-except → 降级
       Key 未配置 → 自动跳过
前端层：按钮防重复提交
       错误分支不崩溃
容错性：单 Agent 故障不影响网关
        LLM 完全不可用仍可输出完整推荐
```

**建议配图：** 异常处理流程图（正常路径 vs 降级路径）

**讲解要点：**
- 降级后规则引擎可独立运行
- 核心推荐链路不受 LLM 不可用影响
