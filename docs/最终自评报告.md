## 验收时应该重点展示的亮点

### 亮点 1：混合推理机制（最大加分项）

**展示方式：** 打开浏览器开发者工具 Network 标签，发送一个请求，逐步展开响应 JSON，指出：
- `decision_basis.rule_engine`：规则引擎的理由（中文可读）
- `decision_basis.llm_vote`：LLM 的投票结果
- `decision_basis.llm_summary`：LLM 生成的结构化分析报告

**话术：** "注意看——LLM 投票是 Microservices，但最终推荐是 Event-Driven。因为规则评分（7 分 vs 5 分）主导了排序。这就是混合推理的核心价值：规则保下限，LLM 只做轻量增强，不会颠覆规则决策。"

### 亮点 2：三层可解释证据链

**展示方式：** 在前端页面从上往下指：
1. 特征卡片 → "识别到了什么需求信号"（feature_hits）
2. 对比矩阵的 key_reasons 列 → "为什么进候选集"（rule_engine）
3. 推荐摘要的 LLM 分析报告 → "为什么最终选它"（llm_summary）

### 亮点 3：数据驱动拓扑图

**展示方式：** 连续输入两条不同需求，展示拓扑图自动切换。
- 输入 IM 需求 → Event-Driven 拓扑图
- 输入政务需求 → Client-Server 拓扑图

**话术：** "拓扑图不是静态图片。10 种架构风格的 Mermaid 定义存储在知识库 JSON 中，API 动态返回，前端实时渲染。修改图定义只需改 JSON，无需改代码。"

### 亮点 4：降级鲁棒性

**展示方式：** 展示 evaluation-agent 代码中的多层防护（LLM 未配置检测→20s/25s 超时→try-except→格式化降级摘要）。可选：临时把 .env 中的 API_KEY 改错，展示降级后仍可输出完整推荐。

**话术：** "知识进化权重拉取也有独立的降级保护——matching-agent GET /feedback/weights 失败时静默跳过，不影响核心评分链路。"

### 亮点 5：自动化测试体系

**展示方式：** 终端运行 `python scripts/check_assignment.py`，展示全部通过。运行 `python tests/run_regression.py`，展示 20/20 通过、5 项指标 100%。运行 `pytest tests/unit/ -v` 展示 23 条单元测试全部通过。

### 亮点 6：知识进化闭环演示（新增）

**展示方式：** 按以下步骤现场演示完整闭环：

1. **初始推荐**：输入"政务系统私有化部署，强调安全隔离和可靠交付" → 系统推荐 Client-Server（得分靠部署约束标签）
2. **提交反馈修正**：调用 `POST /feedback` 3 次，每次 `user_choice="Layered Architecture"`, `comment="修正为分层架构"`
3. **验证权重生成**：展示 `learned_weights.json` 内容 —— `{"reliability": {"Layered Architecture": 3}, "deployment_constraint": {"Layered Architecture": 3}, "security": {"Layered Architecture": 3}}`
4. **再次推荐**：重新提交同一条需求 → Layered Architecture 的得分里出现 `"learned boost: reliability->Layered Architecture (confirmed 3x)"` / `"learned boost: deployment_constraint->Layered Architecture (confirmed 3x)"` / `"learned boost: security->Layered Architecture (confirmed 3x)"` 三条加分记录，总分上升，可能超越 Client-Server 成为核心推荐

**话术：** "这就是知识进化的完整闭环：用户提交反馈 → 系统从需求文本中提取特征 → 更新特征-风格关联权重 → 下次匹配时读取权重并加分 → 推荐结果随反馈积累持续优化。虽然是基于计数累加的简单算法，但闭环链路完整、框架可平滑升级为贝叶斯权重或 TF-IDF 加权。"
