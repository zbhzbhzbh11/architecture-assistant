## 可能被老师质疑的地方

| # | 质疑点 | 被问概率 | 应对策略 |
|---|--------|---------|---------|
| 1 | **语义理解主要靠关键词** | 中 | Few-shot LLM 在规则命中少时触发补全, "规则优先 + LLM 补齐" 是工程上更可控的设计, 避免 LLM 幻觉扩散 |
| 2 | **知识进化算法简单** | 中 | 完整闭环已实现（反馈→提取特征→更新权重→评分生效）。框架支持平滑升级（计数→贝叶斯→TF-IDF） |
| 3 | **Neo4j 非必需, 加复杂度** | 低 | JSON fallback 保证零外部依赖可运行, Neo4j 是增强能力。`KNOWLEDGE_BACKEND=auto/json/neo4j` 按需切换 |
| 4 | **缓存是内存/SQLite 而非 Redis** | 低 | 课程项目单进程场景足够, 生产环境可平滑切换（接口兼容）。TTL + knowledge_version 机制设计完备 |
| 5 | **平均时延 ~10s（LLM 模式）** | 低 | 缓存命中后 ~5ms。LLM 两次调用已通过 asyncio.gather 并行化。课程场景为离线辅助决策, 可接受 |
