# 前端验证案例

## 1. 验证目标
用于验证前端页面从输入需求到展示推荐结果的完整交互是否正常，重点覆盖：
- 页面交互与按钮状态
- 接口调用成功与失败分支
- 结果展示完整性（推荐结论、对比矩阵、拓扑图、原始响应）
- 基本可解释性字段展示

## 2. 执行前检查
1. 启动后端与前端服务。
2. 浏览器打开前端页面。
3. 打开浏览器开发者工具，观察 Network 中 recommend 请求。

## 3. 前端通过标准
每条成功用例都应满足：
1. 点击开始推荐后，按钮进入分析中状态，完成后恢复。
2. 页面显示推荐结果区域。
3. 推荐结果中最终推荐不为空。
4. 架构对比矩阵至少有 3 行候选。
5. 推荐拓扑图成功渲染（Mermaid 图可见）。
6. 原始响应中包含 extracted_features、feature_hits、candidates、final_report 字段。

## 4. 功能验证案例（成功路径）

| 用例ID | 前端输入文本 | 预期最终推荐（参考） | 重点检查 |
|---|---|---|---|
| FE-01 | 开发跨平台即时通讯系统，支持万人在线，消息实时可靠，后续扩展视频通话。 | Event-Driven Architecture | 实时类需求命中，拓扑图应为事件驱动样式 |
| FE-02 | 构建电商平台，订单支付库存需要强一致事务，双11高峰要抗压。 | Layered Architecture | 输出需含一致性相关解释，矩阵完整 |
| FE-03 | 企业内部审批系统，流程复杂、规则多、需要稳定迭代。 | Layered Architecture | 结果稳定，说明偏可维护与分层治理 |
| FE-04 | 日志分析平台，每秒采集百万条日志并实时告警。 | Event-Driven Architecture | 高频事件场景，推荐应偏事件驱动 |
| FE-05 | 银行核心账务系统，要求高可靠和强一致，审计可追溯。 | Layered Architecture | 风险建议中应体现可靠性与审计约束 |
| FE-06 | 在线教育平台，课程直播和互动消息需要低延迟。 | Event-Driven Architecture | 低延迟与实时特征应可见 |
| FE-07 | 多团队协作开发SaaS平台，需要模块独立发布。 | Microservices | 推荐应体现服务拆分与独立发布 |
| FE-08 | 医疗影像处理系统，图像处理链路长，适合分阶段流水线。 | Pipeline-Filter | 拓扑图和解释应体现流水线处理思路 |
| FE-09 | 内容管理系统，业务逻辑清晰，强调维护性和权限分层。 | Layered Architecture | 分层理由明确，矩阵完整 |
| FE-10 | 外卖平台，订单峰值高，配送状态实时更新。 | Event-Driven Architecture | 高并发与实时状态驱动推荐 |

说明：
- 预期最终推荐为参考值，允许在知识库更新后出现合理变化。
- 若最终推荐变化，但候选完整、解释充分、矩阵正确，也可判定通过。

## 5. 异常与边界案例

| 用例ID | 操作步骤 | 预期结果 |
|---|---|---|
| FE-ERR-01 | 输入为空，点击开始推荐 | 不应出现前端崩溃；若后端返回错误，页面应显示请求失败信息 |
| FE-ERR-02 | 输入超长文本（例如 5000 字） | 页面可提交；若超时应展示失败提示，按钮最终恢复 |
| FE-ERR-03 | 停止网关服务后点击开始推荐 | 页面显示请求失败，且不白屏、不死锁 |
| FE-ERR-04 | 网速限速或断网后提交 | 页面应进入失败分支，输出错误文本 |
| FE-ERR-05 | 连续快速点击开始推荐 | 按钮禁用生效，避免重复并发提交 |

## 6. 可解释性核验点
在原始响应中抽查以下内容：
1. extracted_features 中有多个 true 特征。
2. feature_hits 中能看到命中的关键词证据。
3. candidates 至少 3 项，且包含主流架构之一。
4. final_report 中有 recommended_style、decision_basis、comparison_matrix、risk_and_suggestions。

## 7. 回归建议
1. 每次修改需求解析词典、匹配规则、评估逻辑后，至少重跑 FE-01 到 FE-10。
2. 每次修改前端展示逻辑后，必须重跑 FE-ERR-01 到 FE-ERR-05。
3. 发布前抽样检查 3 条成功用例和 2 条异常用例，确保按钮状态和错误分支可用。
