## Q1: "Compound AI System"——这个说法怎么理解？你的系统哪里 Compound 了？

**面试官为什么问**：Compound AI 不是标准术语，面试官可能在测试你是否在堆概念。

**推荐回答**：

> Compound AI System 的核心思想是"用多个组件协作完成 AI 任务，而不是靠一个模型"。我的系统是典型的 Compound AI：规则引擎做确定性特征提取和评分（可解释但僵硬），Neo4j 知识图谱做结构化关系推理（可查询但数据有限），LLM 做模糊语义补全和自然语言生成（灵活但不可靠）。三者各司其职——规则主导、图谱增强、LLM 辅助。这就是"Compound"——不是一个模型做所有事，而是多个组件各展所长。

**回答中的加分点**：
- 直接定义 Compound AI 的含义
- 三项能力各自的优势和局限都说清楚

**回答时不要踩的坑**：
- 不要只说是"多个 AI 组件协作"——太泛，要说清楚每个组件做什么

## Q2: "6 个独立微服务"——你说独立，到底怎么独立的？能不能合并？

**面试官为什么问**：判断"微服务"是真拆分还是假拆分。

**推荐回答**：

> 每个服务有独立的 Dockerfile、独立的 Python 进程、独立的端口、独立的 healthcheck 端点。它们通过 HTTP 通信（httpx），通过环境变量配置下游地址。任何两个服务不共享内存、不共享数据库连接。合并技术上可以——比如把三个 Agent 合并为一个 FastAPI 应用。但我保留拆分是因为职责清晰和故障隔离——evaluation-agent 的 LLM 调用超时不会阻塞 requirements-agent 的关键词提取。

**回答中的加分点**：
- 四个"独立"（Dockerfile/进程/端口/healthcheck）
- 能说出合并的代价（故障传播）

**回答时不要踩的坑**：
- 不要说"因为微服务就是好"——要说出具体的"独立"标准

## Q3: "LangGraph StateGraph 编排多 Agent 工作流"——如果没装 LangGraph，系统还能跑吗？

**面试官为什么问**：判断编排层的依赖是否硬编码。

**推荐回答**：

> 能跑。启动时 `build_workflow()` 先尝试 `import langgraph`——如果失败，返回 None，系统标记 `WORKFLOW_ENGINE = "manual"`，所有请求走手动 httpx 编排。运行时如果 LangGraph 编排抛异常，`try/except` 捕获后也切换到手动模式。手动编排和 LangGraph 编排的输出完全一致——功能等价。答辩时我当着评委的面注释掉 langgraph 的 import，系统继续正常运行。

**回答中的加分点**：
- 双重 fallback（启动时 + 运行时）
- "功能等价"——强调用户无感知
- 提到"答辩时当面演示"——增加可信度

**回答时不要踩的坑**：
- 不要说"LangGraph 一般不会出问题"——这是回避问题

## Q4: "7 条架构规则 + 学习权重"——7 条规则具体是什么？学习权重怎么更新的？

**面试官为什么问**：简历上的数字必须能展开讲。

**推荐回答**：

> 7 条规则分别是：高并发→Event-Driven +1、多团队→Microservices +1、强一致→Layered +1、实时+数据密集→Pipeline-Filter +1、高并发+数据密集→CQRS +1、高并发+强一致→Microservices +1。
>
> 学习权重来自用户反馈：每次用户提交反馈时，系统从反馈文本中提取活跃特征，对每个特征-风格关联的权重 +1。当关联累计确认 ≥2 次时，`score_style()` 给予 +1 的学习加分。当前 `learned_weights.json` 仅有 3 条记录——框架已就绪，数据待积累。

**回答中的加分点**：
- 7 条规则一一说出
- 学习权重的触发条件和数据现状都诚实

**回答时不要踩的坑**：
- 不要只说"就是一些架构经验"——必须能逐条说出

## Q5: "Neo4j 知识图谱增强推理"——增强体现在哪？和只用规则有什么区别？

**面试官为什么问**：判断图谱是否只是"挂个 Neo4j 的名字"。

**推荐回答**：

> 增强体现在两个层面。第一，特征-风格的隐式关联——规则引擎只覆盖了 7 种明确关联，图谱的 HAS_QUALITY 关系可以覆盖规则未定义的组合。比如某个特征和风格之间没有硬编码规则，但图谱中有一条边，`graph_match()` 就能给出 +2 的图谱分。第二，多跳信息——图谱可以同时告诉匹配 Agent："这个风格适合什么场景"、"有什么风险"、"可以和哪些风格组合"——这些是 JSON 读取需要多次查询才能拼出来的。
>
> 但图谱加分有 50% 上限——它增强推荐但不主导推荐。和只用规则的区别是：图谱能发现规则没覆盖的关联，但发现不了规则没定义的——因为图数据本身也是预定义的。

**回答中的加分点**：
- 两个层面（隐式关联 + 多跳信息）
- 提到 50% cap——没有夸大图谱的作用

**回答时不要踩的坑**：
- 不要暗示图谱可以自动发现新知识——图的数据是预定义的

## Q6: "Few-shot LLM 语义补全"——Few-shot 是什么？怎么设计的？

**面试官为什么问**：Prompt Engineering 的核心技能。

**推荐回答**：

> Few-shot Prompt 是在 Prompt 前面放几个范例，让 LLM 学习输出格式和判断风格。需求侧 6 个示例覆盖模糊语义、否定过滤、安全合规、数据密集、强一致、重构倾向。评估侧 3 个示例覆盖 Event-Driven/Microservices/Layered 三种核心风格的完整报告格式。每个示例都是"（需求文本，期望输出 JSON）"的配对。

**回答中的加分点**：
- 能举例 6+3 的覆盖场景
- "范例让 LLM 学习输出格式"——简洁的 Few-shot 解释

**回答时不要踩的坑**：
- 不要把 Few-shot 说成 Fine-tuning——完全不同的概念

## Q7: "内存/SQLite 双后端 LLM 缓存"——为什么做双后端？怎么切换？

**面试官为什么问**：判断你是否有"为不同环境设计不同方案"的意识。

**推荐回答**：

> 内存缓存是默认方案——零依赖、零配置、重启丢失。适合开发环境和单次答辩。SQLite 缓存提供跨重启持久化——适合长期运行积累缓存。两个后端通过环境变量 `CACHE_BACKEND` 切换——api-gateway 在启动时根据变量 import 不同的模块，两个模块的接口完全一致（`get/set/clear/stats`）。上层代码不感知使用的是哪个后端。

**回答中的加分点**：
- 两个后端各适合什么场景
- 接口一致性 + 启动时切换

**回答时不要踩的坑**：
- 不要说"SQLite 更好"——场景不同选择不同

## Q8: "ADR 架构决策自动生成"——ADR 是什么？为什么需要它？

**面试官为什么问**：ADR 对于非架构方向的面试官可能不熟悉。

**推荐回答**：

> ADR 是 Architecture Decision Record——架构决策记录。就像医生写病历：记录症状、诊断、用药、建议。每次推荐完成后，系统自动创建一条 ADR——包含需求是什么、推荐了什么风格、为什么推荐（规则理由 + LLM 摘要）、有哪些备选、有什么风险。ADR 的价值是"决策追溯"——三个月后回头看，你能知道当时为什么做了这个选择。

**回答中的加分点**：
- "病历"的类比——非架构方向的面试官也能理解
- 说出了 ADR 包含的具体字段

**回答时不要踩的坑**：
- 不要说"就是记录日志"——ADR 比日志结构化得多

## Q9: "架构组合推荐"——组合推荐和单一推荐有什么区别？会冲突吗？

**面试官为什么问**：判断你是否区分了两个功能。

**推荐回答**：

> 不冲突——它们是互补的。单一推荐回答"单个架构风格谁最合适"，组合推荐回答"哪两个风格联手效果更好"。报告里两者同时呈现——单一推荐（核心推荐+备选）在前，组合推荐作为"进阶建议"在后。组合评分公式是：组成风格的分数和 + 特征覆盖互补加分 + 图谱 CAN_COMBINE_WITH 加分 - 复杂度惩罚。5 种预定义组合模式覆盖了最常见的组合场景。

**回答中的加分点**：
- "互补"的定位清晰
- 四个因素的评分公式说得出

**回答时不要踩的坑**：
- 不要说"组合推荐更好"——两者适合不同场景

## Q10: "架构重构建议"——你的重构建议可靠吗？怎么识别坏味？

**面试官为什么问**：判断你是否诚实——基于关键词的检测天然不可靠。

**推荐回答**：

> 可靠性有限——这是诚实评价。坏味检测基于需求文本中的关键词（如"单体""耦合""老系统"），而非对实际代码的架构度量。这意味着：用户说"我们之前的系统是单体的"会触发坏味，但他可能根本不需要重构。这是关键词检测的固有局限。LLM 润色可以改善表述质量但不能改善判断准确性。

**回答中的加分点**：
- 直接诚实说"可靠性有限"
- 区分了"关键词检测"和"代码度量"

**回答时不要踩的坑**：
- 不要夸大——"能自动检测架构坏味"就是夸大

## Q11: "全链路 LLM 降级"——具体哪些地方降级了？怎么降的？

**面试官为什么问**：简历上的"全链路"必须能一一列举。

**推荐回答**：

> 8 个降级点。requirements-agent 语义补全：没配 LLM 或超时 → 返回纯规则特征。evaluation-agent 投票：LLM 不可用 → 返回 None，规则排序不变。evaluation-agent 摘要：LLM 不可用 → `_fallback_summary()` 用规则理由拼结构化中文报告。refactoring-agent 润色：LLM 不可用 → 使用规则模板。LangGraph 编排：ImportError 或运行时异常 → 手动 httpx。Few-shot 模块：ImportError → 零样本 Prompt。ADR 写入：超时/失败 → `adr_status: "failed"`。重构建议调用：超时 8s → `refactoring_advice: {}`。
>
> 每一个降级点都有对应的 try/except 或条件检查。这就是我说的"全链路"——没有遗漏任何一个 LLM 调用点。

**回答中的加分点**：
- 8 个点一一列举，不跳跃
- "没遗漏任何 LLM 调用点"——证明全链路是真实的

**回答时不要踩的坑**：
- 不要说"有一些降级"——要能全部列出

## Q12: "43 项自动验收 100% 通过"——验收检查了什么？你自己写的脚本检查自己的项目，有什么说服力？

**面试官为什么问**：测试你的自我批判能力。

**推荐回答**：

> 43 项覆盖 7 大类——目录结构、Agent 数量、知识库、测试用例、LLM 集成、API 端点、可视化。自动验收验证的是"存在性"而非"正确性"——它检查"文件在不在"、"API 有没有"、"风格够不够"，不检查"推荐对不对"。它的价值是"省去手动检查"——不用一个个去确认"测试用例够 20 条吗"、"每个风格都有 pros/cons 吗"。

> 说服力来自于它不是唯一验证——我还有 20 条回归用例的 5 维验证（这才是验证"对不对"的），以及 76 条单元测试（验证"函数行为对不对"）。验收是"门禁"——确保项目骨架完整。回归测试是"体检"——确保核心功能正常。

**回答中的加分点**：
- 区分"存在性验证"和"正确性验证"
- "门禁 vs 体检"的类比

**回答时不要踩的坑**：
- 不要说"我自己写的当然能通过"——这说明不了什么

## Q13: "FastAPI 微服务"——FastAPI 的性能优势在你的系统中有体现吗？

**面试官为什么问**：判断你不是"为了用 FastAPI 而用 FastAPI"。

**推荐回答**：

> 在当前 QPS ≈ 1 的场景下，FastAPI 的性能优势（异步非阻塞）体现不明显。我选择 FastAPI 更多是因为：第一，Pydantic 自动校验请求/响应——减少了大量手动类型检查代码；第二，自动生成 OpenAPI 文档——6 个服务 23 个端点，Swagger UI 一目了然；第三，async/await 原生支持——api-gateway 的异步编排（httpx AsyncClient + asyncio.gather）写出自然。

**回答中的加分点**：
- 诚实承认性能优势没体现
- 三个理由都和实际开发体验相关——有说服力

**回答时不要踩的坑**：
- 不要说"因为 FastAPI 快"——在当前场景下体现不出来

## Q14: "Docker Compose 编排"——你遇到过容器启动顺序的问题吗？怎么解决的？

**面试官为什么问**：容器编排的实战问题。

**推荐回答**：

> 遇到过。Neo4j 容器需要 15-30 秒才能接受 Cypher 查询，但 `depends_on` 只检查容器是否启动（不检查服务就绪）。解决方案是 Neo4j 容器配置了 healthcheck——`cypher-shell 'RETURN 1'`——其他依赖 Neo4j 的服务（matching-agent、knowledge-base）等 healthcheck 通过后才建立连接。knowledge-base 内部还有应用层重试——如果第一次连接失败，会等待后重试。

**回答中的加分点**：
- 具体的问题 + 具体的解决方案
- healthcheck + 应用层重试——双保险

**回答时不要踩的坑**：
- 不要说"一切顺利"——不真实

## Q15: "请求级缓存，SHA256 键 + knowledge_version 自动失效"——为什么不用简单的字符串相等？

**面试官为什么问**：判断你是否理解缓存键设计。

**推荐回答**：

> 因为缓存键需要承载三个维度的差异性：需求内容（相同需求 = 相同结果）、LLM 模型（换了模型不共享缓存）、知识库版本（知识库更新旧缓存作废）。简单的字符串相等只能覆盖第一个维度。SHA256 将"requirement + model + knowledge_version"三个字符串压缩为固定长度的 key——既节省存储，又保证了唯一性。knowledge_version 用 styles 文件 MD5——改了知识库自动变 key——实现了"零手动"的缓存失效。

**回答中的加分点**：
- 三维度差异性——每个都有实际意义
- "零手动"——设计的优雅之处

**回答时不要踩的坑**：
- 不要说"为了去重"——去重只是第一层

## Q16: 你简历里写了"提高了系统可用性"——你怎么量化"可用性提升"？

**面试官为什么问**：这是简历常见的高风险措辞。

**推荐回答**：

> 坦白说，"提高了系统可用性"这六个字在我的项目里不准确——因为它暗示有一个"之前"和"之后"的对比，而我的降级设计是从一开始就内建的，没有对比基线。更准确的表述是"通过全链路降级设计，系统在 LLM 不可用时仍能正常运行"——这是一个"能力描述"而非"提升度量"。量化的话，可以用"20 条回归用例在 LLM 全关状态下 100% 通过"来证明降级机制的有效性。

**回答中的加分点**：
- 主动修正自己的表述——自我批判
- 用"能力描述"替代"提升度量"——严谨

**回答时不要踩的坑**：
- 不要硬编一个提升百分比——没有基线就没有提升
