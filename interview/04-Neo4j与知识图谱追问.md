# 04 · Neo4j 与知识图谱追问

> 15 个问题。面试官在这里判断你理解"图数据库区别于关系型数据库的核心价值"，以及对 fallback 设计的工程意识。

---

## Q1: 为什么用 Neo4j 而不是 PostgreSQL？

**面试官为什么问**：最经典的问题。判断你是否真正理解图数据库的优势。

**推荐回答**：

> 架构知识天然是图状的——一个架构风格关联多个质量属性，一个质量属性被多个风格共享，风格之间还可以互组合。用关系型数据库，这种多对多关系需要关联表和 JOIN。用 Neo4j，关系是一等公民——`(s)-[:HAS_QUALITY]->(q)` 一行 Cypher 就能遍历所有关联。但诚实地说，在当前 10 种风格的规模下，PostgreSQL 和 Neo4j 的性能差异可以忽略。Neo4j 的价值更多是**模型匹配**——图模型更自然地表达了架构知识的拓扑结构。

**回答中的加分点**：
- 诚实承认规模小所以性能差异不大
- "模型匹配"而非"性能优势"——抓住了图数据库的核心价值

**回答时不要踩的坑**：
- 不要只说"图数据库快"——在小数据量下不快
- 不要贬低关系型数据库——两者各有适用场景

## Q2: 你的 Neo4j 图模型是什么样的？画出来。

**面试官为什么问**：判断你是否真正理解自己的图模型——能画出来才算掌握。

**推荐回答**：

> *（在白板上画）*
>
> 六种节点。核心是 ArchitectureStyle（10 个），它通过 HAS_QUALITY 关联 QualityAttribute（10 个，如 high_concurrency、real_time），通过 SUITABLE_FOR 关联 Scenario（如 "enterprise app"），通过 HAS_RISK 关联 Risk（如"事件一致性设计难度大"），通过 COMPLEMENTS 关联到另一个 ArchitectureStyle（表示可组合）。
>
> ADR 节点通过 RECOMMENDS 关系指向推荐的 ArchitectureStyle，通过 BASED_ON 关系指向依据的 QualityAttribute。Feedback 是独立节点，记录用户反馈。
>
> 当前数据规模：10 个风格节点 + 10 个质量属性节点 + 约 12 个场景节点 + 约 6 个风险节点 = 约 38 个节点。关系总数约 80-100 条。

**回答中的加分点**：
- 说得出每种节点的数量和关系的大致数量
- 能说明"哪种关系最重要"（HAS_QUALITY 是核心推理关系）

**回答时不要踩的坑**：
- 不要画错关系方向——HAS_QUALITY 是 Style→Quality，不是反过来

## Q3: 如果 Neo4j 挂了，你的系统怎么办？

**面试官为什么问**：判断你是否考虑了外部依赖的故障。

**推荐回答**：

> 通过 `KNOWLEDGE_BACKEND` 环境变量控制三种模式。`auto` 模式（默认推荐）：优先 Neo4j，不可用时自动 fallback 到 JSON 文件存储。切换对上层 Agent 完全透明——`_repo()` 统一调度函数在 GraphRepository 返回 None 时自动调用 JsonRepository。核心推荐链路不受影响——只是图推理增强的额外加分（capped 到 50%）会缺失。

**回答中的加分点**：
- 三种模式说得清楚（json/neo4j/auto）
- "切换对上层透明"——良好的抽象设计

**回答时不要踩的坑**：
- 不要说"Neo4j 不会挂"——所有外部服务都可能挂

## Q4: Neo4j 和 JSON 的数据一致性怎么保证？

**面试官为什么问**：双存储的核心挑战。

**推荐回答**：

> 当前的一致性保证是"JSON 为主、Neo4j 同步"。新增风格时通过 `_repo()` 调度同步写入两边。但如果一边成功一边失败（如 Neo4j 写入超时但 JSON 写入成功），当前没有分布式事务保证——两边可能短暂不一致。在课程项目的规模下（数据量小、写入频率极低），这种不一致的概率极低。如果要在生产环境用，可以引入 Outbox 模式——写入 JSON 的同时写入 outbox 事件，独立 worker 消费事件同步到 Neo4j，保证最终一致性。

**回答中的加分点**：
- 诚实承认没有事务保证
- 能说出生产环境应该用什么方案（Outbox / 事件溯源）

**回答时不要踩的坑**：
- 不要说"两边完全一致"——没有分布式事务就不可能完全一致

## Q5: 你的 graph_match() 查询逻辑是什么？

**面试官为什么问**：这是知识图谱最核心的查询，判断你是否理解自己的代码。

**推荐回答**：

> 输入是活跃特征列表（如 `["high_concurrency", "real_time"]`）。对每个特征，执行 Cypher 查询：找到名为该特征的 QualityAttribute 节点，然后沿着 HAS_QUALITY 关系的反向找到关联的 ArchitectureStyle 节点。同时拉取该风格关联的 Scenario、Risk 和 COMPLEMENTS 风格。每个匹配的质量属性为该风格贡献 +2 图谱分。多个特征可能匹配到同一个风格，分数累加。最终按 graph_score 降序返回。

**回答中的加分点**：
- 说了查询方向（HAS_QUALITY 反向遍历）
- 提到了分数累加逻辑

**回答时不要踩的坑**：
- 不要把匹配逻辑说反——是从 QualityAttribute 找 ArchitectureStyle

## Q6: 你的知识图谱数据是怎么初始化的？

**面试官为什么问**：判断你是否理解数据的来源和维护。

**推荐回答**：

> 通过初始化脚本 `init/init_neo4j.py`。脚本读取 `architecture_styles.json` 的数据，对每个风格创建 ArchitectureStyle 节点（含 name、pros、cons、topology_mermaid 等属性），遍历 tags 创建或匹配 QualityAttribute 节点并建立 HAS_QUALITY 关系，遍历 best_for 创建 Scenario 节点并建立 SUITABLE_FOR 关系。JSON 是"数据源"（source of truth），Neo4j 是"推理引擎"。

**回答中的加分点**：
- "JSON 是数据源，Neo4j 是推理引擎"——角色定位清晰

**回答时不要踩的坑**：
- 不要说"手动一个个创建的"——有脚本，不是手动的

## Q7: 图数据库在 10 种风格的规模下，真的比 JSON 快吗？

**面试官为什么问**：测试你是否诚实——小数据量下图不会更快。

**推荐回答**：

> 说实话，不会。10 种风格、约 80-100 条关系——这个规模下 JSON 的内存查找可能比 Neo4j 的 Cypher 查询更快（省去了网络往返和查询解析）。Neo4j 的优势在数据量 100+ 风格、1000+ 关联时才会体现——多跳关系遍历在关系型数据库中需要多表 JOIN，在图中是 O(1) 的关系跳转。在当前的规模下，Neo4j 的价值体现在**模型匹配**和**框架就绪**，而非性能优势。

**回答中的加分点**：
- 诚实承认小数据量下图不快
- 说出了图的性能优势在什么条件下体现

**回答时不要踩的坑**：
- 不要说"Neo4j 就是比 JSON 快"——这是错的

## Q8: COMPLEMENTS 关系是怎么来的？

**面试官为什么问**：判断你是否理解图中每个关系的含义和来源。

**推荐回答**：

> COMPLEMENTS 关系表示两个架构风格可以组合使用。数据来源于对架构文献和工程实践的分析——比如 Microservices 和 Event-Driven 经常组合（微服务管理服务边界，事件驱动管理服务间通信），CQRS 和 Event Sourcing 经常组合（读写分离 + 事件溯源）。当前这些关系是预定义的——初始化脚本从 JSON 数据中读取并创建。不是从数据中自动挖掘的。

**回答中的加分点**：
- 说明了关系的来源（文献 + 实践）
- 诚实说"预定义"而非自动挖掘

**回答时不要踩的坑**：
- 不要说"自动发现的"——没有图学习算法

## Q9: 为什么要把 ADR 同步到 Neo4j？有什么价值？

**面试官为什么问**：判断你是否理解"知识积累"的价值。

**推荐回答**：

> ADR 在 Neo4j 中成为了连接需求、风格和质量属性的枢纽节点。通过 `(:ADR)-[:RECOMMENDS]->(:ArchitectureStyle)` 和 `(:ADR)-[:BASED_ON]->(:QualityAttribute)`，你可以查询"Event-Driven 风格在哪些需求场景下被推荐过"、"高并发特征触发了多少条 ADR"——这些查询在 JSON 文件中需要遍历所有记录并手动过滤，在 Neo4j 中一行 Cypher 就能完成。这为后续的推荐效果分析提供了基础——比如发现"某个风格被推荐但用户反馈率低"，从而调整评分权重。

**回答中的加分点**：
- 具体的查询示例（"某个风格的推荐频次"）
- 把 ADR 同步和"推荐效果分析"联系起来

**回答时不要踩的坑**：
- 不要说"因为课程要求 Neo4j 存储 ADR"——虽然这是真实原因之一

## Q10: JSON 存储和 Neo4j 存储的接口为什么能统一？

**面试官为什么问**：判断你是否理解接口抽象的设计。

**推荐回答**：

> 因为 GraphRepository 和 JsonRepository 实现了完全相同的方法签名——`get_styles()`、`add_style()`、`get_feedback()`、`add_feedback()`、`add_adr()`、`get_adrs()`、`get_adr()`、`get_combinations()`、`graph_match()`、`get_feedback_stats()`、`get_learned_weights()`。方法的返回值结构也完全一致。`_repo()` 调度函数的工作就是"先调 GraphRepository，返回 None 则调 JsonRepository"——它对上层暴露的是统一的 `_repo("method_name", ...)` 接口。

**回答中的加分点**：
- 列出了所有统一的方法——说明是系统设计的而非偶然
- "上层调的是 _repo() 而不是直接调 Repository"——清晰的抽象层次

**回答时不要踩的坑**：
- 不要说"两边代码一样"——实现不同但接口相同才是关键

## Q11: 为什么不在 Neo4j 里存所有的数据？还要保留 JSON？

**面试官为什么问**：测试你对"最小依赖"设计原则的理解。

**推荐回答**：

> 因为 Neo4j 是可选依赖。我的设计原则是"零外部依赖也能跑"——如果同学没有装 Docker 或者 Neo4j 镜像拉不下来，系统不能因此无法启动。JSON 文件是纯 Python 标准库就能读写的——不需要额外的数据库、不需要驱动、不需要网络。Neo4j 是增强能力，JSON 是兜底能力。

**回答中的加分点**：
- "零外部依赖也能跑"——清晰的工程原则
- 考虑到"同学的环境可能不好"——务实的场景考量

**回答时不要踩的坑**：
- 不要说"因为 JSON 更简单"——要讲原则而非偏好

## Q12: 你的知识图谱能自动学习新知识吗？

**面试官为什么问**：区分"静态知识图谱"和"学习型知识图谱"。

**推荐回答**：

> 不能自动学习。知识图谱的数据来源于两个渠道：第一，初始化脚本从 JSON 预定义数据导入——这是固定的专家知识；第二，ADR 和 Feedback 是运行时自动创建的节点——这是动态数据。但从"自动学习"的角度，系统不会从 ADR 或 Feedback 中自动抽取新的 HAS_QUALITY 关系或发现新的 COMPLEMENTS 关系。学习权重（`learned_weights.json`）是唯一接近"学习"的机制——它基于用户反馈动态调整特征-风格的关联强度。

**回答中的加分点**：
- 区分了"静态知识"、"动态数据"和"自动学习"
- 指出了学习权重是唯一接近学习的机制

**回答时不要踩的坑**：
- 不要说"有自动学习能力"——图中所有的关系都是预定义或直接创建的

## Q13: 如果要扩展到 100 种风格，你的图模型需要怎么改？

**面试官为什么问**：判断你的图模型是否可扩展。

**推荐回答**：

> 图模型本身不需要改——6 节点 + 6 关系的 schema 足够通用，新增风格只是新增节点和边。需要改的是数据管理：JSON 的 `architecture_styles.json` 会变得很大，可能需要拆分为多个分类文件。初始化脚本需要支持增量更新而非全量重建——否则每次加一个风格就要重建整个图。另外，知识图谱中可能出现"孤岛"——某些风格和任何质量属性都不关联——需要在查询时处理空结果集的情况。

**回答中的加分点**：
- "schema 不需要改"——说明模型设计通用
- 考虑到数据管理和查询边界问题

**回答时不要踩的坑**：
- 不要只乐观地说"没问题"——要考虑到数据管理和边界处理

## Q14: Neo4j 的 Python driver 你是怎么用的？有没有连接池？

**面试官为什么问**：判断你是否有关注数据库连接的工程细节。

**推荐回答**：

> 当前使用方式比较朴素——每次查询创建新的 driver，查询完成后 `driver.close()`。没有使用连接池。这在当前 QPS ≈ 1 的场景下完全够用——driver 创建开销远小于 Cypher 查询本身的耗时。如果 QPS 升高，应该改用单例模式持有 driver（Neo4j driver 本身是线程安全的，内置了连接池），避免重复创建连接的开销。

**回答中的加分点**：
- 诚实现有实现的朴素但说明了为什么够用
- 知道 driver 是线程安全且内置连接池

**回答时不要踩的坑**：
- 不要说"用了连接池"——当前代码每次创建新 driver

## Q15: 如果知识图谱中的质量属性和 requirements-agent 的特征维度不一致怎么办？

**面试官为什么问**：判断你是否意识到两个模块之间的契约。

**推荐回答**：

> 这是一个关键的一致性约束。requirements-agent 的 10 个 FEATURE_LABELS_ZH（如 high_concurrency → "高并发"）和 Neo4j 中 QualityAttribute 节点的 name（如 "high_concurrency"）必须保持一致——否则 `graph_match()` 用特征名去 Cypher 查询 `QualityAttribute {name: $feat}` 时会匹配不到。当前一致性是靠人工维护的——两边用的都是同一套 10 个维度名称。如果要更健壮，应该在初始化脚本中校验：读取 requirements-agent 的特征列表，确认 Neo4j 中每个都有对应的 QualityAttribute 节点。

**回答中的加分点**：
- 识别到了这是一个"契约"问题
- 提出了具体的校验方案

**回答时不要踩的坑**：
- 不要说"不会不一致"——代码中没有自动校验机制
