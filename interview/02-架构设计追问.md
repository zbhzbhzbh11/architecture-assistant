# 02 · 架构设计追问

> 16 个问题。面试官在这个阶段判断你的架构设计能力——不只是"用了什么"，更是"为什么这样设计"。

---

## Q1: 你的系统自身是什么架构风格？为什么？

**面试官为什么问**：这是"用架构推荐系统的人自己怎么架构"的经典自反性问题。

**推荐回答**：

> 微服务架构。6 个独立容器通过 Docker Compose 编排。选择微服务有四个理由：第一，领域职责解耦——需求提取（词典规则）、风格匹配（规则引擎+图谱）、LLM 评估（外部 API 调用）各属不同领域，拆开后各自独立演进；第二，故障隔离——LLM 调用是外部依赖，独立部署 evaluation-agent 后 LLM 波动不影响其他模块；第三，异构能力集成——requirements-agent 是计算密集型，evaluation-agent 是 IO 密集型，拆分后可独立优化；第四，技术栈灵活——matching-agent 可以独立升级 Neo4j 驱动而不影响其他服务。

**回答中的加分点**：
- 四个理由互不重叠（解耦/隔离/异构/灵活）
- "计算密集 vs IO 密集"——体现代码级思考

**回答时不要踩的坑**：
- 不要说"因为大家都用微服务"——没有自己的判断

---

## Q2: 为什么不用单体架构？单体也能完成功能。

**面试官为什么问**：这是追问 Q1 的必然延伸。面试官故意唱反调。

**推荐回答**：

> 单体确实能完成功能——三个 Agent 的逻辑合并在一个 FastAPI 应用里，不做服务拆分，也能跑通。但有两个问题。第一，故障传播——LLM 调用超时导致线程阻塞，关键词提取也跟着受影响。微服务架构下，LLM 波动只影响 evaluation-agent。第二，代码边界模糊——当一个文件里既有关键词词典、又有评分函数、又有 LLM prompt，改一个功能很容易不小心影响另一个。微服务的物理隔离强制了代码边界——evaluation-agent 的代码改不了 matching-agent 的行为。

**回答中的加分点**：
- 先承认"单体也能跑通"——不回避问题
- "物理隔离强制代码边界"——这是微服务常被忽略的核心价值

**回答时不要踩的坑**：
- 不要只说"解耦""可扩展"等抽象词——要落到具体场景

---

## Q3: 你的微服务拆分粒度是否过细？6 个服务是否过度设计？

**面试官为什么问**：测试你是否考虑过"微服务的代价"。

**推荐回答**：

> 从 QPS 看确实有点过细——这个系统的请求量完全可以用单体处理。但微服务的价值在这个项目中不体现在"扛并发"，而体现在"职责清晰"和"故障隔离"。对于课程项目来说，清晰的模块边界更重要——它让代码评审和答辩讲解变得容易。而且 6 个容器通过 Docker Compose 一键启动，运维成本并没有显著增加。如果是生产环境且 QPS 确实很低，我会考虑把三个 Agent 合并为一个服务。

**回答中的加分点**：
- 诚实承认"从 QPS 看是过细的"
- "课程项目的模块边界比性能更重要"——场景化思考

**回答时不要踩的坑**：
- 不要死扛"微服务就是好"——显得教条
- 不要完全让步"就是过度设计了"——要给出场景化的理由

## Q4: 画出你的系统架构图，并解释每个组件的职责。

**面试官为什么问**：测试你是否能徒手画出自己的架构。画不出来的大概率不是自己做的。

**推荐回答**：

> *（在白板上从左到右画）*
>
> 最左边是用户浏览器 → 前端（Nginx :3000）→ API Gateway（FastAPI :8000）。
>
> Gateway 下面是三个核心 Agent 的并行排列：requirements-agent (:8001) 做特征提取——10 维关键词匹配 + LLM 语义补全；matching-agent (:8002) 做风格匹配——规则引擎评分 + Neo4j 图推理 + 组合推荐；evaluation-agent (:8003) 做评估决策——规则排序 + LLM 投票和摘要。
>
> matching-agent 和 evaluation-agent 下面连接 knowledge-base (:8004)——它有两种存储后端：左边是 Neo4j 图数据库 (:7687)，右边是 JSON 文件（fallback）。
>
> 最右边是 refactoring-agent (:8005)——架构坏味检测和重构模式推荐。它不在主链路上，是 Gateway 在主推荐完成后异步调用的，超时 8 秒，失败不阻塞。
>
> 整个编排流是：Gateway 收到请求 → 先查缓存 → 缓存 miss 则进入 LangGraph 工作流 → extract → match → evaluate → trace → 返回结果 → 异步调重构建议 → 结果注入报告。

**回答中的加分点**：
- 按数据流方向画（左→右），不跳跃
- 提到"超时 8 秒，失败不阻塞"、"先查缓存"——体现细节

**回答时不要踩的坑**：
- 不要跳着画——面试官会晕
- 不要只画不解释——每个组件一句话

## Q5: API Gateway 为什么不直接调 Agent，中间为什么要加 LangGraph？

**面试官为什么问**：判断你对"编排层"价值的理解。

**推荐回答**：

> LangGraph 的价值不是"能把三个函数串起来"——手写 50 行 httpx 也能做到。它的价值是：第一，形式化建模——控制流从业务代码中分离为显式的图结构，新增/调整步骤不需要改业务逻辑；第二，状态管理——TypedDict 定义了每一步的输入输出契约，字段由类型系统校验；第三，框架就绪——当前是线性 DAG，但如果未来需要条件路由（低置信度走人工审批）、并行 fan-out（批量评估）、或人机交互节点，LangGraph 已经提供了原生的支持。

**回答中的加分点**：
- 主动承认"手写 50 行也能做到"——诚实
- "框架就绪"——解释了为什么当前价值不明显但长期有价值

**回答时不要踩的坑**：
- 不要说"因为课程要求 LangGraph"——虽然这是事实，但面试官想听技术理由

## Q6: 为什么选择 FastAPI 而不是 Flask 或 Express？

**面试官为什么问**：测试你的技术选型能力。

**推荐回答**：

> 三个原因。第一，Pydantic 的类型校验——Agent 间的请求/响应结构比较复杂（如 evaluation-agent 的 EvaluateRequest 有 4 个嵌套字段），Pydantic 的 BaseModel 自动校验类型和必填字段，少写了很多 `if "key" not in data` 的防御代码。第二，自动 OpenAPI 文档——6 个服务共 23 个端点，Swagger UI 让各 Agent 的接口一目了然。第三，async/await 原生支持——api-gateway 需要异步调用多个 Agent（httpx AsyncClient），FastAPI 的异步路由天然匹配。

**回答中的加分点**：
- 三个理由互不相同（类型安全/文档/异步）
- 提到"少写防御代码"——体现实战经验

**回答时不要踩的坑**：
- 不要说"因为大家都用 FastAPI"——没有自己的判断
- 不要踩 Flask——"Flask 也能做到但 FastAPI 更方便"比"Flask 不好"更得体

## Q7: 服务间通信为什么用同步 HTTP 而不是消息队列？

**面试官为什么问**：测试你是否考虑过异步通信，以及为什么没选。

**推荐回答**：

> 场景决定的。当前系统是一个同步的推荐请求——用户提交需求，等几秒拿到完整报告。这个场景天然适合请求-响应模式（HTTP）。消息队列适合异步场景——比如提交一个批量评估任务，后台慢慢处理，完成后通知用户。如果未来需要异步处理，可以在 Gateway 和 Agent 之间插入消息队列——但当前的设计在 Gateway 和 Agent 之间已经是解耦的（通过 URL 配置），加入消息队列不需要改 Agent 代码。

**回答中的加分点**：
- "场景决定技术选择"——不是教条地说"HTTP 比消息队列好"
- "Gateway 和 Agent 之间已经解耦"——展示了扩展思维

**回答时不要踩的坑**：
- 不要说"消息队列太重了不适合"——显得经验不足

## Q8: 你的系统有没有考虑过服务发现和负载均衡？

**面试官为什么问**：判断你是否有生产环境意识。

**推荐回答**：

> 当前设计没有服务发现和负载均衡——所有服务地址通过 Docker Compose 的环境变量静态配置。这对于 6 个容器、QPS ≈ 1 的场景是够用的。如果要在生产环境部署，我会引入 Consul 或 Kubernetes Service 做服务发现，配合 Kubernetes HPA 或 HAProxy 做负载均衡——但这需要额外的运维基础设施。当前 Docker Compose 的静态配置是"复杂度刚好"——够用但不过度。

**回答中的加分点**：
- 诚实说没有，但知道如果要加应该用什么
- "复杂度刚好"——体现务实

**回答时不要踩的坑**：
- 不要说自己实现了负载均衡——代码里没有
- 不要说"不需要负载均衡"——要承认"当前不需要"，但"未来可能需要"

## Q9: 前端为什么是原生 HTML 而不是 React/Vue？

**面试官为什么问**：判断你是否会为了用框架而用框架。

## Q10: Docker Compose 编排中你遇到了什么问题？

**面试官为什么问**：判断你是否有容器化实践经验。

**推荐回答**：

> 主要问题是 Neo4j 的启动时间。Neo4j 容器需要 15-30 秒才能接受 Cypher 查询，但 Docker Compose 的 `depends_on` 只检查容器是否启动（不检查服务是否就绪）。解决方案是 Neo4j 容器配置了 healthcheck——`cypher-shell 'RETURN 1'`——其他依赖 Neo4j 的服务等 healthcheck 通过后才启动连接。Docker Compose v3.9 原生支持 `depends_on` 的 `condition: service_healthy`，但在本项目的版本中我用的是默认行为配合应用层的重试。

**回答中的加分点**：
- 具体的问题 + 具体的解决方案
- 知道 `condition: service_healthy` 的版本限制

**回答时不要踩的坑**：
- 不要说"一切顺利"——没有真实项目是完全没有问题的

## Q11: 为什么不用 Kubernetes 而是 Docker Compose？

**面试官为什么问**：判断你是否理解容器编排工具的适用场景。

**推荐回答**：

> Kubernetes 的价值是集群管理——多节点调度、自动扩缩容、滚动更新、服务网格。对于 6 个容器的单机部署场景，Docker Compose 完全够用。引入 K8s 需要额外的集群搭建和维护——对课程项目来说是过度工程化。但我的服务设计已经为 K8s 做好了准备——每个服务独立 Dockerfile、通过环境变量配置、无状态——迁移到 K8s 只需要写 Deployment 和 Service YAML，服务代码不需要改动。

**回答中的加分点**：
- "服务设计已为 K8s 做好准备"——前瞻性思维
- 具体的"只需要改什么、不需要改什么"

**回答时不要踩的坑**：
- 不要说"K8s 太重了"——但在实际生产中它是标配
- 不要显得不知道 K8s 是什么

## Q12: 你的系统如何保证高可用？

**面试官为什么问**：这是一个陷阱题——课程项目通常没有真正的高可用设计。面试官想看你能不能识别陷阱。

**推荐回答**：

> 诚实地说，当前系统没有生产级的高可用设计——没有多副本、没有自动故障转移、没有优雅重启。但我做了"降级高可用"——不是保证服务不挂，而是保证"服务挂了之后核心功能不受影响"。具体来说：LLM 不可用时降级为规则模式，Neo4j 不可用时降级为 JSON 存储，LangGraph 不可用时降级为手动编排。这是一种 pragmatic 的高可用——不是通过冗余保证服务在线，而是通过降级保证功能可用。

**回答中的加分点**：
- 识别陷阱——不说"我做了高可用"而说"我做了降级高可用"
- "pragmatic 的高可用"——把局限包装成设计哲学

**回答时不要踩的坑**：
- 不要说自己实现了多副本部署（代码里没有）
- 不要说"课程项目不需要高可用"——虽然真实但显得不够专业

## Q13: 如果你要支持 1000 QPS，哪些地方会成为瓶颈？

**面试官为什么问**：测试你在高压场景下的架构分析能力。

**推荐回答**：

> 三个瓶颈。第一，LLM API 调用——外部 API 有速率限制，1000 QPS 下可能需要队列削峰和结果缓存。第二，Neo4j 图查询——`graph_match()` 对每个活跃特征执行独立 Cypher 查询，高并发下需要连接池和查询优化。第三，同步 HTTP 编排——当前三步串行调用限制了吞吐量。解决方向：引入消息队列做异步处理、扩大请求级缓存（相同需求直接返回缓存）、对 matching-agent 做只读副本扩展（因为它不依赖 LLM，纯计算和数据库查询）。

**回答中的加分点**：
- 三个瓶颈对应三个不同层面（外部 API / 数据库 / 编排）
- 每个瓶颈都有解决方向

**回答时不要踩的坑**：
- 不要说"去掉 LLM 就行了"——太简单粗暴

## Q15: API Gateway 的缓存键是怎么设计的？为什么这样设计？

**面试官为什么问**：缓存键设计是工程师的"品味"体现。

**推荐回答**：

> 缓存键 = SHA256(requirement + model + knowledge_version) 的前 16 位。包含 requirement 是显然的——相同需求应该返回相同结果。包含 model 是因为不同 LLM 模型（如 deepseek-chat vs qwen-turbo）可能给出不同的投票和摘要，不应该共享缓存。包含 knowledge_version 是最关键的一层——`knowledge_version` 默认是 architecture_styles.json 文件的 MD5 前 8 位。一旦我修改了知识库（新增/修改/删除风格），所有旧缓存自动失效。这意味着不需要手动清缓存——知识的更新自动触发缓存的更新。

**回答中的加分点**：
- "knowledge_version 用文件 MD5"——自动失效的精妙设计
- "知识更新自动触发缓存更新"——点出了设计哲学

**回答时不要踩的坑**：
- 不要说"就是为了去重"——缓存键的设计远不止去重

---

## Q16: 为什么不用 Redis 做缓存？

**面试官为什么问**：判断你是否了解常见缓存方案以及取舍。

**推荐回答**：

> Redis 是更好的缓存方案——支持分布式、LRU 淘汰、持久化、更丰富的 TTL 策略。但对于这个项目，Redis 意味着多一个容器、多一个外部依赖。我的设计目标是"零外部依赖也能跑"——所以默认缓存是内存模式（`simple_cache.py`，96 行纯 Python）。SQLite 模式（`sqlite_cache.py`，155 行）提供了跨重启的持久化。两个实现的接口完全一致，如果未来需要升级到 Redis，只需要实现 `get/set/clear/stats` 四个方法，Gateway 代码不需要改动。

**回答中的加分点**：
- 知道 Redis 的优势
- "接口一致，未来可替换"——展示扩展设计

**回答时不要踩的坑**：
- 不要说"Redis 太重了不适合"——要说明"当前不需要，但设计支持替换"
