# 06 · 测试与工程化追问

> 15 个问题。面试官在这里判断你是否有工程素养——不只是"写对了"，更是"能证明写对了"。

---

## Q1: 你的测试策略是什么？

**面试官为什么问**：判断你是否有系统化的测试思维。

**推荐回答**：

> 四层测试金字塔。最底层是 76 条单元测试（pytest）——覆盖全部 6 个模块的核心函数，不需要外部服务，秒级反馈。第二层是 20 条回归用例——需要系统运行，每条验证 5 个维度（候选数、主流覆盖、推荐完整性、可解释性、对比矩阵）。第三层是冒烟测试——部署后或演示前快速验证核心链路。顶层是 43 项自动验收——逐条对照课程要求，一键生成通过率报告。

**回答中的加分点**：
- 四层递进，职责清晰
- 每层都说明了"需要什么条件、验证什么"

**回答时不要踩的坑**：
- 不要说"写了一些测试"——没有层次感

---

## Q2: 你为什么写 76 条单元测试？76 是怎么来的？

**面试官为什么问**：判断测试数量是自然覆盖结果还是刻意凑数。

**推荐回答**：

> 76 是自然覆盖的结果，不是目标。6 个模块，每个模块大约 10-15 条测试——matching-agent 的测试最多（因为评分逻辑复杂，需要测试各种特征组合和边界情况），refactoring-agent 的测试最少（因为主要是关键词检测，case 少）。我没有设"必须写到 X 条"的目标——每个函数的正常路径、边界条件和异常路径覆盖到了就算完。

**回答中的加分点**：
- "自然覆盖结果不是目标"——测试不是为了凑数
- 知道哪个模块测试多、哪个少、为什么

**回答时不要踩的坑**：
- 不要说"凑到 76 条"——测试最忌"为了数量"

## Q3: 回归测试的 5 个验证维度是什么？为什么要这 5 个？

**面试官为什么问**：判断测试维度是否有设计考量。

**推荐回答**：

> 5 个维度：`has_top3`（候选 ≥ 3 个）、`mainstream_covered`（主流风格在候选集）、`has_final`（有最终推荐）、`explainable`（有规则理由和 LLM 摘要字段）、`has_matrix`（对比矩阵 ≥ 3 行）。
>
> 这 5 个维度不是随机的——它们对应系统的 5 个核心功能承诺：完整性（返回足够的候选）、可靠性（主流风格不掉链子）、明确性（有结论不模棱两可）、可解释性（推荐有据可查）、可比较性（用户能横向对比）。如果任何一个维度失败，说明系统的核心链路出了问题。

**回答中的加分点**：
- 每个维度对应一个功能承诺
- "任一失败说明核心链路出问题"——明确了 5 维的必要性

**回答时不要踩的坑**：
- 不要只说维度名字不解释为什么

## Q4: 20 条回归用例覆盖了哪些场景？

**面试官为什么问**：判断测试用例是否有设计方法。

**推荐回答**：

> 覆盖 8 大类场景：即时通讯（2 条不同措辞）、电商平台（高并发+强一致）、企业系统（审批/数据中台/内容管理）、金融系统（银行账务/支付网关）、大数据系统（日志分析/ETL/科研数据）、特殊领域（医疗/政务/物联网/证券）、在线教育、短视频/SaaS/外卖/校园。
>
> 设计原则是"风格覆盖"——确保 10 种架构风格都能被至少一条用例触发为预期推荐。比如 Event-Driven 对应即时通讯和日志分析，Microservices 对应电商和跨境支付，Layered 对应审批系统和内容管理，Pipeline-Filter 对应科研数据处理。

**回答中的加分点**：
- 测试用例有明确的设计原则（风格覆盖）
- 能说出每种风格对应的典型用例

**回答时不要踩的坑**：
- 不要说"就是收集了 20 个常见需求"——要有设计方法

## Q5: 你的测试能在 CI 中自动运行吗？

**面试官为什么问**：判断你是否有持续集成的意识。

**推荐回答**：

> 当前没有 CI/CD 配置。单元测试（`pytest`）可以在任何安装了 Python 依赖的环境中自动运行——不需要 Docker、不需要外部服务。回归测试需要系统运行（`docker compose up`），当前是手动执行。如果要接入 CI（比如 GitHub Actions），需要写 workflow 文件——先 docker compose up，等 healthcheck 通过，再跑回归测试。框架已经就绪但没配 CI。

**回答中的加分点**：
- 诚实说没有，但知道如果要加怎么做
- 区分了"能在 CI 跑"（单元）和"不能在 CI 跑"（回归）

**回答时不要踩的坑**：
- 不要把"本地能跑"说成"有 CI"

## Q6: 自动验收检查了什么？为什么是 43 项？

**面试官为什么问**：判断验收体系的完整性。

**推荐回答**：

> 43 项覆盖 7 大类：核心目录结构（4 项）、Agent 模块检测（含多个 Agent 文件的检测）、知识库完整性（10 种风格字段完整）、测试数据集（20 条用例）、LLM 集成（4 个配置项 + 10 处调用代码）、Web API（23 个端点）、可视化与文档。
>
> 43 不是目标数字——是逐条对照课程要求后自然得出的。每一条都对应课程要求的一个具体检查点。自动验收验证的是"存在性"——文件在不在、API 端点有没有、风格数量够不够。它不验证"正确性"——那是回归测试的工作。

**回答中的加分点**：
- 区分了"存在性验证"（验收）和"正确性验证"（回归）
- 7 大类说得清楚

**回答时不要踩的坑**：
- 不要把验收测试说成功能测试——它们的职责不同

## Q7: 你如何保证改了代码后不破坏已有功能？

**面试官为什么问**：回归防护能力的判断。

**推荐回答**：

> 三步验证。第一步，本地改完代码后跑 `pytest tests/unit/ -v`——秒级反馈，确认核心函数行为不变。第二步，部署后跑 `python tests/run_regression.py`——20 条用例全部通过 5 维验证才算安全。第三步，答辩前跑 `python scripts/check_assignment.py`——43 项验收全部绿色才放心。这三步形成一个"改动 → 单元验证 → 端到端验证 → 课程要求验证"的闭环。

**回答中的加分点**：
- 三步有明确的顺序和触发时机
- "改动→单元→端到端→要求"的闭环表述

**回答时不要踩的坑**：
- 不要说"我自己测一下就知道了"——手动测试不可靠

## Q8: 冒烟测试和回归测试有什么区别？

**面试官为什么问**：判断你是否清楚测试分层。

**推荐回答**：

> 冒烟测试只验证"核心链路能不能跑通"——一个请求、一个预期 200 响应、一个完整报告。速度快（通常 10-20 秒），适合部署后立刻执行。回归测试验证"核心链路在各种场景下对不对"——20 条用例、5 维验证、覆盖 8 大类场景。速度慢（需要串行 20 次推荐），适合代码改动后、演示前执行。简单说：冒烟告诉你"系统没挂"，回归告诉你"系统没坏"。

**回答中的加分点**：
- "冒烟告诉你没挂，回归告诉你没坏"——精炼

**回答时不要踩的坑**：
- 不要混为一谈——两者的目的和执行频率都不同

## Q9: 你有没有做过压力测试？

**面试官为什么问**：判断你是否有性能验证意识。

**推荐回答**：

> 没有。系统的设计 QPS 是 1（课程演示场景），没有做过任何压力测试。如果要做的的话，瓶颈显然在 LLM 调用——外部 API 有速率限制，单次调用延迟 2-5 秒。缓存可以缓解重复请求的压力（相同需求直接命中内存/SQLite），但不同需求的并发请求会遇到 LLM API 的限流。压力测试主要验证的应该是"LLM 调用队列 + 缓存命中率"的组合效果。

**回答中的加分点**：
- 诚实说没有，但知道瓶颈在哪
- 能说出压力测试应该验证什么

**回答时不要踩的坑**：
- 不要说"不需要压力测试"——任何系统在讨论可靠性时都需要

## Q10: 你有没有考虑过安全测试？

**面试官为什么问**：安全意识。

**推荐回答**：

> 没有系统化的安全测试。当前系统的主要安全风险包括：API 端点全部开放无认证（任何能访问 `:8000` 的人都能调推荐接口）、用户输入直接传给 LLM（潜在 Prompt Injection）、没有速率限制（可能被恶意刷接口触发大量 LLM 调用）。这些在当前课程演示场景下是可接受的——系统运行在本地 Docker 网络内，不暴露到公网。但如果要对外服务，至少需要 API Key 认证、输入长度限制、速率限制。

**回答中的加分点**：
- 列出具体风险而不是泛泛地说"需要注意安全"
- 指出"当前场景可接受"——有场景判断能力

**回答时不要踩的坑**：
- 不要说"不需要安全测试"——要承认风险但说明当前场景

## Q11: 你的代码有类型检查吗？

**面试官为什么问**：判断你的代码质量意识。

**推荐回答**：

> 部分有。Pydantic BaseModel 提供了请求/响应模型的运行时类型校验——所有 API 端点都有 request schema。LangGraph 的 TypedDict（ArchitectureWorkflowState）提供了状态字典的类型安全。但 Python 代码本身没有静态类型注解——函数参数和返回值大部分没有 type hints。这不是最佳实践，主要是开发节奏的妥协。如果代码规模继续增长，我会引入 mypy 做静态类型检查。

**回答中的加分点**：
- 知道哪里做了类型检查、哪里没做
- 知道 mypy

**回答时不要踩的坑**：
- 不要说"Python 不需要类型"——但可以解释"当前规模下开销收益比不划算"

## Q13: 如果评审老师要求你证明系统确实"可靠"，你会怎么做？

**面试官为什么问**：将抽象的可靠性转化为可验证的指标。

**推荐回答**：

> 我展示三样东西。第一，回归测试的 20 × 5 维通过矩阵——证明系统在 20 种不同场景下都能给出结构完整的推荐。第二，LLM 全关测试的结果——证明在 LLM 完全不可用的情况下，系统仍然能正常运行，推荐的准确率（以规则引擎的评分为准）不受影响。第三，自动验收的 43/43 通过报告——证明系统满足了课程的全部客观要求。这就是我理解的"可靠"——不是承诺零故障，而是能证明在各种条件下行为符合预期。

**回答中的加分点**：
- "可靠不是零故障而是行为可预期"——成熟的定义
- 三样证据层次分明

**回答时不要踩的坑**：
- 不要说"我测试了很多次没出问题"——没有说服力

## Q14: 你对测试覆盖率有要求吗？

**面试官为什么问**：判断你是否了解测试覆盖率的概念和局限。

**推荐回答**：

> 没有设置覆盖率门禁（如"必须 80% 行覆盖"）。我用 pytest 但没有配 coverage.py。主要原因是课程项目场景下，我更关注"关键路径是否覆盖"而不是"行覆盖率是否达标"。matching-agent 的 score_style() 是评分核心，它的测试最充分——正常路径、边界条件、图谱融合都有覆盖。但像 FastAPI 路由层、httpx 错误处理等胶水代码，测试价值有限。如果要设置覆盖率目标，行覆盖 70%+ 是合理的——但核心逻辑（评分、特征提取、降级）应该接近 100%。

**回答中的加分点**：
- 理解覆盖率的局限性（"不是行覆盖达标就可以"）
- 区分了核心逻辑和胶水代码的测试价值

**回答时不要踩的坑**：
- 不要说"不需要覆盖率"——需要解释"为什么不是第一优先级"

## Q15: 如果我把你的代码 clone 下来，需要几步才能跑通测试？

**面试官为什么问**：判断项目是否真正可复现。

**推荐回答**：

> 单元测试：两步——`pip install -r requirements.txt && pytest tests/unit/ -v`。不需要 Docker、不需要 LLM 配置、不需要 Neo4j。
>
> 回归测试：四步——`docker compose up --build`（等待所有容器 healthy）→ `docker compose exec knowledge-base python init/init_neo4j.py`（初始化图数据）→ 确认 api-gateway :8000 可访问 → `python tests/run_regression.py`。
>
> 自动验收：一步——`python scripts/check_assignment.py --project-root .`。不需要服务运行。

**回答中的加分点**：
- 三种测试各自需要什么条件说得清楚
- 具体命令可复制

**回答时不要踩的坑**：
- 不要说"装好环境就能跑"——太模糊
